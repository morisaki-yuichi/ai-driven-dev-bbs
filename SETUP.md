# SETUP

AI掲示板（仮）のセットアップ・起動・リセット手順。評価者（人間・AIいずれも）がクリーンな
`git clone` 直後の状態から、他に何も知らずにこの手順だけで進められることを前提に書く。

前提として、Docker と Docker Compose（`docker compose` サブコマンド）が使えること。それ以外の
ミドルウェア（PostgreSQL 等）をホストに直接インストールする必要はない。すべて Docker
コンテナとして起動する。

秘匿キー（人間が値を用意する必要のあるもの）は**無い**。`.env.example` はDB接続情報の固定値
（開発・評価用のプレースホルダで、実際の秘密情報ではない）のみを持つ。

## 初回セットアップ

リポジトリ直下で以下を実行する。

```bash
cp .env.example .env
```

`.env` は `compose.yaml` が参照する `POSTGRES_USER` / `POSTGRES_PASSWORD` / `POSTGRES_DB` を
供給する。このファイルが無いと `docker compose up` は起動しない（値は変更しなくてよい）。

これでセットアップは完了。値の書き換えやその他の準備は不要。

## アプリの起動

リポジトリ直下で以下を実行する。

```bash
docker compose up
```

- `db`（PostgreSQL 17）と `app`（このリポジトリの Rust アプリケーション）の2コンテナが起動する。
- `app` は `db` の healthcheck が通るのを待ってから起動し、起動時にマイグレーション
  （`app/migrations/`）を自動適用する。空のDBから追加のコマンドなしでスキーマが揃う。
- 起動が完了すると、ブラウザで `http://localhost:3000/` にアクセスできる。
- 停止するには `Ctrl+C`、またはバックグラウンド起動（`docker compose up -d`）した場合は
  `docker compose down` を使う。

### 初回起動にかかる時間

初回の `docker compose up` は `app` イメージを Rust の release ビルドでゼロから作るため、
数分（依存クレートの量やホストの性能によっては5分程度）かかることがある。ビルドは
`cargo chef` で依存関係のレイヤーをキャッシュしているが、初回はそのキャッシュ自体が
無い状態から作るため最も時間がかかる。ログに `Compiling` や `Building` 系の出力が続いて
いる間はビルド中であり、停止・タイムアウトではない。2回目以降の起動はレイヤーキャッシュが
効くため大幅に短くなる。コード変更後の再ビルドも、変更箇所より下流のクレートだけが
再コンパイルされるため初回より速い。

### 起動成功の確認方法

以下が確認できれば起動は成功している。

- `docker compose ps` で `app` コンテナの `STATUS` が `Up`（起動したまま）と表示され続けて
  いる（`Restarting` を繰り返していない）。`app` は通常運用時、標準出力へログを一切出さない
  ため、ログの有無ではなくコンテナが再起動ループに入っていないことで判断する。
- ブラウザで `http://localhost:3000/` にアクセスすると、未認証状態のため `303 See Other` で
  `/login` へリダイレクトされ、ログイン画面が表示される。これが「起動成功かつアプリが
  正常に応答している」ことの最も分かりやすい確認方法。

### ホストの 5432 番ポートが既に使われている場合

`compose.yaml` は `db` コンテナの 5432 番ポートをホストへ publish する。ホスト側で
既に別のプロセスやコンテナが 5432 番を使っていると、`docker compose up` はポートの
衝突で起動に失敗する（`db` コンテナが起動できない、または `Bind for 0.0.0.0:5432 failed`
のようなエラーが出る）。この場合は `compose.yaml` を変更するのではなく、衝突している側
（ホストにインストール済みの PostgreSQL、他のプロジェクトの Docker コンテナなど）を
停止してから、改めて `docker compose up` を実行する。

### `.env.example` は2箇所にある

このリポジトリにはリポジトリ直下の `.env.example` と `app/.env.example` の2つが存在する。
評価者経路（`docker compose up`）で使うのは**リポジトリ直下**の `.env.example` のみで、
「初回セットアップ」節の `cp .env.example .env` はこのファイルを指す。`app/.env.example`
は別用途（ホストの `cargo` を使う開発時に `app/` 側で参照するもの）であり、混同しないこと。

## 評価をやり直す場合（空DBへのリセット）

一度使ったDBのボリュームを残したまま `docker compose up` すると、前回までのデータ
（登録済みユーザー・スレッド・コメント）が残った状態で起動する。評価シナリオを最初から
やり直すなど、空のDB状態から始めたい場合は、名前付きボリュームごと削除してから起動し直す。

```bash
docker compose down -v
docker compose up
```

`down -v` は `db` コンテナのデータボリューム（`db-data`）を削除する。次の `docker compose up`
はマイグレーションだけが適用された空のDBから起動する（シードデータは投入されない）。
