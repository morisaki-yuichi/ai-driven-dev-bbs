## 概要

このプロジェクトはAI駆動開発を体験するための教材用プロジェクトです。
必要な要件をファイルとしてまとめ、実際に開発を始めるための初期ファイルを内部に持ちます

## この教材を利用した開発時のルール

1. 人間によるソースコードの編集禁止
   - デバッグや修正はすべてAIに依頼する
2. 開発標準ルールのドキュメントなどもすべて人間の編集禁止
3. 人間によるgitの操作禁止
   - コミット、プッシュ、プル、ブランチ操作などすべて禁止
4. 人間ができる作業は以下の通り
   - AIへの指示
   - AIエージェントへの指示に関するファイルの編集
     - .AGENTS.md
     - .CLAUDE.md
     - .github/や.claude/以下のSKILLやagents定義ファイルなど
   - エディタなどの設定ファイルの編集
   - コマンドの実行（ただし極力AIに実行させる）
5. 人間はソースコードに対してエンジニアリングするのではなく、AIエージェントに対してエンジニアリングを行うことを意識する
6. AIの動作が行き詰った場合
   - どうしてもAIのみで解決できない場合は、問題がある個所を人間がデバッグしてもよい
   - ただし、その場合でもAIに解決方法や原因などをフィードバックし同じ問題が再発しないようにすること（ドキュメント化やルール・SKILLの追加などを推奨）

## 開発内容

ブラウザ上で動作する掲示板ページ

### 要件

- ユーザー登録機能
  - ユニークIDとパスワード・表示名を設定する
  - ユニークIDは他ユーザーとの重複不可
  - パスワードは12文字以上で、英数字記号をすべて含む
    - 記号の種類は「!@#$%^&\*()\_+-=[]{}|;':",./<>?」のみ
  - 表示名は15文字以内の任意の文字列で重複可
  - パスワードを忘れた場合の特別の救済機能は不要
- ログイン機能
  - ユニークIDとパスワードを入力してログインする

### 以下機能はログイン後に利用可能

- ログアウト機能
  - ログイン状態を解除する
  - ログイン後に利用可能なすべての機能が利用不可になる
- ユーザーの編集機能
  - 表示名のみ編集可能
- スレッドの作成機能
  - スレッドにはタイトルと本文がある
  - スレッドのタイトル・本文は編集不可
- スレッドの削除機能
  - 削除できるのはスレッドの作成者のみ
  - すでにコメントがあるスレッドは削除できない
  - 削除したスレッドは復元できない
- コメントの作成機能
  - コメントの編集不可
- コメントの削除機能
  - 削除できるのはコメントの作成者のみ
  - 削除したコメントは復元できない
  - ただし、＜このコメントは削除されました＞とスレッドのコメント位置に表示する

### スレッド・コメントの表示に関する機能

- スレッド一覧表示
  - スレッドのタイトル
  - スレッドの本文
  - スレッドの作成日時
  - スレッドの作成者
  - コメント数
  - 最終更新日時
- スレッド詳細表示
  - スレッドのタイトル
  - スレッドの本文
  - スレッドの作成日時
  - スレッドの作成者
  - コメント一覧
    - コメントの本文
    - コメントの作成日時
    - コメントの作成者
- スレッド本文・コメントの検索機能
  - 検索文字列を含む本文・コメントを検索する
  - 検索結果を一覧表示する
  - 検索結果を選択することで対象のスレッド詳細表示の該当箇所にスクロールする
- スレッドのソート機能
  - 作成日時順
  - コメント数順
  - 最終更新日時順
- スレッドのページネーション機能
  - 1ページあたりの表示件数は10件
    - 10件以外の表示件数への切り替え機能はあってもよいが、必須ではない
  - 全N件という表示は必須ではない
  - 最低でも「前に戻る」「次に進む」ことができる機能は必須
    - 1ページ目と最終ページでは「前に戻る」「次に進む」が表示されない
      - ※「前に戻る」「次に進む」の機能に該当するラベルやボタンは、任意の表示でよい
    - ほかの移動方法はあってもよいが、必須ではない

### 技術スタック

- 完全に自由
- 掲示板ページはブラウザ上で動作すること
- CSSについては言及しない
  - CSSで装飾してもよいし、しなくてもいい
  - 一切CSSのないHTMLのDOMだけでもよい
  - Tailwind CSSやUIライブラリなどを利用してもよい
  - ただし、動作確認の際にAIがブラウザを操作する都合上、見えない要素や操作できない要素は避けること
- DBなどのデータ永続化については言及しない
  - 自分の技術選定に任せて適切な方法で実装すること

### 動作PCに対する前提条件

- 無条件に前提として設定してよい項目は以下の4点
  1. gitがインストールされていること
  2. Docker/Dcoker composeがインストールされていること
  3. 動作するOSをUbuntuやWindowsで指定すること
  4. 技術スタックで選定した言語のパッケージマネージャがインストールされていること
     - 例: JavaScriptやTypeScriptを使用するならnpmやyarn、pnpmがインストールされていること
     - Pythonを使用するならpipやuvがインストールされていること
     - などなど
- 上記以外は前提条件として設定しないこと
  - 例えば、ローカルPCにPostgreSQLやRedisがインストールされていることは前提としないこと
  - データベースやキャッシュサーバーなどはDockerコンテナで起動することを前提としてよいが、そのための起動コマンドなどはREADME.mdから参照可能なドキュメントにまとめ、AIが自律的に発見、実行できるようにすること

### 実装後の動作確認について

- 以下の手順で動作確認を行う
  1. gitからcloneした直後のリポジトリに対して以下の作業を行う
     1. AIに対して以下の指示を実行する

     ```
     このプロジェクトのREADME.mdを読み込み、アプリケーションの初期セットアップ方法を把握してください。README.mdから参照されているドキュメントやスクリプトがあれば、それらも参照してください。
     その後、手順に従って初回のセットアップを実行してください。
     ユーザーが書き込むべき秘匿キーなどがある場合は、その旨をユーザーに伝えてください。それ以外はすべてAIが自動で実行してください。
     初回セットアップが完了したら、その旨をユーザーに伝えてください。
     ```

     2. 初回セットアップが完了したら、続けて以下の指示を実行する

     ```
     このプロジェクトのREADME.mdを読み込み、アプリケーションの起動方法を把握してください。README.mdから参照されているドキュメントやスクリプトがあれば、それらも参照してください。
     その後、手順に従ってアプリケーションを起動してください。
     起動が完了したら、その旨をユーザーに伝えてください。
     ```

  2. アプリケーションが起動した後、事前に用意したシナリオに従ってテストを実行する
     - このシナリオは自然言語で記述される
     - playwrightなどで実行するのではなく、自然言語で記述されたシナリオをもとにAIがagent-browserを利用してテストする
