# 実装フェーズ・オーケストレーション プレイブック（第2段階）

**位置づけ**: `human-guide/purpose.md` §5 の第2段階（ゲート統治）。手動運用（`human-guide/new-project-setup-prompts.md` の実-A/実-B＝第1段階・正本）と同じ内容を、**オーケストレータAIがモデル指定つきサブエージェントで駆動**し、人間は裁定ゲートのみを担う形に写したもの。

- 本ファイルは**AI（オーケストレータ）が読む**。`human-guide/`（AI除外）とは別扱いで、新プロジェクトへは `human-guide/` と同時にコピーし、`permissions.deny` の対象に**入れない**。
- ワーカーのプロンプト本文は手順書の実-A/実-B と**意図的に重複**している（ゲート分割に伴う文言差も意図的）。手順書側を変えたら本ファイルも同期する。
- モデル配分は手順書の補足メモ「実行モデルの配分」（2026年7月時点の判断）から導出。世代交代時はそちらを再導出してから本ファイルを追随させる。

## 起動プロンプト（人間がオーケストレータのセッションに貼る）

```
あなたはこのリポジトリの実装フェーズを進行するオーケストレータです。orchestration/implementation-playbook.md を読み、その規律・ゲート・ステップ定義に厳密に従って進行してください。まず S1（状態復元・着手提案）をサブエージェントで実行し、結果を提示して私の裁定（G1）を待ってください。
```

## オーケストレータの規律

1. **自分では作業しない**: 各ステップは指定モデルのサブエージェントに委譲する。オーケストレータが行うのは、起動・結果の検分・停止判断・人間への提示のみ。
2. **プロンプトは定義どおり渡す**: ワーカーには該当ステップのプロンプトブロック（`<...>` を差し替えたもの）だけを渡す。自分の要約・推測・他ステップの内容・過去ワーカーの生ログを混ぜない（レビューの独立性と、指示の再現性を守るため）。
3. **ゲートは必ず停止する**: G1〜G4 で人間の裁定を得るまで次のステップを起動しない。裁定を推測・代行しない。
4. **STATUS を検分する**: 各ワーカーは報告末尾に共通の STATUS ブロック（後述）を出す。`completed` 以外は即停止して人間に提示する。STATUS ブロックが無い・壊れている場合も停止する（成功と見なさない）。
5. **git はゲートの後ろ**: ワーカーにコミット・push をさせない（ブランチ作成は可）。コミット・PR は G4 の承認後に S5 で行う。
6. **ワーカーの結果を偽らない**: 実行中・未完了のステップの結果を先取りして報告しない。
7. **G1 承認スコープと S2 報告を突き合わせる**: G1 で承認したスコープ項目を控えておき、S2 の報告と突き合わせる。承認された項目のうち報告で言及されていないものがあれば、実施漏れか記載漏れかを切り分けるまで先へ進めない。STATUS が completed であることは、承認したスコープが充足されたことを意味しない。

## ワーカー共通の出力契約（全ステップのプロンプト末尾に付加する）

```
報告の末尾に、次の形式の STATUS ブロックを必ず出力してください:
STATUS: completed | blocked | escalation
BLOCKERS: なし | <内容を1行ずつ>
DECISIONS_LOGGED: なし | <このセッションで /log-decision した番号>
NEEDS_HUMAN: なし | <人間の裁定が必要な事項を1行ずつ>
- escalation は「同一の失敗への修正試行が3回連続で進展しない」（テスト・Lean証明とも）に達したときに使い、試したこと・仮説・残る症状を本文に整理すること。
- blocked は前提不足・環境問題など自力で解消できない停止に使う。
```

## 人間ゲート

| ゲート | タイミング | 人間が裁定すること |
|---|---|---|
| **G1 着手承認** | S1 の後 | 着手する機能の選択／提示された AC・テスト観点・変更範囲の妥当性／難所指定（S2 のモデルを Opus 4.8 に上げるか） |
| **G2 エスカレーション裁定** | STATUS: escalation / blocked の発生時（随時） | 方針転換・追加情報・打ち切りの判断。証明の非収束ではモデル昇格（Sonnet 5→Opus 4.8、難証明は Fable 5 単発）もここで指示 |
| **G3 レビュー裁定** | S3 の後 | /code-review・/security-review の中/軽指摘の扱い（反映 or 保留）／NEEDS_HUMAN 事項 |
| **G4 完了承認** | S4 の後 | DoD 充足の確認とコミット・PR の承認（コミットメッセージは S5 が提示） |

## ステップ定義

### S1 状態復元・着手提案（モデル: Sonnet 5）

```
このセッションでは現状把握と着手提案のみを行います。実装はしません。

1. CLAUDE.md を読む（恒久ルール）。
2. dev-docs/session-logs/ の最新ファイルを読み、前回の「次にやること」「未解決事項」を把握する。
3. dev-docs/decisions/README.md の索引で、これまでの確定事項（特に importance: critical）を把握する。
4. 実装対象の仕様は docs/product/issues/（機能とAC）と docs/evaluation/scenarios/（合格基準）にある。未着手・仕掛かりの機能を確認する。
5. 依存順（認証を先に、その後スレッド/コメント/検索・ソート/プロフィール）を踏まえ、次に着手すべき機能の候補と推奨順を挙げる。
6. 最有力候補について、対象issueのAC・テスト観点・想定する変更範囲・対応する不変条件（formal/README.md の一覧から）を要約する。認証・セッション・WebSocket 等の難所に当たるかの見立ても添える。

git 操作は行わないこと。
```

### S2 証明＋TDD実装（モデル: Sonnet 5。G1 で難所指定があれば Opus 4.8）

G1 で承認された機能名を `<機能>` に差し替える。

```
次の1機能を実装します: <機能>（着手内容は進行役経由で承認済み）

進め方（CLAUDE.md と dev-docs/ のルールに従う）:
- main は保護。feat/<name> の短命ブランチで作業する。git commit は行わない（コミットは人間の承認後に別途行う）。
- Lean 形式化（Must）: 実装に入る前に、この機能に対応する不変条件（formal/README.md の一覧を参照）を formal/ で証明する（通常の Red より一段強い Red）。証明で確定した場合分け・境界は実装とテストに反映し、モデルを単体テストのオラクル（同一入力での出力照合）に使う。証明の非収束もエスカレーションルール（3回）の対象。
- TDD厳守: Red(失敗するテストを先に書く) → Green(最小実装で通す) → Refactor(テストは緑のまま整理)。
- 非収束時のエスカレーション: 同一の失敗への修正試行が3回連続で進展しなければ、それ以上試行せず、試したこと・仮説・残る症状を整理して STATUS: escalation で終了する。
- 実装方針(Action/Calculation/Data): ロジックは純粋な計算(Calculation)に寄せて単体で厚く固め、副作用(Action: DB/セッション/HTTP)は薄く隔離する(functional core / imperative shell)。Data は型で表す。単純さを壊す過剰な層は作らない。
- テストの層: 単体(純粋な計算=ロジック/バリデーションを厚く) → 結合(API+DB=Action) → E2E。対応する docs/evaluation/scenarios/ の受け入れ基準を満たすこと。
- 意図の置き場所: コード=How／テスト=What／コミット本文=Why(decision参照)／コメント=Why・Why-not(Howを繰り返さない)。
- 独自の技術判断をしたら、実装を続ける前に /log-decision で記録する。
- Definition of Done のうち本ステップ分を満たす: テスト緑 → /verify で実挙動を確認 → UIを触るなら dev-docs/ui-ux-guidelines.md のチェックを満たす（agent-browserで操作可能・隠し要素にアクションを強いない）。※ /code-review・/security-review は次のステップで別途実行される。
- 評価ハーネス前提を壊さない: 空DBから 01→05 を逐次実行でき、シード無し・テスト間でDBを消さない状態を保つ。マイグレーションで再現可能にする。

実装後、何がどのテストで緑になったか・証明した不変条件・/verify の結果を報告してください。
```

### S3 レビュー（モデル: Opus 4.8・新しいサブエージェント）

```
機能 <機能> の作業差分（feat ブランチの未コミット変更を含む）をレビューします。

1. /code-review を実行する。
2. この機能がユーザー入力を扱う場合は /security-review も実行する。
3. 重大の指摘のみ自分で修正し、修正後にテストが緑であることを再確認する。
4. 中/軽の指摘は修正せず、対応の要否の一覧として報告する（NEEDS_HUMAN に含める）。
5. このサイクルで `sorry` のまま残す形式モデルの言明について、それが偽になっていないか検査する。偽の言明を `sorry` 付きで残すと、未証明のTODOに見えて実は証明不可能であり、将来のセッションが補題として使えばそこから導かれるものがすべて無価値になる。反例が見つかった場合は、証明を後回しにせず言明そのものに必要な仮定を補って真の形に直したうえで `sorry` を残す。
```

### S4 ログ（モデル: Sonnet 5）

```
今回の機能実装セッション群の内容をログにします。/log-session を実行し、フェーズ（開発）・今回やったこと（機能 <機能>・証明した不変条件・レビュー結果の要点）・決めたこと（関連 decision 番号）・次にやること・未解決事項を dev-docs/session-logs/ に1ファイルとして記録してください。git 操作は行わないこと。
```

### S5 コミット・PR（G4 承認後。モデル: Sonnet 5）

```
機能 <機能> の作業を確定します。dev-docs/workflow.md に従い、1論理変更=1コミットで feat ブランチにコミットし（ヘッダ=簡潔なWhat／本文=Why・関連 decision を参照）、コミットメッセージを提示のうえ、承認済みの内容で PR を作成してください。push と PR 作成まで行い、URL を報告してください。
```

## 進行フロー（1機能 = 1サイクル）

```
S1 → [G1 着手承認] → S2 → (escalation/blocked なら [G2]) → S3 → [G3 レビュー裁定]
   → S4 → [G4 完了承認] → S5 → 次の機能へ（S1 に戻る）
```

- どのステップでも STATUS が completed 以外なら進行を止めて人間へ。
- 全シナリオ（01→05）対応の機能が完了したら、空DBからの逐次E2E通し確認を人間に提案して終了する。
