# /log-decision

`dev-docs/decisions/` に決定記録を1件追加するスキル。番号採番からREADME索引更新まで、抜け漏れなく機械的に行うために存在する（手動で連番を数えたり索引を書き忘れたりする事故を防ぐ）。

## このスキルが記録する対象

- 仕様（`docs/` 原典）が沈黙している論点について、AIが前提を置いた／複数の選択肢を比較して選んだ／ユーザーが指示した、という**判断そのもの**。
- 実装の詳細な手段（変数名、関数分割など）は対象外。「どちらでも仕様を満たすが、どちらかに決めなければ先に進めない」という分岐点が対象。

## 引数の受け取り方

`$ARGUMENTS` を空白区切りで解釈する。

```
/log-decision <slug> <importance> <decided_by> [タイトル...]
```

- **`<slug>`**（必須）: ファイル名に使う英小文字ケバブケース。例: `csrf-protection`。
- **`<importance>`**（必須）: `critical` / `major` / `minor` のいずれか。`dev-docs/decisions/README.md` の定義（「覆したときに何がやり直しになるか」）に従って判断する。**引数で与えられなかった、または3値以外なら、AIが憶測で埋めずユーザーに確認する。**
- **`<decided_by>`**（必須）: `ai` / `user` / `ai+user` のいずれか。同じくREADMEの定義に従う。**この会話で実際に人間の承認を得ていないなら `ai` にする**（`ai+user` にできるのは、この会話内でユーザーが採否を判断した場合のみ）。不明な場合はユーザーに確認する。
- **`[タイトル]`**: 残りの引数すべてを結合したもの。省略された場合は、会話の文脈から簡潔な日本語タイトルを起こす。

引数が不足・不正な場合は、他の項目を先に進めず、その場でユーザーに質問して確定させる（importance と decided_by は特に、後から一覧の索引を歪めるため確実性を優先する）。

## 手順

### 1. 直前の会話から決定内容を抽出する

この会話の中で、何が論点で、どんな選択肢を検討し、何を結論としたかを振り返る。これが本文の「背景」「選択肢」「提案」「決定」「影響」の元になる。会話に無い内容を憶測で埋めない。原典（`docs/` / `dev-docs/requirements-analysis.md`）を根拠として引用できる場合は引用する。

### 2. 次の番号を決める

既存ファイルの4桁連番の最大値+1を使う。欠番を作らず、取り下げられた決定の番号も再利用しない（`dev-docs/decisions/README.md` のファイル名規約）。

```bash
ls dev-docs/decisions | grep -E '^[0-9]{4}-' | sed -E 's/^([0-9]{4})-.*/\1/' | sort -n | tail -1
```

上記の最大値に1を足し、4桁ゼロ埋めした値を新番号とする。

### 3. TEMPLATE.md を複製して本文を書く

`dev-docs/decisions/TEMPLATE.md` を元に `dev-docs/decisions/<新番号>-<slug>.md` を作成する。frontmatter は以下のとおり埋める。

```yaml
---
id: <新番号>
title: <タイトル>
date: <今日の絶対日付 YYYY-MM-DD>
importance: <引数のimportance>
decided_by: <引数のdecided_by>
status: <下記の対応表>
---
```

`status` の既定値（decided_by から機械的に決める。ユーザーから明示指定があればそちらを優先）:

| decided_by | status |
| :--- | :--- |
| `user` / `ai+user` | `決定済` |
| `ai` | `提案`（人間の承認を経ていない暫定決定であることを示す） |

本文は TEMPLATE.md の見出し構成（関連論点／関連原典／背景／選択肢／提案／決定／影響／変更履歴）をそのまま使う。

- **背景**: `[事実]`（原典が書いていること）と `[空白]`（原典が沈黙していること）を分けて書く。`dev-docs/requirements-analysis.md` §3 の論点ID（D00〜D20）に対応する場合は明記する。
- **決定**の見出しは `decided_by` に応じて使い分ける（TEMPLATE.md の指示どおり）: `ai+user` → `## 決定（YYYY-MM-DD ユーザー判断）` / `ai` → `## 決定（YYYY-MM-DD AI判断・未承認）` / `user` → `## 決定（YYYY-MM-DD ユーザー指示）`。
- **影響**: この決定がどのファイル・スキーマ・画面・他の decision に波及するかを書く。
- **変更履歴**: 新規作成時は空でよい（既存決定を覆す場合のみ追記する。その場合は既存ファイルを直接更新し、新規番号は起こさない——このスキルは新規作成のみを担当する）。

### 4. `dev-docs/decisions/README.md` の索引を更新する

以下の3箇所を確認し、該当すれば更新する。

1. **一覧テーブル**（`## 一覧`）に新しい行を番号順に追加する。
2. **`decided_by = ai` の索引**（`### 索引: decided_by = ai`）: 今回の `decided_by` が `ai` なら、この節に追記する（「該当なし（0件）」の文言だった場合は削除して表形式に変える）。**この節に続く説明文**（「20XX-XX-XX 時点で、0001〜XXXX のすべてが `ai+user`」のような文）**も、今回追加した番号を踏まえて書き換える**。表だけ更新して説明文を古いままにすると、索引が自己矛盾する。
3. **`importance = critical` の索引**（`### 索引: importance = critical`）: 今回の `importance` が `critical` なら、この節に1行追加する。他の critical 決定との依存関係が明らかならその旨も書き添える。

### 5. `decided_by = ai` の場合、コード側にも番号コメントを残す

`decisions/README.md` の decided_by 定義は、`ai`（未承認の暫定決定）について「コード側にも同じ番号のコメントを残すこと」を要求している。これは、レビューされていない前提の上に実装が積まれている箇所を後から洗い出すための仕掛け。**この決定に基づいて実装コードを書く／既に書いてある場合は、該当箇所に `// decision <番号>: 提案（未承認）` のような一言コメントを残す**（実装がまだ無ければ、次にそのコードを書くときに残すことを報告に含める）。

### 6. 報告する

作成したファイルパス・番号・importance・decided_by・status を要約してユーザーに報告する。`decided_by = ai` で status が `提案` の場合は、**未承認の暫定決定であること**と、コード側に番号コメントを残す（残した／残す予定である）ことを明示的に伝える。
