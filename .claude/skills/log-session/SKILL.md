# /log-session

`dev-docs/session-logs/` に、この会話1回分の作業記録を残すスキル。

## decision記録との違い（重要）

- **`dev-docs/decisions/`（[log-decision](../log-decision/SKILL.md) スキル）**: 仕様が規定していない**個々の論点**に対する結論を、論点ごとに1ファイルで記録する。実装がそれに従う規範。
- **`dev-docs/session-logs/`（本スキル）**: **1回のセッションで何をしたか**を時系列に記録する作業日誌。その回に決定記録を作った／更新した場合は「決めたこと」の項目でその decision 番号を参照するが、決定内容そのものを複製しない。

**両者は別系統であり、session-logs にdecisionの本文を書き写したり、decisionsにセッションの作業ログを書いたりしない。**

## 形式の正典

ファイル名規約・見出し構成・記述ルールは **`dev-docs/session-log-format.md` を正とする**。本スキルはそれを複製しない。作業前に必ずそのファイルを読み、規約（ファイル名の日付・通し番号・スラッグ、5項目の見出し、「なし」の明記ルールなど）に従うこと。

## 手順

### 1. `dev-docs/session-log-format.md` を読む

複製しない理由は、形式が変わったときにスキル側の更新漏れで齟齬が起きるのを防ぐため。必ず最新の内容をその場で確認する。

### 2. 次の通し番号を決める

`dev-docs/session-log-format.md` の規約どおり、日付をまたいでもプロジェクト全体で連番を継続する。

```bash
ls dev-docs/session-logs | grep -E '^[0-9]{4}-[0-9]{2}-[0-9]{2}-[0-9]{2}-' | sed -E 's/^[0-9]{4}-[0-9]{2}-[0-9]{2}-([0-9]{2})-.*/\1/' | sort -n | tail -1
```

上記の最大値に1を足し、2桁ゼロ埋めした値を使う。

### 3. スラッグとタイトルを決める

`$ARGUMENTS` が与えられていればそれをスラッグに使う。無ければ、このセッションでの主な作業内容から英小文字ケバブケースのスラッグを起こす（例: `phase2-foundation-skills`）。タイトルはセッションログの見出しに使う日本語の短い説明。

### 4. この会話を振り返り、5項目を埋める

会話全体（このセッションで実際にやり取りした内容）から、`dev-docs/session-log-format.md` の見出し構成（フェーズ／今回やったこと／決めたこと／次にやること／未解決事項の5項目）に従って書く。

- **フェーズ**: 現在の作業フェーズ番号（0/1/2/3等）。フェーズをまたぐ橋渡し的なセッションであれば、それが分かる書き方でよい（過去ログ #04・#05 を参照）。
- **今回やったこと**: 実際に行った作業（読んだファイル、作成・編集したファイル、実行した検証など）。
- **決めたこと**: この会話中に下した決定・確定した方針。対応する decision 番号があれば `（decision 0016）` のように併記する。decision化していない口頭の判断もここに含めてよいが、その場合は後で `/log-decision` が必要かどうかユーザーに確認する。
- **次にやること**: 次セッション以降に着手する予定の作業。承認待ちの成果物があれば明記する。
- **未解決事項**: 持ち越した論点・ユーザー判断待ちの事項。`dev-docs/requirements-analysis.md` の論点ID（D00〜D20 等）があれば併記する。

5項目はすべて必須。該当がなければ「なし」と書き、項目自体を省略しない（`session-log-format.md` の記述ルール）。日付は相対表現を使わず絶対日付で書く。

### 5. ファイルを作成する

`dev-docs/session-logs/<日付>-<連番>-<スラッグ>.md` に、`session-log-format.md` の見出し構成どおりの内容を書き込む。

### 6. 報告する

作成したファイルパスを短く報告する。
