## 2. 選定軸（本決定で適用したもの）

ユーザー指示により、以下の階層で評価した。**軸の定義から機械的に点を導かず、候補ごとにこのアプリのエラー面・状態の持ち方に即した論証を行う**ことを条件とした。

- **Tier 0（ハード制約・合否ゲート）**: 仕様充足と評価ハーネス適合。**実際に候補を落とせる基準のみを置く。** どの候補も落とさない基準は空のゲートなので Tier 0 に置かず、その程度差を B・C で重み付けする。
- **Tier 1（主要最適化軸）**
  - **A. relevant shift-left 被覆率** — **このアプリのエラー面**のうち、どれだけを実行時でなくビルド時に検出できるか。**型システムの絶対的強度ではなく、この案件で現金化できる早期検出**で採点する。評価は `agent-browser`（ランタイム）なので、ランタイム失敗はブラウザ越しデバッグとなり最も高コスト。
  - **B. 残余の反復摩擦** — A で取り返せない**純粋な反復コストのみ**を減点する。「コンパイル段がある／型がある」こと自体は罰しない（A との二重計上を避ける）。**コンパイラが本物の欠陥で弾く＝A、既に正しいコードを証明できず弾く＝B。**
- **Tier 2（二次/タイブレーク）**: C. エコシステム成熟度・ドキュメント密度（＝モデルの生成信頼性）／D. 運用コスト・性能／E. 一般的学習価値／F. テスト容易性・TDD エルゴノミクス／G. 再現性のあるビルド。

> **ラベルの凡例（混同注意）**: Tier 0 のゲートは **`T0-1`〜`T0-3`**、Tier 1・Tier 2 の軸は **`A`〜`G`** と表記する。**両者は別系統であり、`T0-n` は Tier 2 の軸 `G`（再現性のあるビルド）の細目ではない。**

## 3. Tier 0 — ゲート（実際に落ちたもの）

| ゲート | 根拠 | 実際に落ちた候補 |
| :--- | :--- | :--- |
| **T0-1 SSR / MPA であること** | [0008](0008-browser-back-is-out-of-model.md)。AC03-2 のブラウザバックを `agent-browser` が実際に押す | React / Vue SPA、Next.js App Router のクライアントルーティング、Phoenix LiveView、Inertia |
| **T0-2 RDBMS ＋ リクエスト単位トランザクション** | [0002](0002-action-failure-semantics.md)（失敗時の完全巻き戻し）、[0007](0007-session-multiplicity.md)（セッションも DB） | ファイル／JSON 永続化、KVS 単独、インメモリ、MongoDB |

**Tier 0 のゲートはこの2つだけである。**

### Tier 1 の減点に転換した基準（ゲートに置かなかったもの）

**T0-3 候補だった「`LIKE '%kw%'` で大文字小文字を区別できる」は、ゲートに置かず Tier 1 の減点として扱う。**

- **根拠**: [0011](0011-search-matching-and-normalization.md)（`rust` は `Rust` にヒットしない）。
- **ゲートにしない理由**: MySQL は `COLLATE utf8mb4_bin`、SQLite は `PRAGMA case_sensitive_like=ON` を明示すれば充足でき、**どの候補も落ちない**。§2 の宣言「どの候補も落とさない基準は空のゲートなので Tier 0 に置かない」に従う。
- **減点先の軸（明示）**:
  - **主に B**。DB のセットアップに**恒久的な追加設定が1つ増える**。これは機能を1つも増やさない純粋な儀式であり、B の定義（捕まえたバグを伴わない摩擦）に合致する。
  - **副次的に A**。設定を落とした場合、**DB の既定が決定済み仕様（0011）を静かに破る**。エラーにならず検索結果が黙って変わるだけなので、ビルド時にもテスト時にも落ちず、`agent-browser` 越しにしか観測できない。「既定が正しい」DB を選べばこの失敗経路自体が存在しない、という意味での早期性である。
- **結果**: PostgreSQL のみが**既定のまま**充足するため、この減点は PostgreSQL に対してゼロ、SQLite・MySQL に対して非ゼロとなった（§6 の DB 却下理由）。

### 空ゲートとして Tier 0 から外したもの

「ブラウザで動く」「Docker 化できる」「永続化できる」「README 起点でセットアップが完結する」「空 DB から 01→05 を逐次実行できる」「AI が完成させられる」は、**検討したすべての候補で成立した**。ご指示に従い、これらは合否ゲートに置かず、その**程度差**を B（セットアップ手順の複雑さ・反復コスト）と C（生成信頼性）に回した。

## 4. このアプリのエラー面（A の採点対象）

A は「型の強さ」ではなく「**このアプリで現金化できる早期検出**」で採点するため、まず何が壊れうるかを確定した。

| # | エラー面 | このアプリでの具体例 | ランタイム露出のコスト |
| :--- | :--- | :--- | :--- |
| E1 | 型不一致 | ページ番号を文字列のまま `LIMIT` に渡す。コメント数を文字列比較してソートが壊れる | 中 |
| E2 | null / 未処理ケース | 存在しないスレッド ID で 404 を返さず 500 になる（C-10 違反）。削除済みコメントの分岐漏れ | **高** |
| E3 | **SQL / スキーマ不一致** | 列名の typo、マイグレーションと参照コードのドリフト、表示名解決 JOIN（[0015](0015-display-name-resolution.md)）の列取り違え | **最高** |
| E4 | **テンプレート不整合** | ビューに渡し忘れたフィールドが空表示になる。固定文言 `＜このコメントは削除されました＞`（C-01）の埋め込みミス。前後ボタンの表示条件（C-12）の取り違え | **最高** |
| E5 | ドメイン不変条件 | C-06（作成者 **かつ** コメント0件）の条件取り違え。削除済みコメントをコメント数から落とす（C-16 違反） | 高 |

**E3・E4 が支配的**である。いずれも「ブラウザで動かして初めて分かる」種類であり、E4 に至っては 500 にすらならず**静かに表示が壊れる**ため、`agent-browser` による自然言語判定（H-12）で見逃されうる。したがって A の採点は実質「**SQL とテンプレートをビルド時に検査できるか**」で決まる。

### 静的保証（メモリ安全・データ競合の排除）に、このアプリは守る対象を持つか

ご指示により、加点も 0 点化も論証を経て行う。

- 全状態は PostgreSQL 上にある。[0002](0002-action-failure-semantics.md) により 1 リクエスト＝1 トランザクション、[0007](0007-session-multiplicity.md) によりセッションすら DB。インメモリキャッシュもレート制限器も要件にない（D05: 原典に記述なし）。
- したがって**アプリプロセス内に共有可変状態が存在しない**。並行性はリクエスト間にしか現れず、その調停は DB のトランザクション分離が担う。
- 残るのは接続プールだが、これは**どの候補でもライブラリ側の実装**であり、我々が書くコードではない。

> **結論: このアプリにおいて、所有権・借用検査・`Send`/`Sync` が守る対象はほぼ空である。**
> Rust を採用したが、**その看板の利点は本件の A にほとんど加点していない**。この点を A の論拠に含めることは誇張になるため、明示的に除外した。学習価値としての意義は E に計上している。
> 同じ論証により、GC を持つ候補に「データ競合リスク」を理由とした D の減点も行っていない。

## 5. 候補バンドルと比較

T0-1・T0-2 を通過し、かつ E3 / E4 への態度が異なる3案を主候補とした。

| | **案T: TypeScript** | **案R: Rust（採用）** | **案G: Go** |
| :--- | :--- | :--- | :--- |
| 言語 | TypeScript (strict) / Node | **Rust** | Go |
| Web FW | Hono | **Axum** | chi / net/http |
| テンプレート | JSX（型付き） | **Askama**（コンパイル時検査） | templ（codegen） |
| データ層 | Drizzle ORM | **sqlx（`query!` マクロ）** | sqlc（codegen） |
| DB | PostgreSQL | **PostgreSQL** | PostgreSQL |
| テスト | Vitest | **cargo test + `#[sqlx::test]`** | go test |

### 5.1 A — relevant shift-left 被覆率

◎ = ビルド時に必ず落ちる ／ ○ = ほぼ落ちる（規律依存）／ △ = 部分的 ／ × = ランタイム

| エラー面 | 案T | **案R** | 案G | 案P（Python 参考） |
| :--- | :---: | :---: | :---: | :---: |
| E1 型不一致 | ◎ | ◎ | ○ | × |
| E2 null / 未処理ケース | ○ | **◎** | △ | × |
| **E3 SQL/スキーマ不一致** | ◎ | ◎ | ◎ | × |
| **E4 テンプレート不整合** | ◎ | ◎ | ◎ | × |
| E5 ドメイン不変条件 | ○ | **◎** | △ | × |

**候補ごとの論証**

- **E3 が3案とも ◎ なのは機構が異なるため。** 案R の `sqlx::query_as!` は**実 DB に接続して**SQL を検証する（最強）。案G の sqlc は `schema.sql` に対して SQL をパース・型付けする。案T の Drizzle は `schema.ts` が**マイグレーションの生成元でもある**ためドリフトが構造的に起きない。**強度は R > G ≈ T だが、本件の E3 は「列名 typo」「JOIN の取り違え」の水準であり、3案とも同じように捕まえる。差は現金化されない。**
- **E4 も3案とも ◎。** 案G のみ `templ generate` の codegen 儀式という代償がある（→ B）。
- **差が実際に出るのは E2 と E5 のみ。**
  - 案R: `Option` / `Result` の網羅マッチが**強制**される。さらに `formal/Bbs/` の Lean モデルは `Except Error` とエラー ADT で書かれており、**Rust の `enum Error` + `match` へほぼ 1 対 1 で移せる**。モデルと実装の乖離がコンパイルエラーとして現れる箇所が増える（本プロジェクト固有の加点）。
  - 案T: 判別可能ユニオン ＋ `strictNullChecks` で近づけるが、網羅性は `never` チェックを書く**規律依存**。境界（フォーム入力）は Zod 等で固めれば実質同等だが、強制ではない。
  - 案G: **ゼロ値が構造的な弱点。**「表示名を詰め忘れた」が `""` として静かにレンダリングされ、E2 と E4 の隙間から漏れる。sum type がなく E5 の状態表現も弱い。E2 は本件で明示された主要エラー面の一つであり、軽い減点ではない。
- **案P（Python + Flask + Jinja2 + SQLAlchemy）は主候補から除外した。** Jinja は未定義変数が既定で空文字になり、生 SQL も非検査で、**支配的な E3・E4 を一つも shift-left できない**。C（エコシステム密度）は最高だが、A の欠落を C で埋める設計は本件の軸の建て付けに反する。
- **案C#（ASP.NET Core Razor Pages + EF Core）も検討し落とした。** Razor はビューをコンパイルし、EF Core の code-first はスキーマドリフトを防ぐため A は案T 同等以上。しかし C（日本語圏の情報密度・モデルの生成信頼性）と D（イメージサイズ）で案T・案G に劣り、積極的に選ぶ理由が立たなかった。

**A の順位: 案R > 案T > 案G ≫ 案P。ただし案R と案T の差は E2 / E5 に限定され、支配的な E3 / E4 では差がつかない。**

### 5.2 B — 残余の反復摩擦

「コンパイル段がある」こと自体は罰していない。**バグを伴わない edit-compile 反復・儀式・待ち時間のみ**を減点した。

所要時間は**桁のバンドによる予測**である（精密な秒数は書かない）。**実装フェーズ初回セッションで実測と突き合わせること。**

| 指標 | 案T | **案R（採用）** | 案G |
| :--- | :--- | :--- | :--- |
| (a) 1ファイル変更 → 再コンパイル | 1秒未満 | **数十秒**（リンク段が支配的） | 数秒 |
| (b) 単体テスト一式 | 数秒 | **数十秒**（ビルド込み。実行のみなら数秒） | 数秒 |
| (c) 型／静的検査のみの高速コマンド | 数秒（`tsc --noEmit`） | **数秒〜数十秒**（`cargo check`） | 数秒（`go build`） |
| (d) 変更ごとの必須儀式 | drizzle-kit generate（スキーマ変更時のみ・数秒） | **`cargo sqlx prepare`**（SQL 変更のたび・数十秒）。テンプレート編集でも再コンパイルが要る | `sqlc generate` ＋ `templ generate`（クエリ／テンプレ変更のたび＝**高頻度**・各数秒） |
| コールドビルド | 数十秒 | **分単位** | 数十秒 |
| ホットリロード／即観測 | あり | **実質なし** | 部分的 |
| cycles-to-green（正しいコードを通すだけの反復） | 低 | **高** | 低〜中 |

**B の順位（良い順）: 案T > 案G > 案R。案R が最下位である。**

案R の摩擦は3層で効く。(i) テンプレートの文言を1行直すだけでも数十秒の再コンパイルが挟まる。(ii) SQL を1本足すたびに `sqlx prepare`。(iii) 借用検査・`&str`/`String`・ライフタイムを**通すだけ**の反復。

**(iii) は本件では純粋な B である。** §4 で論証したとおり、このアプリには借用検査が守る共有可変状態が存在しない。したがってその反復は**捕まえたバグを伴わない摩擦**であり、A へ振り替えることはできない。これが案R の最も痛い箇所であり、正直に減点した。

## 6. 決定（2026-07-19 ユーザー判断）

**案R（Rust + Axum + Askama + sqlx + PostgreSQL）を採用する。**

| レイヤ | 採用 | 却下した代替案と理由 |
| :--- | :--- | :--- |
| 言語 | **Rust** | TypeScript（A の E2/E5 が規律依存）／Go（E2 ゼロ値の穴）／Python（A 全滅）／C#（C・D で劣後） |
| Web フレームワーク | **Axum** | Actix Web（本件規模で性能差が現金化されない。C は Axum が優位）／Rocket（nightly 依存の歴史があり G を損なう） |
| テンプレート | **Askama**（SSR、CSS は最小限のプレーン CSS） | maud（HTML が Rust コード内に埋まり、教材としての可読性＝E で劣る）／Tera（**ランタイム評価**のため E4 が ◎→× に落ちる。**これは本決定の A を無効化するので不可**） |
| データ層 | **sqlx**（`query_as!` マクロ ＋ オフラインキャッシュ） | Diesel（DSL の学習コストと C の薄さ）／SeaORM（生成物が厚く B を悪化）／生 tokio-postgres（E3 が非検査） |
| DB | **PostgreSQL 17** | SQLite（§3 の「大文字小文字の区別」に `PRAGMA case_sensitive_like` が必要＝[0011](0011-search-matching-and-normalization.md) の決定を DB 既定が静かに破る。E でも劣る）／MySQL（同基準に `COLLATE` 明示が必要） |
| 認証／セッション | **`sessions` テーブル自作 ＋ HttpOnly / SameSite=Lax Cookie**、パスワードは `argon2` crate | JWT（[0007](0007-session-multiplicity.md) のログアウト失効と噛み合わない）／Redis（[0007](0007-session-multiplicity.md) が明示的に却下済み） |
| 実行環境 | **Docker Compose（app ＋ db）**。開発時はホストで `cargo` | app をホスト直実行のみ（H-06 のセットアップ完結性が落ちる） |
| テスト | **`cargo test` ＋ `#[sqlx::test]`**（テストごとに使い捨て DB） | 手動の DB 準備（F を損なう） |

### 6.1 判断の内訳（Tier 1 と Tier 2 のどちらが効いたか）

**この記録で最も重要な点なので、事実どおり書く。**

- **Tier 1（A・B）だけで採点した AI の推奨は案T だった。** A の差は E2 / E5 に限定される一方、B の差は構造的で規律では埋まらないため。
- **採否を分けたのは Tier 2 の E（学習価値）**、具体的には**本プロジェクト固有の「Lean 形式化（`formal/`）と実装の対応」を重視するというユーザー判断**である。`Except Error` ＋ エラー ADT で書かれた Lean モデルを、`Result` ＋ `enum` へほぼ 1 対 1 で移せるのは Rust だけであり、これは AI 駆動開発教材としての本プロジェクトのユニークな特徴にあたる。
- ガードレールに従い、**この E を A に密輸していない。** §5.1 の A 採点は Lean 対応のうち「網羅マッチによりモデルとの乖離がコンパイル時に出る」という**検出可能性の部分のみ**を加点しており、「教材として学びが多い」ことは A に一切算入していない。

### 6.2 高い B を A で取り返す論証（退避条項は置かない）

ご指示により、反復コストが高い候補は**その分を A の高さで明示的に取り返して見せる**こととした。「詰まったら別言語へ」といった退避条項は置かない。

案R が案T に対して支払う B の超過は、主に (a) 数十秒バンドの再コンパイル、(b) `sqlx prepare` 儀式、(c) 借用検査を通すだけの反復である。これに対して取り返す先は次のとおり。

1. **E2 の完全な強制（○ → ◎）。** 本件で 404 要件（C-10）は全画面に効き、削除済みコメント・削除済みスレッド・範囲外ページ（[0013](0013-pagination-edge-cases.md)）と分岐が多い。案T では `never` チェックを書き忘れれば穴が空くが、案R では `match` の非網羅がコンパイルエラーになる。**分岐の見落としはブラウザ越しにしか現れない種類**であり、ここでの1件の取りこぼしは (a) の数十秒を何十回分も上回る。
2. **E5 の完全な強制（○ → ◎）。** C-06（作成者 かつ コメント0件・削除済みを含む）と C-16 は、Lean モデルで既に述語として書かれている。これを `enum` ＋ 網羅マッチへ移すことで、**モデル側の場合分けを実装側で1つ落とす**という最も起きやすい乖離がコンパイル時に出る。
3. **(b) の `sqlx prepare` は頻度が低い。** SQL の本数は本アプリでは十数本程度で頭打ちになり（一覧・検索・詳細・各 CRUD）、儀式が走るのは SQL を変更したときだけである。**画面調整のたびに走る案G の codegen とは頻度が1桁違う。**

> **取り下げた勘定項目（2026-07-19・フェーズ1レビュー 中-1 により削除）**
>
> 当初はここに4項目めとして「`sqlx::query_as!` は実 DB に対して SQL を検証する（案T・案G はスキーマ**宣言**に対する検証にとどまる）。マイグレーションを書き換えた直後にクエリ側を直し忘れる、という E3 の典型がビルド時に落ちる」を挙げていた。**これは水増しであり、取り下げる。**
>
> 理由: §5.1 は E3 について「3案とも ◎。強度は R > G ≈ T だが、本件の E3 は『列名 typo』『JOIN の取り違え』の水準であり、**差は現金化されない**」と自ら採点している。さらに案T については「Drizzle は `schema.ts` がマイグレーションの生成元でもあるためドリフトが**構造的に起きない**」と論証済みで、案G の sqlc もマイグレーションの正である `schema.sql` に対して検証する。**A の採点で「差がつかない」と判定した項目を、B の回収勘定で差があるかのように再計上していた。**
>
> **「本件で現金化される具体シナリオを示して残す」という選択肢は採らなかった。** それは指摘された弱さを論証の再構成で見かけ上解消する行為であり、この決定記録の価値（AI の判断過程の正確な記録）を損なう。勘定は項目1・2・3 の3本に減る。**それでも §6.2 末尾の結論（純粋な損失を承知のうえで E を理由に選んだ）の形は変わらない。**

**取り返せていない部分も明記する。** (a) の「テンプレート1行の修正でも数十秒」と (c) の借用検査反復は、上記のどれによっても回収されない**純粋な損失**である。§7 の緩和策で桁を下げにいくが、案T の 1 秒未満バンドには届かない。**この損失を承知のうえで、E を理由に案R を選んだ**——というのが本決定の正確な形である。

### 6.3 その他の判断（推奨どおり採用）

- **J-2 DB**: PostgreSQL。[0011](0011-search-matching-and-normalization.md) の「大文字小文字を区別する」が**既定のまま満たされる**唯一の候補。
- **J-3 CSS**: 最小限のプレーン CSS。Tailwind はビルド段と儀式を増やし B を悪化させる。H-02 は素の DOM でも可としている。
- **J-4 decision 番号**: 本記録を **0016** とした。当初の指示は「0001-tech-stack.md」だったが、0001 は `0001-readme-editing-policy.md` として既に存在し、本 README の「番号を再利用しない」規約と衝突するため。`importance` / `decided_by` は 0016 以降を frontmatter、0001〜0015 は README 索引を正とする（過渡期の扱いは README に明記）。

## 7. 影響

### 7.1 実装に効く制約

- **`.sqlx/` オフラインキャッシュをリポジトリにコミットすることが必須。** `sqlx::query_as!` はコンパイル時に DB 接続を要求するが、評価者は clone 直後（H-13）に Docker イメージをビルドする。ビルド時に DB が居ない前提でなければ H-06 が破綻するため、**`SQLX_OFFLINE=true` ＋ コミット済み `.sqlx/` を前提とする**。SQL を変更したら `cargo sqlx prepare` を再実行してコミットする規律が要る（B の (d) の実体）。
- **マイグレーションは `sqlx::migrate!` でバイナリに埋め込み、起動時に自動適用する。** これにより「空の DB ボリュームから `docker compose up` するだけで H-08 を満たす」形になり、評価者が手動でマイグレーションを走らせる手順が不要になる（H-07）。
- **`Cache-Control: no-store` を認証必須画面に付与する**（C-11 / AC03-2）。[0008](0008-browser-back-is-out-of-model.md) が「データモデル外の要件」と整理したとおり、これは HTTP レイヤの責務である。
- **ドメイン層を DB 非依存の純粋な Rust コードとして切り出す。** `formal/Bbs/Validation.lean`・`Query.lean` の対応先であり、`#[sqlx::test]` を使わずミリ秒で回せる（F の確保）。§6.2 の 1・2 が成立する前提でもある。
- **Askama のバージョンは実装初回に実測で確定・固定する。** Askama は rinja のフォークと統合を経て API に変遷があり、**C（生成信頼性）で本スタック最大のリスク箇所**である。バージョンを `Cargo.toml` でピン留めし、公式ドキュメントを参照して書く（記憶で書かない）。

### 7.2 未決定として残るもの

- **D02（スキーマ詳細）**: 主キーの型、タイムスタンプ型の具体、インデックス。フェーズ2 で別 decision として起票する。[0009](0009-time-granularity-and-sort-tiebreak.md)（保存 UTC・ミリ秒精度）から `timestamptz(3)` が示唆される。
- **D14（空 DB へ戻す手段）**: `docker compose down -v` を手順ドキュメントに明記する方針。フェーズ2 で確定。
- **D05・D06・D10・D18・D19・D20**: 本決定の射程外。実装フェーズで順次起票する。

### 7.3 検証すべき予測

- **§5.2 の B バンドはすべて AI の予測である。** 実装フェーズ初回セッションで (a)〜(d) を実測し、乖離があれば本記録に変更履歴として追記する。予測が大きく外れた場合、それ自体が「AI の技術選定における見積もり精度」という教材上の観測対象になる。
- [0011](0011-search-matching-and-normalization.md) §実装上の落とし穴 と [0003](0003-character-counting-and-charset.md) の大文字小文字の扱いを、PostgreSQL 上で**1回まとめて実測確認する**（DB 選定直後のタスク）。
