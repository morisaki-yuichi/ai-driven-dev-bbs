# 0011 検索の照合方式・正規化・空クエリ

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: D07
- 関連原典: AC11-2、シナリオ04-1（`Rust` で検索）
- 影響範囲: `Bbs.Query.containsSubstr`、DB のインデックス設計

## 背景

- **[事実]** 原典は「本文またはコメントにその文字が**含まれる**スレッドが一覧表示される」と言うのみ。
  **照合方式の指定がない。**
- **[空白]** 大文字小文字の区別、全角半角の正規化、Unicode 正規化。
- **[空白]** **空クエリで検索した場合の挙動**（全件表示 / エラー / 検索前状態）。
- **[空白]** 検索結果一覧にソート・ページネーションが効くべきか。`ui_design.md` に検索結果専用画面はなく、
  P03（一覧）に結果を出す前提と読める。

## 提案

1. **素朴な部分文字列一致（`LIKE '%kw%'` 相当）**。全文検索エンジンの形態素解析は、
   `Rust` のような語が分かち書きの都合で意図どおりヒットしないリスクがある（D07 の技術的示唆）。
   評価規模では性能上の問題は起きない。
2. **正規化は行わない**（大文字小文字を区別、全角半角を区別）。シナリオは `Rust` を
   そのままの表記で入力・検索するため、正規化なしで通る。**原典が要求していない挙動を足さない。**
3. **空クエリは全件表示**とする。モデルの `containsSubstr s "" = true` がこの挙動。
4. **検索結果にもソートとページネーションを適用する**（P03 と同じ画面である以上、一貫させる）。

## 決定（2026-07-19 ユーザー判断）

**提案どおり 1〜4 をすべて採用する。**

1. **素朴な部分文字列一致**（`LIKE '%kw%'` 相当）。全文検索エンジンは使わない。
2. **正規化しない**（大文字小文字を区別、全角半角を区別）。
3. **空クエリは全件表示**（検索前の一覧と同じ振る舞い）。
4. **検索結果にもソートとページネーションを適用する。**

| 本文 `プログラミング言語Rustの特徴` に対して | 結果 |
| :--- | :--- |
| `Rust` | ヒット（シナリオ04 が使う表記） |
| `rust` | ヒットしない |
| `Ｒｕｓｔ` | ヒットしない |
| `（空）` | 全件表示 |

## ⚠ 実装上の落とし穴

**2 は DB 任せにすると意図せず破られる。** decision 0003（ユニークIDの大文字小文字）と同じ罠。

| DB | `LIKE` の既定の挙動 | 対応 |
| :--- | :--- | :--- |
| PostgreSQL | 大文字小文字を**区別する** | そのままでよい |
| MySQL / MariaDB | 既定の照合順序（`utf8mb4_general_ci` 等）は**区別しない** | `COLLATE utf8mb4_bin` 等の明示指定が必要 |
| SQLite | ASCII 範囲のみ区別しない（`LIKE` は既定で case-insensitive） | `PRAGMA case_sensitive_like=ON` 等 |

**フェーズ1 で DB を選んだ直後に実測して確認すること。** decision 0003 とまとめて1回で確かめられる。

## 影響

- 全文検索インデックス（`pg_trgm`、`MeCab` + 全文検索等）は**不要**。セットアップが軽くなり、
  H-06（README 起点でセットアップ完結）に有利。
- 4 により、**検索キーワード・ソートキー・ページ番号をすべてクエリパラメータで保持**する必要がある。
  `GET /?q=Rust&sort=created_desc&page=2` の形。C-13（ページをめくってもソート順が維持）と
  同じ仕組みで済むので、実装は1箇所にまとまる。
- 3 により、検索窓を空で送信したときの専用エラーは不要。バリデーションが1種類減る。
- 形式モデルの `Query.containsSubstr` / `search` は既にこの形。変更不要。
