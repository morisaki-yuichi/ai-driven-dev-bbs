# 0004 「空」の定義とトリム方針

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: D16
- 関連原典: AC05-2（タイトルまたは本文が空ならエラー）、AC07-2（コメント本文が空ならエラー）
- 影響範囲: `Bbs.Validation.nonEmptyText`、全入力フォーム

## 背景

- **[事実]** 原典は「空の場合エラー」と書くのみ。**空白のみの入力を空とみなすか、前後空白をトリムするかの規定がない**。
- 空白のみ（`"   "`、全角スペース、改行のみ）を許すと、一覧・詳細に空行のスレッドが並ぶ。

## 選択肢

1. トリムせず、長さ0のみを空とする → `"   "` が通る。
2. **前後空白をトリムしてから長さ0判定**し、保存はトリム後の値で行う。
3. 空白のみを空とみなすが、保存は入力どおり（トリムしない）。

## 決定（2026-07-19 ユーザー判断）

**選択肢2 を採用し、さらに保存もトリム後の値とする。** 表示名にも同じ規則を適用する。

| 入力 | 判定 | 保存値 |
| :--- | :--- | :--- |
| `""` | エラー | — |
| `"   "` | エラー | — |
| `"　　"`（全角スペース） | エラー | — |
| `"  本文  "` | OK | `"本文"` |
| `" テストユーザー01 "` | OK（9文字） | `"テストユーザー01"` |

- **本文中の**連続空白・改行は正規化しない（前後のみトリム）。
- 表示名の 15 文字判定は**トリム後**に行う。`" あいうえおかきくけこさしすせそ "` は 15 文字で通る。

## ⚠ 実装上の落とし穴（この決定の主要な帰結）

**「空白」の定義は言語・関数ごとに違う。全角スペース `U+3000` を拾うかどうかで割れる。**

形式化の過程で判明したが、**Lean の `Char.isWhitespace` は ASCII 限定**で `U+3000` に対して
`false` を返す。当初のモデルはこれを使っており、`"　　"`（全角スペースのみ）が
「空でない」と判定されて素通りしていた。`Validation.isSpaceChar` として
ASCII 空白＋`U+3000`＋`U+00A0` を明示的に定義し直してある。

主要言語の挙動（**採用言語で必ず実測して確認すること**）:

| 言語・関数 | `U+3000` を空白とみなすか |
| :--- | :--- |
| Lean `Char.isWhitespace` | **いいえ** |
| Java `String.trim()` | **いいえ**（`strip()` は はい） |
| Java `Character.isWhitespace()` | はい |
| JavaScript `String.prototype.trim()` | はい |
| Go `strings.TrimSpace` | はい |
| Python `str.strip()` | はい |
| Rust `str::trim()` | はい |

日本語入力を扱うアプリで `U+3000` を取りこぼすと、**全角スペースだけのスレッドやコメントが
投稿できてしまう**。評価では agent-browser が一覧を目視するため、空欄のカードが並ぶと
AC09-2（表示項目が正しく表示されている）の判定を乱すおそれがある。

## 影響

- `Bbs/Validation.lean` に `isSpaceChar` / `trim` を追加し、`isBlank` と
  `displayNameFailure` をトリム後判定に変更した。
- `Bbs/Op.lean` の `register` / `updateDisplayName` / `createThread` / `createComment` が
  トリム後の値を保存するようにした。
- `Bbs/Scenario.lean` に回帰確認を4件追加（全角スペースのみの拒否、本文・表示名のトリム保存、
  トリム後15文字の許容）。
- decision 0005（表示名の空文字）と整合: 空白のみの表示名も `displayNameEmpty` で弾かれる。
