# 0009 時刻の粒度・同時刻の衝突・ソートのタイブレーク

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: D17, D12
- 関連原典: AC09-2（作成日時の表示）、AC12-3〜5、シナリオ04-2-3（一番古いスレッドがトップ）
- 影響範囲: `Bbs.Query.leOf`、DB のタイムスタンプ型、一覧の並び

## 背景

ソート述語を書こうとした時点で、**同値キーのときにどちらを先に出すかが決まらない**ことが判明した。

- **[事実]** 原典はソート条件を3種（作成日時／コメント数／最終更新日時）挙げるだけで、
  **同値時のタイブレークを規定していない**。
- コメント数順は容易に衝突する。評価シナリオ04-3 では 11 件中 9 件がコメント0件になり、
  **9件が同順位**になる。最終更新日時順も、秒精度なら連続作成で衝突しうる。
- タイブレークが不定だと、**ページをまたいだときに同じスレッドが2回出たり消えたりする**
  （C-13「ページをめくってもソート順が維持」の違反）。DB の `ORDER BY` は同値行の順序を保証しない。

## 提案

1. **全ソートキーに第2キーとして ID を付け、全順序にする。** モデルの `leOf` は
   `(主キー, id)` の辞書式順序で比較する。実装でも `ORDER BY <key>, id` を必ず書く。
2. **タイムスタンプはミリ秒以上の精度**を持たせ、作成日時が実質衝突しないようにする。
   モデルでは `Db.clock` を単調増加の論理時刻とし、状態変更ごとに 1 進めることで衝突を構造的に排除した。
3. **保存は UTC、表示はローカル**（またはアプリ固定のタイムゾーン）とする。

## 決定（2026-07-19 ユーザー判断）

1. **全ソートキーに第2キーとして ID を付け、全順序にする。** 実装では必ず `ORDER BY <key>, id` と書く。
2. **タイムスタンプはミリ秒以上の精度**を持たせる。
3. **保存は UTC、表示は JST**（`timestamptz` 相当の型を使う）。
4. **一覧の初期表示は作成日時の降順。**

## 影響

### 第2キーは必須（この決定の核心）

シナリオ04-3 では 11 件中 **9 件がコメント0件で同順位**になる。SQL の `ORDER BY` は
**同値行の順序を保証しない**ため、第2キーが無いとページをまたいだときに
同じスレッドが 2 回出たり、1件も出ずに消えたりする（C-13「ページをめくってもソート順が維持」の違反）。
これは**評価で確実に踏む**（シナリオ04-3 が 11 件＝2ページにまたがる状態を作る）。

- 形式モデルの `Query.leOf` は 4 つのソートキーすべてで `(主キー, id)` の辞書式順序になっている。
- 実装の SQL でも `ORDER BY comment_count DESC, id` のように**必ず第2キーを書く**こと。
- 第2キーを ID にするか作成日時にするかは任意。**決定的でありさえすればよい。**

### 実時刻は単調とは限らない

モデルの `Db.clock` は単調増加する論理時刻で、`tick` により作成日時が構造的に衝突しない。
**実装は実時刻を使うのでこの保証がない**（NTP による巻き戻し、同一ミリ秒内の連続作成）。
順序の一意性は**もっぱら ID による第2キーに依存する**。

### タイムゾーン・表示形式

- DB: `2026-07-19T05:30:00Z`（UTC）／ 画面: `2026-07-19 14:30`（JST）
- 評価は日時の値そのものを検証しないが、AC09-2「作成日時が表示されている」と
  シナリオ04-2-3「一番古いスレッドがトップ」の目視確認は行う。**表示値から順序が読める形式**にすること
  （秒まで出すか、日付だけにするかは任意だが、同日内の複数スレッドを区別できる粒度が要る）。
- `timestamp without time zone` を使うと変換の行き違いが出やすい。**タイムゾーン付きの型**を選ぶこと。
