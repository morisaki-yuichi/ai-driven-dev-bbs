# 0036 必須フィールド欠落POSTが403でなく422になる件を受容する

- 関連論点: D05（CSRF対策の方式、`dev-docs/requirements-analysis.md` §3）。
- 関連原典: なし（**原典は一言も述べていない**。`docs/evaluation/scenarios/` はいずれも422/403の区別に言及していない）。
- 影響範囲: `app/src/web/csrf.rs`（`CsrfForm<T>::from_request`）。F05以降、CSRFトークン検証を伴うすべてのPOSTハンドラ（スレッド作成・コメント作成・削除系・ログイン・登録・プロフィール編集）。

## 背景（原典が何を言い、何を言っていないか）

**[事実]** decision 0021でCSRF対策は二重送信トークン方式に確定しており、決定6として「拒否経路ではトランザクションを開かない」ことが定められている。

**[空白]** 原典は、CSRFトークン不一致とリクエストボディの構造的な不備（必須フィールド欠落）が競合した場合にどちらを優先してどのステータスを返すべきかについて何も述べていない。

**問題の実体**: `app/src/web/csrf.rs`の`CsrfForm<T>::from_request`は次の順で動く。

1. Cookieから`csrf_token`を取得する。
2. `Form::<T>::from_request`でリクエストボディ全体をserdeでデシリアライズする。
3. (2)が成功して初めて`tokens_match`を呼び、CSRFトークンの一致を検査する。

必須フィールドが1つでも欠けたPOSTでは(2)でaxumの`FailedToDeserializeForm`となり422が返る。CSRF専用の403ロジック(3)は一度も実行されない。

F05実装時にこの挙動が最初に観測されて以来、F05→F12・持ち越しサイクルを通じて6回連続で持ち越されてきた。今回のサイクルでS1調査により根本原因を特定し、コードは変更せず受容する判断を記録することで持ち越しを打ち切る。

decision 0021の決定6（拒否経路ではトランザクションを開かない）はこの422経路でも満たされている。`Form::from_request`が失敗した時点でハンドラ本体（トランザクションを開く箇所）が一切呼ばれないため。

## 選択肢

1. **現状(422)を受容する**（コード変更なし）。
2. `CsrfForm`を2段構成にし、生ボディから`csrf_token`フィールドだけを先に取り出して検証してから、残りをフォーム型へデシリアライズする。CSRF検証が常に最優先で走るようになるが、抽出ロジックが複雑化し、`HasCsrfToken`トレイトの意味（「デシリアライズ済みの型が持つ」)も変わってしまう。
3. `Form::from_request`のデシリアライズ失敗を`AppError`経由の専用エラーページに写し替える。UIの一貫性（422という素のaxumレスポンスでなく、アプリ側の見た目のエラー画面になる）は改善するが、「CSRF不一致と構造不備のどちらを優先するか」という問題設定自体は解消しない。

## 提案（および理由）

選択肢1（現状を受容）を提案する。

- **実害が薄い**: 真のクロスオリジンCSRF攻撃は、`same_origin_guard`ミドルウェアが`CsrfForm`より外側にあり、Origin/Refererの不一致を先に403として撥ねる（`app/src/web/mod.rs`のミドルウェア順序）。422が起きるのは「同一オリジンからの、必須フィールドが丸ごと欠けたPOST」という限定的な状況のみで、これはCSRF攻撃の成立条件（悪意あるサイトからの偽装送信）とは別物。
- **agent-browser経由では到達しにくい（ただし現行テンプレート限定の事実）**: 実ブラウザのフォーム送信では`<input>`要素が存在する限り値が空でもキー自体は送信される——これは**チェックボックス・ラジオ・`disabled`属性付き・`name`無しの要素には成り立たない**（これらは値が無ければキーごと送信されない）。現在この主張が成立しているのは、CSRF保護対象のフォームのテンプレートにそれらの要素が存在しないからにすぎず、一般に常に成り立つ性質ではない。422を起こすには、現状ではDOMから`<input>`要素ごと除去するなどの非定型的なJS操作が要るが、これはテンプレートの現状に依存した結論である（下記「再検討を要する条件」参照）。`docs/evaluation/scenarios/`のいずれのシナリオもこの経路を踏まない。
- **decision 0021の決定6は満たされている**（上記「背景」参照）。DB書き込みが起きる余地は無い。
- 選択肢2・3はいずれも、原典が要求していない振る舞い（422/403の区別）のために実装の複雑さを増やすコストに見合わない。

## 決定（2026-07-20 ユーザー判断）

選択肢1を採用する。**コードは変更せず、この挙動（必須フィールド欠落POSTは403ではなく422になる）を受容する。**

この判断が再検討を要する条件:

- `docs/evaluation/scenarios/`に422/403を明示的に区別するシナリオが追加された場合。
- `same_origin_guard`を外す（または`CsrfForm`より内側に移す）判断が別途下された場合。この場合、`same_origin_guard`が担っていた「クロスオリジンPOSTを先に撥ねる」防御が失われ、422経路の実害が変わりうる。
- **CSRF保護対象のフォームにチェックボックス・ラジオ・`disabled`属性付き入力・`name`無し入力を追加する変更が入った場合**。これらの要素は値が無いと送信時にキーごと送信されない（上記「agent-browser経由では到達しにくい」の前提が崩れる）ため、非定型的なJS操作を介さずとも必須フィールド欠落と同じ422経路に自然に入りうる。
- **リクエストのContent-Typeが`application/x-www-form-urlencoded`以外になる変更が入った場合**（ファイルアップロード導入など）。`Form::from_request`は非urlencodedボディを拒否するため、CSRF検証（本decisionが前提とする「(2)のデシリアライズが先に走る」経路）に入る前に、別の機構によって同種の422が発生する。

## 影響

- `app/src/web/csrf.rs`のコード自体への変更は無い。
- 既存の結合テスト（各`*_test.rs`のCSRF関連ケース）はそのまま有効。新規テストの追加は行っていない（挙動を変えていないため）。
- 将来この挙動を変える場合は、本decisionを更新し（新規番号は起こさず）、変更履歴に追記する。
