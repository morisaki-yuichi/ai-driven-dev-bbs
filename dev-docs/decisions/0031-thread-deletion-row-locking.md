# 0031 F06スレッド削除における行ロック方針(F08のTOCTOU対策パターンを移植できない場合の代替)

- 関連論点: 新規（F06実装レビューで判明。F08のTOCTOU対策パターンをF06にそのまま適用したところ、実測で500エラーが再現した）。
- 関連原典: AC06-2（コメントが1件以上あるスレッドは削除できない。削除済みコメントも数える）、C-06、C-08（削除は不可逆）、decision 0002（1リクエスト＝1トランザクション）、decision 0014（スレッド削除は物理削除）。
- 影響範囲: `app/src/db/threads.rs`（`find_by_id_for_update` / `find_by_id_for_share` / `delete`）、`app/src/db/comments.rs::delete`（対比対象）、今後「削除・更新の可否が別テーブルの状態に依存する」実装全般。

## 背景（原典が何を言い、何を言っていないか）

**[事実]** 原典（`docs/`）はAC06-2として「コメントが1件でもあるスレッドは削除できない」ことのみを要求し、同時実行下での競合制御の実装手段には一言も触れていない。

**[事実]** F08（コメント削除、`db/comments.rs::delete`）では、TOCTOU対策として「条件付き1文＋`rows_affected`判定」パターンを確立した。`update comments set deleted_at = now() where id = $1 and deleted_at is null` のように、削除可否の条件（`deleted_at is null`）を更新対象の行自身（`comments`の当該行）に持たせることで、PostgreSQLの`EvalPlanQual`（更新対象行が同時に書き換わった場合、その行に限り条件を再評価する機構）が働き、1文だけで競合を閉じられた。

**[空白／実測で判明した事実]** F06（スレッド削除）でこのパターンをそのまま移植し、`delete from threads where id = $1 and not exists (select 1 from comments where comments.thread_id = $1)` という1文だけで実装したところ、レビューで次の不具合が実測された。F06の削除可否条件（コメントの有無）は更新対象行（`threads`）自身ではなく**別テーブル（`comments`）の状態**を見るサブクエリであるため、`EvalPlanQual`の再評価対象にならない。

実測した競合の経路は次のとおり。

1. 削除トランザクションが`delete`文を実行。`not exists`サブクエリは**文開始時点のスナップショット**で評価され、別トランザクションが未コミットで挿入中のコメントを見落として「0件」と判定する。
2. その直後、`threads`行自体のロック取得で削除側が待たされる（挿入側がFK制約の検証のために`comments_thread_id_fkey`経由で`threads`行に`for key share`を保持しているため）。
3. 挿入側がコミットすると削除側のロック待ちが解け、`delete`が実行されてしまう。
4. 文末の外部キー制約検査（`comments_thread_id_fkey`、`on delete`指定なし＝`no action`）がここで初めて矛盾を検出し、`23503`（外部キー違反）の`Err`を返す。孤児コメントは生じない（DBの整合性は保たれる）が、呼び出し側は`Ok(false)`ではなく`Err`を受け取り、ユーザー向けの「コメントがあるので削除できません」ではなく**画面に500が出る**。

**[事実／副次的に判明したこと]** F07（コメント作成）のサイクルで「スレッド存在確認をトランザクション内に移す」という緩和を先に入れていたが、これは上記の競合を防げていなかった。素の`select`はPostgreSQLでは行をロックしない。トランザクション内に読み取りを置くことと、その読み取りが行をロックすることは別の性質であり、前者だけでは競合窓は閉じない。

## 選択肢

### (a) F08と同じ「条件付き1文」のみで済ませる

`not exists`サブクエリを`delete`と同一文に書けば十分、とみなす。

- 利点: 実装が単純で、F08とコードの形が完全に揃う。
- 欠点: 上記のとおり実測で500エラーが再現する。条件が別テーブルを参照する場合、`EvalPlanQual`による安全性がそもそも成立しないため、この選択肢は棄却する。

### (b) 明示的な行ロックで直列化する（採用）

削除対象の`threads`行を`for update of threads`で排他ロックしてから`delete`を実行する。競合しうる側（コメント作成）は`for share of threads`で共有ロックを取り、両者を同じ行のロックを介して直列化する。

- 利点: `for update`は同一行への同時書き込みを待たせるため、削除中は他のコメント挿入（`for share`によりロックが競合する）をブロックできる。ロック取得後に実行する`delete`文は、待ちが解けた時点の最新スナップショットでコメント有無を判定するため、取りこぼしが起きない。
- 欠点: F08と実装の形が完全には揃わなくなる（ロック取得の追加ステップが要る）。ロック粒度の見積もりを都度考える必要がある。

## 提案（および理由）

**(b) を採る。**

条件付き1文＋`rows_affected`判定という「形」自体は維持しつつ、削除条件が別テーブルを参照する場合に限り、その前段で明示的な行ロックを追加する。理由は、(a)がそもそも安全性の前提（`EvalPlanQual`が働く条件）を満たしていないと実測で確認されたため。ロックを追加コストとして払うのは、条件が別テーブルにある場合のみでよく、F08のように条件が対象行自身に閉じる場合は引き続き(a)のパターンで十分（該当決定を覆すものではない）。

## 決定（2026-07-20 ユーザー判断）

- F08（コメント削除、`db/comments.rs::delete`）で確立した「条件付き1文＋`rows_affected`判定」によるTOCTOU対策は、**条件が更新対象行自身に付く場合にのみ成立する**（`update comments ... where id = $1 and deleted_at is null`）。PostgreSQLの`EvalPlanQual`による条件再評価は更新対象の行自身にしか働かず、別テーブルを見るサブクエリには効かない。
- **F06（スレッド削除）にはこのパターンを移植できない。** `delete from threads where id = $1 and not exists (select 1 from comments where comments.thread_id = $1)` は、READ COMMITTEDでは`not exists`サブクエリが文開始時点のスナップショットで評価されるため、別トランザクションの未コミットのコメントを見落とす。実測した経路: サブクエリが「0件」と判定 → `threads`行のロック取得で待たされる（相手がFKの`for key share`を保持） → 相手のコミット後に削除が実行される → 文末のFK検査が`23503`を投げ、画面には500が出る。
- したがって、**条件が別テーブルにある場合は明示的な行ロックで直列化する**。F06のスレッド削除は`find_by_id_for_update`（`for update of threads`）で対象行を先にロックしてから`delete`を実行する。F07のコメント作成は`find_by_id_for_share`（`for share of threads`）で対になる共有ロックを取り、両者を直列化する。
- **重要な副次事実**: F07のサイクルで入れた「スレッド存在確認をトランザクション内に移す」という緩和は、素の`select`が行をロックしないため実際には無効だった。トランザクション内に置くことと行をロックすることは別である。
- 将来、削除・更新の可否が別テーブルの状態に依存する同種の判断が必要になった場合（今回のスレッド削除↔コメントの存在検査のような1対多の関係）は、条件付き1文だけでは閉じないことを前提にし、対象行（または関連行）を明示的に`for update`/`for share`でロックして競合する操作同士を直列化する設計を最初から検討すること。

ユーザーが承認した。

## 影響

- `app/src/db/threads.rs`: `find_by_id_for_update`（`for update of threads`、F06の削除前に必ず通す）、`find_by_id_for_share`（`for share of threads`、F07のスレッド存在検査に使う）、`delete`（`not exists`による条件付き1文は維持しつつ、呼び出し前提として行ロックを要求）が、この決定の実装そのもの。
- `app/src/db/comments.rs::delete`（F08）は対比対象として変更なし。条件が対象行自身に閉じるため、引き続き「条件付き1文」のみで安全。
- `app/tests/thread_deletion_test.rs`の`delete_is_graceful_when_a_comment_is_inserted_by_an_overlapping_transaction`が、行ロックを外すと失敗する回帰テストとしてこの決定を担保する。
- 関連decision: 0002（1リクエスト＝1トランザクション。行ロックは同一トランザクション内で完結する前提を置く）、0014（物理削除。削除対象が`threads`の行そのものであることが今回の競合の起点になっている）。
- 今後、削除・更新条件が別テーブルの状態に依存する機能を実装する際は、まず対象行自身に条件を閉じられないか検討し、閉じられない場合はこの決定に従って明示的な行ロックを設計に含めること。
