# 0032 F11検索でのLIKEワイルドカード文字(%・_)のエスケープ

- 関連論点: D07（新規、F11実装中に判明）
- 関連原典: AC11-2。原典は照合方式の内部実装を規定していない（なし）。
- 影響範囲: `app/src/db/threads.rs::search`、`app/src/domain/query.rs::escape_like_pattern`

## 背景（原典が何を言い、何を言っていないか）

- **[事実]** decision 0011 は検索方式を「素朴な部分文字列一致（`LIKE '%kw%'` 相当）」と定め、
  モデル側の `Bbs.Query.containsSubstr` は純粋な部分文字列判定であり、`%`・`_` を特別扱いしない
  （`"50%"` というキーワードは文字どおり `50%` という3文字の並びを探す）。
- **[空白]** 原典・decision 0011 とも、`LIKE` の実装に落とし込む際にワイルドカード文字
  （`%` = 任意の0文字以上、`_` = 任意の1文字）をどう扱うかには触れていない。

## 選択肢

1. ユーザー入力をそのまま `LIKE '%' || kw || '%'` へ埋め込む（バインドパラメータ経由）。
2. ユーザー入力中の `%`・`_`・エスケープ文字自身(`\`)をエスケープしてから埋め込む。

## 提案（および理由）

**選択肢2を採る。**

バインドパラメータはSQLインジェクション（値の型としての注入）を防ぐが、`LIKE` パターン内での
`%`・`_` の**意味**までは中和しない。エスケープなしでは、ユーザーが `50%` や `a_b` を検索すると
`%`・`_` がワイルドカードとして解釈され、`containsSubstr`（Leanモデル）や
`domain::query::contains_substr`（Rust側の同型実装、ともに単純な部分文字列判定）が返す結果と
食い違う。例えば `50%` を検索すると、意図せず `500円` のような文字列にもヒットしてしまう。

## 決定（2026-07-20 ユーザー判断）

`domain::query::escape_like_pattern` でユーザー入力中の `\`・`%`・`_` をバックスラッシュで
エスケープしたうえで `%...%` に包み、`db::threads::search` の `LIKE ... ESCAPE '\'` へ渡す。
処理順序は「元の文字列を1回だけ走査し、該当文字ごとにエスケープ文字を1つ前置する」形にし、
生成した `\` を再度エスケープしてしまう二重エスケープを避ける。

## 影響

- `app/src/domain/query.rs::escape_like_pattern`（純粋関数、単体テスト済み）と
  `app/src/db/threads.rs::search`（`LIKE`パターンの組み立てにこの関数を使う）が対応箇所。
- Leanの`containsSubstr`をオラクルとする単体テスト（`escape_like_pattern_escapes_percent_and_underscore`
  等、`app/src/domain/query.rs`）と、実際の`LIKE`検索が期待どおりリテラル一致になることを確認する
  結合テスト（`app/src/db/threads.rs`の`#[sqlx::test]`）の両方でカバーする。
- decision 0011・0012 の照合方式（大文字小文字区別・正規化なし・削除済みコメント除外）自体は
  変更しない。本決定はその**実装上の落とし穴**への対応であり、方式そのものへの変更ではない。
