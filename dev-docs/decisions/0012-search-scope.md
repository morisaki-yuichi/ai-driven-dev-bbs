# 0012 検索対象の範囲（タイトル／削除済みコメント）

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: D08, D09
- 関連原典: AC11-2、AC11-4、C-14、AC11-3（ヒット箇所へ自動スクロール）
- 影響範囲: `Bbs.Query.search`、`Bbs.Query.DeletedCommentSearchPolicy`

## 背景

### (a) 削除済みコメント（D08 / AC11-4）

- **[事実]** AC11-4 は「固定文言 `＜このコメントは削除されました＞` は検索対象にならない
  （**あるいはヒットしても遷移時に矛盾がない**）ように制御されていること」。
  **原典が明示的に2方式を許容している**、珍しい条項。
- **[解釈]** 「固定文言だけを対象外にする」では不十分になりうる。削除済みコメントの**元本文**が
  ヒットして詳細へ飛ぶと、画面には固定文言しかなく、スクロール先の根拠が失われる（AC11-3 と衝突）。

### (b) タイトル（D09）

- **[事実]** `issues/11` は検索対象を「全スレッドの**本文**、および全コメントの**本文**」と明記。
  **タイトルへの言及がない。**

## 選択肢

(a) について:
1. **論理削除コメントを元本文ごと検索対象から外す**（`excludeDeleted`）。
2. 削除済みも検索対象に含め、ヒット時は固定文言の表示位置へスクロールさせる（`includeDeleted`）。

(b) について:
1. タイトルを検索対象に**含めない**（原典の列挙どおり）。
2. 含める（UX としては自然）。

## 決定（2026-07-19 ユーザー判断）

- **(a) 選択肢1: 論理削除コメントを元本文ごと検索対象から外す。**
  固定文言だけでなく、削除前の本文もヒットしない。
- **(b) 選択肢1: タイトルは検索対象に含めない。**

検索対象は `threads.body` と、**未削除の** `comments.body` のみ。

```sql
-- コメント側
WHERE c.deleted = false AND c.body LIKE '%kw%'
```

| 検索語（シナリオ03 で「テストコメント1」を削除済みの状態） | 結果 |
| :--- | :--- |
| `テストコメント1`（削除済みコメントの元本文） | ヒットしない |
| `削除されました`（固定文言） | ヒットしない |
| `テストコメント2`（未削除） | ヒットする |

## 影響

### 形式モデルを決定内容に固定した

`DeletedCommentSearchPolicy` を**削除**し、`search db kw` / `hitIn db kw` / `searchableComments cs`
の形に確定させた（両方式を残す理由がなくなったため）。

- `Invariant.excludeDeleted_no_deleted_hit` → **`Invariant.no_deleted_hit` に改名**。
  「返る `Hit` が指すコメントは必ず未削除」。これが AC11-4 充足の証明対象になる。
- `Invariant.hit_is_reachable`（ヒット箇所は詳細画面に実在する要素を指す）は**引き続き有効**。
  選択肢2 を採っていたらこちらが必須になっていたが、選択肢1 でも成り立つべき性質なので残す。
- `Bbs/Scenario.lean` に回帰確認を3件追加（元本文・固定文言がヒットしないこと、未削除はヒットすること）。

### AC11-3（自動スクロール）との整合が取れた

選択肢1 により、**ヒット箇所には必ず検索語が実際に表示されている**。
選択肢2 だと「詳細へ飛んでスクロールした先に `＜このコメントは削除されました＞` しかなく、
検索語がどこにも見当たらない」状態になり、原典上は許されるものの利用者からは不審に映る。

### タイトルを含めないことの帰結

シナリオ04 はスレッドA/B のタイトルを指定していないため、**どちらを選んでもシナリオは通る**。
含めないことで、`Hit` に `inTitle` を足す必要がなくなり、スクロール先の場合分けが
「本文（最上部）」と「コメント（アンカー）」の2通りに収まる（D19 / decision 0013 と整合）。
