# 0023 フォームに novalidate を付け、ブラウザネイティブのクライアント側検証を採用しない

- 関連論点: なし（新規。F01実装レビューで顕在化）。
- 関連原典: H-02, H-12（`dev-docs/harness-invariants.md`）。C-02 / C-03 / C-04（検証内容そのものは規定されているが、**どこで検証するか**は規定なし）。
- 影響範囲: `app/templates/register.html`（および今後の全フォームテンプレート P02・P04〜P06）、`dev-docs/ui-ux-guidelines.md` §3 の二重バリデーション要件。

## 背景（原典が何を言い、何を言っていないか）

**[事実]** `dev-docs/ui-ux-guidelines.md` §3 は「すべてのユーザー入力はクライアント側とサーバ側の**両方**で検証する」と定めている。クライアント側は「UX向上のための即時フィードバック」と位置づけられ、サーバ側検証の省略を禁じる文脈で書かれている。

**[事実]** 同 §6（`agent-browser` での操作可能性、H-02）は「**状態変化は画面内のDOMに反映する**」「AIが自然言語で読み取る前に消える表現は避ける」と定めている。評価はAIが `agent-browser` でUIを操作し、画面上の表示を自然言語で観測して行われる（H-12）。

**[空白]** 原典（`docs/`）は検証をクライアント側で行うことを一切要求していない。要求しているのは受け入れ基準（AC01-2〜AC01-4 等）が示す**結果の観測可能性**だけである。また `ui-ux-guidelines.md` §3 も、「クライアント側検証」を **HTML標準の制約属性（`required` / `minlength` / `maxlength` / `pattern`）によるネイティブ検証**で実現するのか、JavaScriptで実現するのかを指定していない。

## 選択肢

### (a) `novalidate` を付けず、HTML制約属性によるネイティブ検証を有効にする

- 利点: `ui-ux-guidelines.md` §3 の字面に最も忠実。追加コードがゼロ。JSを一切書かずに即時フィードバックが得られる。
- 欠点（決定的）: ネイティブ検証の失敗は**バリデーションバブル**（ブラウザが描く吹き出し）として表示される。これは**DOMツリーの一部ではなく**、アクセシビリティツリーにも安定して現れず、フォーカス移動やクリックで即座に消える。`agent-browser` はこれを読めない可能性が高い。結果として、AC01-2〜AC01-4 が要求するエラー提示が「画面上に何も出ていない」ように観測され、**送信自体もブラウザに阻止されるためサーバ側のエラーテキストにも到達しない**。§6 が禁じている「AIが読み取る前に消える表現」そのものになる。
- さらに: 制約属性の判定基準はサーバ側と揃えられない箇所がある。`maxlength` は UTF-16 コードユニット単位で数えるが、C-03 の判定は decision 0003 によりコードポイント単位である。C-02 のパスワード強度（複数観点の内訳を同時に出す、decision 0006）は単一の `pattern` では表現できず、`pattern` 違反は「形式が正しくありません」という単一の汎用メッセージにしかならない。

### (b) `novalidate` を付け、検証をサーバ側に一本化する（エラーは再描画したHTMLのテキストとして出す）

- 利点: すべてのエラーが**通常のDOM要素**（`.field-error` および共通メッセージエリア）として画面に残り、`agent-browser` が自然言語で読める。判定基準はサーバ側の1箇所（`domain/validation.rs`）だけになり、decision 0003・0004・0006 との乖離が構造的に起きない。SSR/MPA（decision 0008）なので送信のたびにページが再描画され、フィードバックの遅延は1往復に収まる。
- 欠点: `ui-ux-guidelines.md` §3 の「両方で検証する」という文言から意図的に逸脱する。ネットワーク往復なしの即時フィードバックが失われる。

### (c) `novalidate` を付けたうえで、JavaScript で独自のクライアント側検証を実装する

- 利点: §3 を字義通り満たしつつ、エラーを通常のDOM要素として出せる（バブル問題を回避できる）。
- 欠点: decision 0008 が確定させた SSR/MPA の前提を越えてクライアント側ロジックを導入することになる。decision 0003・0004・0006 の判定規則を Rust と JS に**二重実装**することになり、両者の乖離が新たな欠陥源になる（§3 自身が「クライアント側検証はサーバ側と矛盾しない」ことを要求しており、これは二重実装では常に守り続けるコストがかかる）。教材アプリの規模に対して割に合わない。

## 提案（および理由）

**(b) を採る。**

決め手は「エラーが `agent-browser` から読めるかどうか」である。この評価環境では、**エラーが画面上のテキストとして観測できないことは、エラー表示が存在しないことと等価**であり、AC01-2〜AC01-4 の不合格に直結する。ネイティブ検証は「即時フィードバック」という利点と引き換えに、この評価環境で最も重要な性質（観測可能性、H-02 / H-12）を壊す。

(c) は §3 を字義通り満たすが、判定規則の二重実装というコストと、SSR/MPA 前提（decision 0008）を越える設計判断を要する。本アプリの規模ではサーバ往復1回の遅延は実害がなく、二重実装のリスクの方が大きい。

なお **`required` / `minlength` / `maxlength` 属性そのものは残す**。`novalidate` によって送信時のブロックは無効化されるが、これらの属性はセマンティクス（スクリーンリーダーへの必須項目の伝達、入力欄の最大長のヒント）として意味を持ち続けるためである。

**（decision 0037 による上書き）** 表示名欄の `maxlength` は、その後 decision 0037 で撤去された。`maxlength` は `novalidate` の影響を受けない別のブラウザ機構（入力イベント自体を止める）であり、これが残ったままだと `agent-browser` が16文字目を入力できず、本決定が解決したはずの観測性の問題が形を変えて再発することが最終ゲートで判明したためである。ユニークID・パスワード欄の `required` / `minlength` は本決定のまま維持されている。

## 決定（2026-07-19 ユーザー判断）

**すべてのフォームに `novalidate` を付け、ブラウザネイティブのクライアント側検証を採用しない。入力検証はサーバ側（`domain/validation.rs`）に一本化し、エラーは再描画したHTML内のテキスト（該当フィールド付近の `.field-error` と、共通メッセージエリアの要約）として提示する。**

これは `dev-docs/ui-ux-guidelines.md` §3 の二重バリデーション要件からの**意図的な逸脱**である。トレードオフとして次を受け入れる。

- **失うもの**: ネットワーク往復なしの即時フィードバック。ユーザーは送信して初めてエラーを知る。
- **得るもの**: すべてのエラーが `agent-browser` から読めるDOMテキストになること。判定基準がサーバ側の1箇所に集約され、decision 0003（コードポイント単位の文字数）・decision 0004（トリム後の空判定）・decision 0006（複数違反の同時提示）との乖離が構造的に起こらないこと。
- **失われないもの**: §3 の本来の主眼である「サーバ側検証を省略しない」は完全に満たされる。むしろサーバ側が唯一の判定者になることで強化される。

## 影響

- `app/templates/register.html` の `<form ... novalidate>` はこの決定に基づく。今後追加するフォーム（P04 スレッド作成・P05 コメント・P06 プロフィール編集）にも同様に `novalidate` を付ける。
- `dev-docs/ui-ux-guidelines.md` §3 を読む際は、本決定が同節への既知の例外であることを前提とする。§3 のうち「サーバ側検証を省略しない」「バリデーションエラーは該当フィールド付近に赤系テキストで表示する」は引き続き有効。
- HTML の `maxlength` と C-03 のコードポイント単位判定の乖離は、`novalidate` により**送信がブロックされなくなる**ため実害が縮小するが、入力欄への打ち込み自体は UTF-16 単位で頭打ちになる。この点は `app/templates/register.html` に Why-not コメントとして残してある（評価シナリオのデータはすべて BMP 内であり乖離しない）。
- decision 0006（複数エラーの同時提示）は、サーバ側一本化によってはじめて素直に実現できる。本決定はその前提を裏打ちする。
