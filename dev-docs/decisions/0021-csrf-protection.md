# 0021 D05: CSRF対策の方式（Origin検証 + セッション非依存の二重送信トークン）

- 関連論点: D05（CSRF・セキュリティ対策一般）。`dev-docs/requirements-analysis.md` §3 D05。
- 関連原典: **なし（原典は一言も述べていない）**。`docs/` および `README.md` にCSRF・XSS・レート制限の記述は一切ない。制約として効くのは評価ハーネス側の H-07（秘匿キー）・H-13（clone直後）・H-02（`agent-browser` 操作性）。
- 影響範囲: `app/src/web/`（新規 `csrf.rs`、`middleware.rs`、`error.rs`、`params.rs`）、`app/src/domain/`（純粋な検証関数）、全フォームテンプレート（hidden input）、`app/templates/`。**DBスキーマは変更しない**（decision 0019 に影響なし）。

## 背景（原典が何を言い、何を言っていないか）

**[事実]** 原典にCSRFへの言及は一切ない（D05）。シナリオ02-2-4「削除URLに直接アクセス→エラーまたはリダイレクト」は権限チェックの要求であり、CSRF対策の要求ではない（D05 の解釈の注意）。

**[事実]** CLAUDE.md「セキュリティ必須要件」が、破壊的操作をPOSTで受け、状態変更エンドポイントにCSRFトークン検証を入れることを恒久ルールとして要求し、方式の確定を D05 に委ねている。

**[空白]** 対策の方式（同期トークン / 二重送信Cookie / Originヘッダ検証）、トークンの生成・保存・検証の場所、未ログイン状態のPOST（`POST /register`・`POST /login`）を保護対象に含めるか、検証失敗時のHTTP応答と画面表示。

方式選定を拘束する既決事項は次の4点である。

1. **秘匿キーを使えない（H-07 / H-13）**。評価者はclone直後にセットアップを走らせる。人間の手入力を要する秘密鍵を必須の環境変数にすると起動できない。`.env.example` は「秘匿キーは無い想定」（foundation-plan §2）であり、実際に現在の `.env.example` はDB接続情報のみを持つ。**→ HMAC署名付きステートレストークンは採用できない**（アプリ起動時にランダム鍵を自動生成する手もあるが、再起動でトークンが一斉に無効化され、H-08〜H-10の逐次実行中の再起動耐性を落とす）。
2. **JavaScriptを使わない（decision 0008 の SSR/MPA 限定）**。**→ 古典的な二重送信Cookie（JSがCookieを読んでフォームに書き込む）はそのままでは使えない**。ただしサーバがGET応答の時点で同じ値をhidden inputに描画すれば、JSなしで成立し、かつCookieを `HttpOnly` のままにできる（古典的方式より強い）。
3. **`sessions` テーブルは `id text primary key` / `user_id bigint not null` の2列**（`app/migrations/0001_init.sql`、decision 0019）。**未ログイン状態を表現できる行が無い**。セッション同期トークンを `POST /register` / `POST /login` にも適用するには、`user_id` をNULL許容にするか別テーブルを足すことになり、decision 0019 のスキーマに手を入れる必要が生じる。
4. **セッションCookieは既に `SameSite=Lax`**（`app/src/web/cookies.rs`）。これ自体がクロスサイトのPOSTにCookieを乗せない第一防壁として効いている。本決定はその上の多層防御である。

## 選択肢

### (a) `sessions` に紐づく同期トークン（synchronizer token）

`sessions` に `csrf_token` 列を足し、フォームのhidden inputに埋め、POSTでセッション行の値と照合する。

- 利点: 最も堅い。トークンがサーバ側状態に束縛され、Cookie注入で偽造できない。
- 欠点: **未ログインPOSTを保護できない**。`POST /register` / `POST /login` にはセッション行が無い。カバーするには上記3のスキーマ変更（`user_id` NULL許容 or `pre_sessions` テーブル）が要り、decision 0019 を開き直すうえ、ログイン成功時に「仮セッション→本セッション」の昇格処理が増える。1リクエスト1トランザクション（0002）の下で扱えないほどではないが、得られる安全性に対して機構が重い。
- 欠点: ログイン画面のGETごとにDB書き込みが発生する（トークン発行）。未認証GETがDBを書くのは経路として好ましくない。

### (b) 二重送信Cookie（double-submit cookie）

セッションとは独立の `csrf_token` Cookieを発行し、同じ値をフォームのhidden inputへサーバが描画、POSTで両者を照合する。

- 利点: **ログイン状態に依存しない**。`/register`・`/login`・`/logout`・認証必須の全POSTを1つの機構で同じように扱える。DBもスキーマ変更も秘匿キーも不要（H-07 / H-13 に完全に無害）。
- 利点: サーバがhidden inputを描画するのでJS不要（decision 0008 と整合）。Cookieを誰も読まないので `HttpOnly` を付けられる。
- 欠点: 同一サイトにCookieを書き込める攻撃者（サブドメイン乗っ取り、HTTP上のMITM）は既知の値を注入して突破できる。本プロジェクトは単一ホスト・サブドメインなし・評価環境はローカルHTTPであり、この攻撃面は実質的に存在しない。ただし単独では原理的に (a) より弱い。

### (c) Origin / Referer ヘッダ検証

状態変更メソッドのリクエストで `Origin`（無ければ `Referer`）のオリジンがリクエストの `Host` と一致することを要求する。

- 利点: 完全にステートレス。トークンもCookieもテンプレート変更も不要。設定値も不要（期待オリジンを固定値で持たず、リクエスト自身の `Host` と突き合わせれば環境非依存になり、H-07 / H-13 に無害）。
- 利点: **ヘッダだけを見るのでミドルウェアに置ける**。ハンドラ側の書き忘れが起こりえない。
- 欠点: 単独では、ヘッダを送らないクライアント・将来のプロキシ構成で誤判定しうる。`Origin` も `Referer` も無いリクエストを拒否すれば安全側だが、それだけを唯一の防壁にするのは心もとない。

### (d) (b) + (c) の組み合わせ

Originを見るミドルウェアを全状態変更リクエストの中央の網とし、その内側でフォーム単位の二重送信トークンを照合する。

## 提案（および理由）

**(d) を採る。** 理由は次のとおり。

- **本論点の核心である未ログインPOSTを、例外処理なしで扱えるのは (b) 系だけである。** (a) はスキーマ変更と仮セッションの昇格という機構を、認証前の2エンドポイントのためだけに要求する。「単純なCRUDに過剰な層・間接を作らない」（CLAUDE.md 実装方針）に照らして割に合わない。
- (b) の唯一の実質的な弱点（同一サイトへのCookie注入）は、(c) が別軸で塞ぐ。攻撃者はCookie注入に加えて正規オリジンからのリクエスト送出も必要になる。逆に (c) の弱点（ヘッダ欠落）は (b) が塞ぐ。**二つの機構の失敗様態が独立している**ことが、組み合わせを選ぶ理由である。
- (c) をミドルウェアに置くことで、ハンドラ実装者がトークン検証を書き忘れても最低限の防御が残る。**「忘れても穴が開かない層」を1枚持つ**ことが、フェーズ2以降にハンドラが増えていく本プロジェクトでは実装上の安全マージンになる。
- 全経路が秘匿キー・DB書き込み・JSのいずれも要さない。H-07 / H-13 / decision 0008 のすべてと無条件に整合する。

### 未ログインPOSTの扱い（本論点の核心）

**`POST /register` と `POST /login` も保護対象に含める。** 発行と検証は次のとおり。

- **発行**: トークンはセッションではなく**ブラウザ**に結び付ける。リクエスト処理の前段で `csrf_token` Cookieの有無を見て、無ければCSPRNGで生成し、リクエスト拡張（extension）に載せたうえで応答に `Set-Cookie` する。ログイン画面・登録画面のGETはこの経路を通るので、**セッションが存在しない時点でトークンが手に入る**。ここがセッション同期トークン方式との決定的な差である。
- **検証**: POSTでCookie値とフォーム値を定数時間比較する。ログイン状態を参照しない。
- **ローテーション**: ログイン成功時とログアウト時に新しい値を発行し直す（セッション固定攻撃と同種のリスクを避ける衛生措置）。認証状態が変わる瞬間だけで十分で、リクエストごとのローテーションはしない（後述のとおり複数タブ・再送を壊す）。
- Cookie属性はセッションCookieに揃える: `HttpOnly` / `SameSite=Lax` / `Path=/`、`Secure` なし（HTTP評価環境）、有効期限なし。

## 決定（2026-07-19 ユーザー判断）

1. **全ての状態変更リクエスト（POST）に、二重送信トークン検証と同一オリジン検証の両方を課す。** 対象は `POST /register` / `POST /login` / `POST /logout` を含む例外なしの全POST。GETには課さない（GETは状態を変更しないため。破壊的操作をGETで実装しないことは D05 の解釈どおり前提とする）。
2. **トークンはセッションから独立させる。** `sessions` テーブルにCSRF用の列を足さない。decision 0019 のスキーマは変更しない。
3. **秘匿キー・環境変数・JavaScriptのいずれも導入しない。**
4. **トークンは一度きり（one-time）にしない。** Cookieの生存期間中は同じ値を使い回す。理由: ワンタイム化すると、複数タブでフォームを開く・ブラウザバックしてから再送する・POST-Redirect-GET後に戻る、といった通常操作で誤検知が出る。`agent-browser` は AC03-2 の検証で実際に戻るボタンを押す（decision 0008）ため、誤検知は評価の失敗に直結する。**評価の安定性を、限界的な安全性より優先する。**
5. **検証失敗時は HTTP 403 を返し、専用のエラー画面を描画する。** 404（C-10）にはしない——C-10は「存在しない／削除済みリソース」に限った規定であり、CSRF失敗はリソースの存否と無関係。リダイレクトで握り潰さない（失敗が観測できなくなる）。画面には固定文言（C-01）を新規に足さず、既存の `error.html` 系のレイアウトに沿った説明文とトップへの導線を置く。サーバ側に `tracing::warn!` を1行残す。
6. **DB書き込みは検証通過後にのみ開始する。** CSRF判定はトランザクション開始より前に完了させ、拒否経路ではトランザクションを開かない（decision 0002 と整合。部分書き込みが原理的に起こらない）。

### 実装時に守るルール

- **配置**: 検証ロジックの純粋部分は `domain/` に置く。`tokens_match(cookie_token, form_token) -> bool`（定数時間比較、空文字列は常に不一致）と `is_same_origin(origin_header, referer_header, host_header) -> bool` の2つは入力→出力のみの純粋関数であり、ユニットテストで境界（ヘッダ欠落・ポート違い・スキーム違い・大文字小文字）を厚く網羅する。副作用（Cookie読み書き・トークン生成・応答生成）は `web/csrf.rs` に隔離する（CLAUDE.md の Action / Calculation 分離）。
- **同一オリジン検証はミドルウェア**（`web/middleware.rs` の `require_auth` とは別の層）として**ルータ全体**に適用する。`Origin` を優先し、無ければ `Referer` のオリジン部分で代替、両方無ければ拒否。期待値は固定値で持たず、リクエストの `Host` ヘッダと突き合わせる（環境非依存。H-13 で任意のホスト名・ポートで起動されても通る）。
- **トークン検証はエクストラクタ**として実装する。ミドルウェアでボディを読むと `Form` の再構築が必要になり煩雑なため、`Form<T>` を包む `CsrfForm<T>` を `web/csrf.rs` に定義し、各POSTハンドラの引数の型として使う。**型で強制されるので、使い忘れがレビューで目視可能になる**。フォームの構造体（`web/params.rs`）には `csrf_token: String` フィールドを持たせる。
- **トークン生成**: CSPRNG由来の値を使う。`db/sessions.rs` のセッションID生成に倣い `uuid::Uuid::new_v4()`（122ビット）とし、新規クレートを増やさない。
- **テンプレート**: 全フォームに `<input type="hidden" name="csrf_token" value="{{ csrf_token }}">` を置く。Askamaの既定エスケープに任せ、`|safe` 相当は使わない（CLAUDE.md セキュリティ必須要件）。トークンはUUIDの英数字とハイフンのみなので、エスケープの有無で値が壊れることはない。
- **`agent-browser` 操作性（H-02 / ui-ux-guidelines §6）との関係**: hidden input は §6 が禁じる「隠し要素にアクションを強いる」設計に**当たらない**。§6 が問題にしているのは、ホバーでのみ現れるメニューのように**AIが操作を要求される**要素であり、hidden input はAIが触る対象ではなく、ブラウザがフォーム送信時に自動で同梱する。AIが操作するボタン・リンクは従来どおり常時可視のまま変わらない。
- **`Cache-Control: no-store`（C-11）との関係**: 認証必須画面は既に `no-store` なのでトークンを含むHTMLがキャッシュされる懸念はない。未認証の `/login`・`/register` にもトークンを含むHTMLを返すため、これらにも `no-store` を付ける。

## 影響

- **スキーマ**: 変更なし。マイグレーション追加も `cargo sqlx prepare` の再実行も不要（新規SQLが無いため）。decision 0019 に波及しない。
- **新規ファイル**: `app/src/web/csrf.rs`。`app/src/domain/` に純粋検証関数（`validation.rs` へ同居させるか専用モジュールにするかは実装時の判断）。
- **既存ファイルの変更**: `web/mod.rs`（ルータへのミドルウェア追加）、`web/error.rs`（`AppError` に CSRF 失敗のバリアントを足し、403へ写像）、`web/params.rs`（各フォーム構造体に `csrf_token`）、`web/cookies.rs`（CSRF Cookieのビルダを追加。属性の集約先を1箇所に保つ）。
- **全フォームテンプレート**が hidden input を必要とする。フォームを持つ画面を今後追加するたびに漏れなく入れること。入れ忘れは403で即座に現れるため、開発時に検知できる（サイレントな失敗にはならない）。
- **`.env.example` は変更しない。** 秘匿キーが無い想定（foundation-plan §2）を維持する。H-07 の「ユーザーに伝えるべき秘匿キー」は本決定によっても発生しない。
- **形式モデル（`formal/`）**: カバー対象外。CSRFはHTTPレイヤの要件であり、decision 0008 が AC03-2 をモデル外としたのと同じ扱いとする。モデルの証明が通ってもCSRF防御は保証されない。
- **decision 0016（技術スタック）**: 新規クレートを増やさないため影響なし。
- **決定 0021 の参照**: 本決定に基づく実装コードには `// decision 0021` のコメントを残すこと。特に「トークンをワンタイムにしない」（決定4）と「未ログインPOSTも保護対象」（決定1）はユーザーが明示的に確認のうえ承認した論点なので、`web/csrf.rs` の先頭に明記する。
