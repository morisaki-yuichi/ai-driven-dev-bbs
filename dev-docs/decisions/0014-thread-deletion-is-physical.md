# 0014 スレッド削除は物理削除とする

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: D02
- 関連原典: AC06-4、AC10-5、C-06、C-08、C-10、`issues/08`（コメントは論理削除と**明記**）
- 影響範囲: `Bbs.Op.deleteThread`、スキーマ、404 の実装

## 背景

- **[事実]** `issues/08` は**コメント**について「DB上から完全に消し去らない」と明記する。
- **[事実]** `issues/06`（スレッド削除）には**論理削除の指定がない**。「削除は不可逆」とのみ書かれる。
- **[空白]** スレッドを物理削除してよいか、コメントに合わせて論理削除にすべきか。

コメントを論理削除にする理由は原典に書かれている（「スレッドの整合性と文脈を維持する」）。
スレッドにはその理由が当てはまらない。**しかも C-06 により、削除できるのはコメント0件のスレッドだけ**
なので、物理削除しても孤児コメントは発生しない。

## 提案

**物理削除**（`db.threads` から行を除く）。

- AC10-5 は「削除されたスレッド、**または**存在しないスレッド」を**同じ 404** に落とすことを要求する。
  つまり原典は両者の区別を求めていない。物理削除で区別が失われても仕様上問題ない
  （`Bbs.Op.findThread` のコメントに記載）。
- コメント0件限定なので、参照整合性（`Invariant.Wf.commentThreadsExist`）は保たれる。

## 決定（2026-07-19 ユーザー判断）

**物理削除。** `DELETE FROM threads WHERE id = ?` で行を除く。`deleted_at` 列は置かない。

## 影響

### 除外条件の書き漏らしが起きない

論理削除だと、一覧表示・検索・詳細表示・ページネーションの件数のすべてで
`deleted_at IS NULL` を漏れなく書く必要があり、**1ヶ所でも漏れると削除済みスレッドが表示される**。
物理削除ならその種の事故が原理的に起きない。

### 参照整合性は保たれる

C-06 により削除できるのは**コメント0件のスレッドだけ**なので、物理削除しても孤児コメントは発生しない
（`Invariant.Wf.commentThreadsExist`）。外部キーに `ON DELETE CASCADE` を付ける必要すらない
（むしろ**付けない**方が、コメント付きスレッドを誤って削除するコードをDBが弾いてくれる）。

### 「削除済み」と「存在しない」の区別が失われるが問題ない

AC10-5 は「削除されたスレッド、**または**存在しないスレッド」を**同じ 404** に落とすことを要求しており、
原典は両者の区別を求めていない。`Bbs/Op.lean` の `findThread` にこの旨をコメントで残してある。

### コメントとの非対称は意図的

| 対象 | 方式 | 根拠 |
| :--- | :--- | :--- |
| コメント | **論理削除** | `issues/08` が「DB上から完全に消し去らない」と明記。理由も明示（スレッドの整合性と文脈を維持） |
| スレッド | **物理削除** | `issues/06` に論理削除の指定がない。コメントの理由がスレッドには当てはまらない |

監査・復旧の観点では論理削除が好まれるが、**原典は復元機能を明確に否定している**（C-08）ため
行を残す動機が薄い。

### 形式モデル

`Op.deleteThread` は既に物理削除（`db.threads` から除去）。変更不要。
