# 0017 削除済みコメント固定文言（C-01）の集約先

- 関連論点: 新規（フェーズ2独立レビュー `dev-docs/reviews/phase2-review.md` の上流エスカレーションにより判明。原典の論点表（D00〜D20）に対応するものは無い）
- 関連原典: C-01（`requirements-analysis.md`）、AC08-2、AC10-3
- 影響範囲: `dev-docs/foundation-plan.md` §5（実装前に作る基盤の一覧）、実装フェーズの `domain/` 配下モジュール構成

## 背景（原典が何を言い、何を言っていないか）

**[事実]** `requirements-analysis.md` C-01 は4つの固定文言の厳密一致を要求する。うち3つ（アプリタイトル・フッター・ユーザー表示）はヘッダー／フッターに属し、`foundation-plan.md` §5 #9 で `templates/layout.html` への集約が既に定義済みだった。残る1つ、削除済みコメントの本文 `＜このコメントは削除されました＞`（AC08-2 / AC10-3）は、ヘッダー／フッターに属さないコメント表示個別の文言であり、`layout.html` には置けない。

**[空白]** `foundation-plan.md` §5 の他13項目（当時）のどこにも、この文言の集約先が割り当てられていなかった。フェーズ2独立レビューが「重大3件・上流エスカレーション1件」のうち上流エスカレーションとして指摘した（レビューは編集を行わず、ユーザー判断に委ねる形で提示）。

[decision 0016](0016-tech-stack.md) §4 は、この文言をE4（テンプレート不整合）の代表例として名指ししており、Askamaのコンパイル時型検査では文字列リテラルの正しさまでは検査されない。集約先が無いまま実装すると、この文言は最も自然には `thread_detail.html` 等への直書きとなり、書き写しのたびに全角山括弧の取り違えリスクが生じる。

## 選択肢

1. **`domain/` に定数として集約**（例: `domain/model.rs` の `DELETED_COMMENT_TEXT`）。テンプレート側はこの定数（またはこれを返すメソッド）を参照するのみにする。`foundation-plan.md` §5 に新規項目を追加する。
2. **既存の #9 の説明を拡張**し、「`layout.html`」という項目名のまま「3文言は`layout.html`、1文言は`domain/`の定数」という内訳を書く。実装構造は選択肢1と同一で、`foundation-plan.md` 上の見せ方のみが異なる。
3. **構造変更をせず、テスト保護のみ**を追加する。`domain/` に定数は作らず、コメント削除機能の実装時に「削除後の表示が固定文言と完全一致すること」を検証する統合テストを書く運用ルールを `CLAUDE.md` の Definition of Done に追記する。コンパイル時の保護は得られない。
4. **対応しない**。現状の `foundation-plan.md` のまま実装フェーズに進み、気づいた時点で対応する。

## 提案（および理由）

選択肢1を推奨。理由:

- `domain/` は本プロジェクトの実装方針（`CLAUDE.md`「実装方針: Action / Calculation / Data の分離」）でDB・フレームワーク非依存の純粋層と位置づけられており、Askamaのレンダリングを介さない単体テストからも文字列一致を検証できる。
- `foundation-plan.md` §5 の既存の粒度（1項目=1基盤ファイル／モジュール）と一貫しており、新規項目の追加は同ドキュメントの構造を壊さない。
- 選択肢2は最終的な実装構造が選択肢1と同一であり、`foundation-plan.md` 上の表現の違いに留まる。選択肢3はコンパイル時の保護を放棄し、E4という支配的エラー面への対処として選択肢1より弱い。選択肢4はレビューが指摘した欠落を未解決のまま実装フェーズへ持ち越すことになる。

## 決定（2026-07-19 ユーザー判断）

**選択肢1（`domain/` に定数として集約、`foundation-plan.md` §5に新規項目を追加）を採用する。**

`foundation-plan.md` §5 に新規 #10（`domain/` の定数、例 `model.rs` の `DELETED_COMMENT_TEXT`）を追加し、以降の項目番号を1つずつ繰り下げた（旧10〜14 → 新11〜15）。「着手順の意図」の範囲表記（6〜10→6〜11、12〜13）も合わせて更新した。

## 影響

- `dev-docs/foundation-plan.md` §5: 新規項目 #10 を追加、旧#10〜#14を#11〜#15に繰り下げ、「着手順の意図」の範囲表記を更新（反映済み）。
- 実装フェーズ: `domain/` 配下（`foundation-plan.md` §5 #12「`domain/` の骨格」と同じモジュール群）に `DELETED_COMMENT_TEXT` 相当の定数を置き、コメント表示テンプレートはこれを参照する実装にする。テンプレートへの直書きはしない。
- 他のdecisionへの波及なし。[decision 0016](0016-tech-stack.md) の採用スタック・トレードオフの結論は変更していない。
