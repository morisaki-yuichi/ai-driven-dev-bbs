# 0022 登録フォーム失敗時、パスワード欄だけは値を再表示しない

- 関連論点: なし（新規。F01実装中に判明）。
- 関連原典: なし（原典はフォーム再表示時の挙動を規定していない）。
- 影響範囲: `app/src/web/register.rs`（`render_form`）、`app/templates/register.html`。P02(ユーザー登録画面)にのみ適用。他フォーム(F04〜F07)は対象外。

## 背景（原典が何を言い、何を言っていないか）

**[事実]** `dev-docs/ui-ux-guidelines.md` §2 は「失敗時に入力済みの値を消さない」ことを一般原則として定めている（シナリオ05-1-5→6の「失敗後に再度正しい値で保存」という操作をやり直しやすくするため）。フィールドを個別に除外する例外は明記していない。

**[空白]** この原則がパスワードのような機微情報を含むフィールドにも無条件に適用されるかは、`ui-ux-guidelines.md`・原典のいずれも述べていない。

## 選択肢

### (a) ガイドライン通り、パスワード欄も直前の入力値を再表示する

- 利点: ガイドラインの文言に忠実。
- 欠点: パスワードそのものをHTMLレスポンス(サーバから見えるログ・ブラウザ履歴・画面共有等)に平文で載せることになり、パスワード入力欄を「送信のたびに空にする」というブラウザ・パスワードマネージャの一般的な慣行にも反する。バリデーション失敗時特有の追加リスクであり、割に合わない。

### (b) パスワード欄だけは再表示せず空にする

- 利点: 機微情報を不要にHTMLへ載せない。ユニークID・表示名は非機微情報なので原則通り保持し、シナリオ05-1のような「他フィールドは直前の値のまま、パスワードだけ再入力する」操作は妨げない(パスワードは元々どのシナリオでも都度入力し直す想定になっている)。
- 欠点: 原則の字面からは外れる。

## 提案（および理由）

**(b) を採る。** パスワードは他の2フィールド(ユニークID・表示名)と性質が異なり、「消さない」ことの利点(やり直しやすさ)より「機微情報をレスポンスに載せない」ことの安全側の利点が上回る。ユニークID・表示名は`ui-ux-guidelines.md`の原則通り保持し、ガイドラインの趣旨(やり直しやすさ)は損なわない。

## 決定（2026-07-19 ユーザー判断）

登録フォーム(P02)の失敗時再表示で、`unique_id`・`display_name`は直前の入力値を保持し、`password`欄は常に空で再表示する。

## 影響

- `app/src/web/register.rs`の`render_form`はパスワード値を受け取らず、テンプレートにも渡さない。
- `app/templates/register.html`の`<input type="password">`は`value`属性を持たない。
- 他のフォーム(F04プロフィール編集・F05スレッド作成・F07コメント作成)にパスワード欄は無いため、この決定は波及しない。
