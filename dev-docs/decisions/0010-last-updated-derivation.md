# 0010 最終更新日時の導出方法とコメント削除の影響

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: 新規（形式化により判明）
- 関連原典: C-15、AC09-4（コメント投稿のたびに更新）、AC12-4
- 影響範囲: `Bbs.Query.lastUpdatedAt`、スレッドテーブルのスキーマ

## 背景

- **[事実]** C-15「最終更新日時 ＝ スレッド作成時刻、または最新コメント投稿時刻」。
  AC09-4「コメントが投稿されるたびに更新されている」。
- **[空白1]** スレッドに `last_updated_at` 列を持たせるか、コメントから**都度導出**するか。
- **[空白2]** **コメントを論理削除したとき、最終更新日時は動くのか。** 原典は「投稿されるたび」としか
  言わない。削除は投稿ではないので動かないと読むのが素直だが、削除も「更新」ではある。
- **[空白3]** 削除済みコメントの投稿時刻は、最終更新日時の計算に**含まれるか**。
  含めないと、唯一のコメントを削除したスレッドの最終更新日時が**過去に巻き戻る**。

## 提案

1. **導出する**（`max(thread.created_at, max(comments.created_at))`）。冗長な状態を持たず、
   更新漏れによる不整合が構造的に起きない。モデルは `Query.lastUpdatedAt` としてこれを実装。
2. **削除済みコメントの投稿時刻も計算に含める**（＝投稿された事実は消えない）。
   これにより「最終更新日時が過去に戻る」ことがなくなり、単調性が保たれる。
3. **コメント削除では最終更新日時を動かさない**（`deleteComment` は論理時刻を進めない）。

## 決定（2026-07-19 ユーザー判断）

1. **コメントから都度導出する。** `max(thread.created_at, max(comments.created_at))`。
   `threads` に `last_updated_at` 列は置かない。
2. **削除済みコメントの投稿時刻も計算に含める。**
3. **コメント削除では最終更新日時を動かさない。**

```sql
SELECT t.*,
       GREATEST(t.created_at, COALESCE(MAX(c.created_at), t.created_at)) AS last_updated_at
FROM threads t
LEFT JOIN comments c ON c.thread_id = t.id   -- deleted の絞り込みを掛けない
GROUP BY t.id
```

## 影響

### 単調性が保たれる

2 と 3 の組み合わせにより、**最終更新日時は決して過去に戻らない**。

```
スレッド作成 t=1
コメント1投稿 t=2 → 最終更新 = 2
コメント1を削除    → 最終更新 = 2（変わらない）
```

削除済みを計算から外していたら、唯一のコメントを削除したスレッドの最終更新日時が
`t=1` に巻き戻り、一覧の並びが削除操作で突然入れ替わっていた。
**シナリオ03 は実際に「コメント投稿 → 削除 → 再投稿」を行う**ので、この振る舞いに入る。

### 更新漏れが原理的に起きない

導出方式なので、コメント投稿時に別テーブルを更新し忘れるバグが存在しえない。
`last_updated_at` 列を置く方式なら、decision 0002 のトランザクション内で
INSERT と UPDATE の両方を行い、かつ**列が常に導出値と一致することを不変条件として検証**する必要があった。

性能面では一覧表示のたびに集約が要るが、評価規模（十数件）では問題にならない。

### 削除済みコメントの扱いが全体で統一された

| 用途 | 削除済みを数えるか | 根拠 |
