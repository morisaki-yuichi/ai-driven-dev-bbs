# 0007 多重ログインと既存セッションの扱い

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: D04
- 関連原典: AC02-4（ログイン状態が維持される）、`issues/03` 冒頭（ログアウトで現在のユーザーセッションを破棄）
- 影響範囲: `Bbs.Op.login` / `Bbs.Op.logout`、セッションストア

## 背景

- **[事実]** 原典が述べるのは「ログイン状態が維持される」「ログアウトで**現在の**セッションを破棄」まで。
- **[空白]** 同一ユーザーが**複数のセッションを同時に持てるか**、新規ログイン時に既存セッションを
  無効化するかは未規定。
- **[空白]** セッションの**有効期限**も未規定。

`issues/03` の「**現在の**ユーザーセッション」という語は、複数セッションが併存しうることを
かすかに示唆するが、断定はできない。

## 提案

- **多重セッションを許す**（ログインのたびに新しいセッションを発行し、既存のものは残す）。
  モデルの `Db.sessions : List Session` はこの形。`logout` は指定したセッションのみを消す。
- **有効期限は設けない**（原典に規定がなく、評価は短時間で完結するため）。

理由: 「現在のセッションを破棄」という原文に最も忠実。既存セッションを無効化する実装だと、
ログアウトが「全端末からのログアウト」になり、原文の「現在の」と食い違う。

## 決定（2026-07-19 ユーザー判断）

1. **多重セッションを許す。** ログインのたびに新しいセッションを発行し、既存のものは残す。
   `logout` は指定されたセッションのみ破棄する。`issues/03` の「**現在の**ユーザーセッションを破棄」に忠実。
2. **セッションは DB に永続化する。** アプリ／コンテナを再起動してもログイン状態が続く。
3. **有効期限は設けない**（原典に規定がなく、評価は短時間で完結するため）。

## 影響

- 形式モデルは既にこの形（`Db.sessions : List Session`、`Op.logout` が当該 sid のみ除去）。変更不要。
- **`sessions` テーブル（または同等の永続ストア）が必要**になる。署名付き Cookie に状態を持たせる方式でも
  「再起動で失われない」は満たせるが、その場合ログアウト時にサーバ側で失効させる手段
  （失効リストなど）が要る点に注意。**サーバ側セッションテーブルの方が素直。**
- **Redis は使わない。** H-04 を満たすには Docker コンテナを1つ増やすことになり、
  H-06（README 起点でセットアップ完結）の手順がその分複雑になる。この規模では過剰。
- 再起動でログアウトされないため、評価者がシナリオ途中でコンテナを再起動しても手順が壊れない。
  H-08〜H-10 は DB を消さない前提なので、**再起動自体は起こりうる**シナリオとして扱う。
- decision 0002（トランザクション）と合わせ、**永続化に RDBMS を選ぶ動機がさらに強まった**
  （ユーザー・スレッド・コメント・セッションのすべてが1つの DB に収まる）。
