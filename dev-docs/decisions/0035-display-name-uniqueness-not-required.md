# 0035 表示名の一意性は要求しない(同名改名によるなりすまし表示を意図的に許容する)

- 関連論点: D16(未定義の入力制限)
- 関連原典: C-03(表示名)、issue 01(`docs/product/issues/01_user_registration.md`)、issue 04(`docs/product/issues/04_user_profile_edit.md`)
- 影響範囲: `app/src/db/users.rs`(`insert`・`update_display_name`)、`users`テーブルスキーマ、F01登録・F04プロフィール編集

## 背景(原典が何を言い、何を言っていないか)

- **[事実]** `dev-docs/requirements-analysis.md` の C-03 は「表示名は15文字以内、**重複可**」と明記している(同ファイル51行目・239行目)。DBスキーマも `users.display_name` に `unique` 制約を付けていない(`unique_id` のみ `unique`)。
- **[事実]** issue 01(`docs/product/issues/01_user_registration.md`)は表示名について「サイト内で表示される名前」「15文字を超える場合、登録できない」とだけ述べ、一意性には一言も触れていない。
- **[事実]** issue 04(`docs/product/issues/04_user_profile_edit.md`)も同様に、編集可能項目は表示名のみ・15文字以内という制約のみを述べ、一意性・重複禁止には触れていない。
- **[事実]** 登録側では既に `app/src/db/users.rs` の `insert_allows_duplicate_display_names` テストが「C-03: 表示名は重複可能」として同名registrationの成功を明示的に許容している。この挙動はF01登録の実装当初からのものであり、今回のF04プロフィール編集レビューで新たに発見した論点ではない。
- **[空白]** 原典は「他人と同じ表示名に**改名**した場合」については明示的に触れていない。ただしC-03の「重複可」がinsert/updateを区別する理由はなく、登録時に重複を許すなら改名でも同じ規則を適用するのが自然な読み方である。

## 選択肢

1. **一意制約を追加しない(現状維持)**。登録・改名のいずれでも他人と同じ表示名を許す。
2. `display_name` に一意制約を追加し、改名時のみ(あるいは登録・改名の両方で)重複を拒否する。

選択肢2は次の副作用を伴う:
- C-03「重複可」という明文の記述と矛盾する。
- 既存の `insert_allows_duplicate_display_names` テスト(登録時の同名成功を明示的に許容)を壊す、または登録と改名で規則を分岐させる必要が生じ、一貫性がなくなる。
- 評価シナリオ側が同名での登録・改名を試みた場合に失敗するUIを新たに作ることになり、原典に無い制約をAI側が追加することになる。

## 提案(および理由)

**選択肢1(一意制約を追加しない)を採用する。** 原典が明文で「重複可」と述べ、登録側で既に同名を許容する実装・テストが確定している以上、改名だけ別ルールにする根拠がない。結果として、悪意のあるユーザーが他人と同じ表示名に改名し、あたかもその人物であるかのように見せる「なりすまし表示」が技術的に可能になるが、これは原典の制約を忠実に実装した結果であり、見落としではなく意図的な許容として記録する。

## 決定(2026-07-20 ユーザー判断)

**一意制約を追加しない。** 表示名の重複(登録時・改名時のいずれも)を許可する現状の実装を維持する。

他人と同じ表示名への改名によるなりすまし表示が可能であることを、リスクとして認識した上で許容する。**ユーザーの同一性を識別する手段はユニークID(`users.unique_id`、一意制約あり)であり、表示名は同一性の識別子ではない。** 掲示板の投稿者欄・ヘッダーに表示されるのは表示名のみだが、ユニークIDでの照合が必要な場面(将来そうした機能が入る場合)では別途ユニークIDを参照すればよい。

## 影響

- `app/src/db/users.rs` の `insert`・`update_display_name` に一意制約起因のエラー処理を追加しない。`is_unique_violation` は引き続き `unique_id` の重複検出専用として使う。
- `users` テーブルのスキーマ変更は不要。
- 将来これが問題になりうる条件: **表示名だけを頼りに人物を識別・信頼する機能**(例: 表示名でのDM宛先解決、表示名ベースの権限付与、表示名でのブロック/ミュート)が追加された場合、なりすましが実害を持つようになる。その時点でこの決定を再検討し、一意制約の追加または「表示名(ID)」のような複合表示への変更を検討する。
