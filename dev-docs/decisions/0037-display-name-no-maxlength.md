# 0037 表示名入力欄から maxlength を外す（0023の観測性方針を novalidate 非対象の属性へ拡張）

- 関連論点: なし（新規。最終ゲート・評価者経路の通し確認で判明）。関連決定: 0023（novalidate 採用）、0003（文字数の単位＝コードポイント）。
- 関連原典: H-02, H-12（`dev-docs/harness-invariants.md`）。AC01-4（表示名15文字超で登録不可）、シナリオ05-1-5（表示名16文字でエラーになることを確認）。
- 影響範囲: `app/templates/register.html`、`app/templates/profile_edit.html` の表示名 `<input>`。`app/tests/register_test.rs`、`app/tests/profile_edit_test.rs`。

## 背景（原典が何を言い、何を言っていないか）

**[事実]** シナリオ05-1-5・AC01-4 は「表示名を16文字にするとエラーになる」ことを要求している。評価は `agent-browser` による自然言語での画面操作・観測で行われる（H-12）。

**[事実]** decision 0023 は、ネイティブ検証バブルが `agent-browser` から読めない問題に対し、フォームに `novalidate` を付けてサーバ側エラーを画面テキストとして見せる方針を決定した。同決定は「`required` / `minlength` / `maxlength` 属性そのものは残す」とし、`maxlength` はセマンティクス（スクリーンリーダーへの伝達・入力欄の最大長ヒント）として維持する前提だった。

**[事実]** `maxlength` は `novalidate` によって無効化されない別のブラウザ機構である。`novalidate` はフォーム送信時のブロック（`checkValidity()` 相当）を止めるが、`maxlength` は**入力イベントの時点**で文字の追加そのものを止める。

**[空白（最終ゲートで判明した実害）]** 全12機能の実装完了後、空DBからの評価者経路の通し確認（シナリオ01〜05）で、表示名欄に `maxlength="15"` が付いたままだと `agent-browser` が16文字目を入力できず、「入力抑止」として現れることが分かった。これは AC01-4・シナリオ05-1-5 が期待する「エラー表示」の観測に失敗する（サーバ側検証は健在で、maxlength を回避した直接POSTでは「15文字以内で入力してください」＋DB未更新になることは確認済み）。0023 が novalidate で解決した問題（ネイティブ検証の結果が観測できない）と同じ構造の問題が、0023 が対象にしなかった `maxlength` という別属性で再発した。

## 選択肢

### (a) maxlength を維持する（0023の記述どおり）

- 利点: 0023の決定文をそのまま踏襲できる。入力欄の最大長ヒントというセマンティクスを保てる。
- 欠点（決定的）: `agent-browser` が16文字目を入力できず、AC01-4・シナリオ05-1-5のエラー表示を字義どおり観測できない。0023が解決したはずの「観測性の欠如」が形を変えて残る。

### (b) maxlength を外す

- 利点: `agent-browser` が16文字を入力でき、サーバ側エラーが画面テキストとして表示される過程を観測できる。判定基準は0023と同じくサーバ側（`domain/validation.rs`、コードポイント単位）の1箇所のみになり、0023が指摘していた「maxlengthはUTF-16コードユニット単位・C-03はコードポイント単位」という単位の乖離（0023 §影響で触れていたBMP限定運用の前提）も、maxlength自体が無くなることで解消する。
- 欠点: 入力欄の最大長ヒントというセマンティクス、および文字数超過時の早期抑止という補助的なUXを失う。ただし0023がnovalidateで既に「即時フィードバックの喪失」を受け入れているので、追加で失うものの性質は同じ。

## 提案（および理由）

**(b) を採る。**

0023の決め手は「エラーが `agent-browser` から読めるかどうか」であり、この基準は `novalidate` が及ぶ属性だけでなく、入力そのものを止める `maxlength` にも同じ強さで当てはまる。`maxlength` を残したままでは、0023が達成したはずの観測可能性がこの1属性によって再び失われる。0023 §影響が「maxlengthは据え置く」とした判断は、その時点では「送信ブロックの有無」という観点（novalidateの影響範囲）でのみ評価されており、「入力そのものの抑止」という別の観点を明示的に検討していなかった。今回そちらの観点で実害（シナリオ観測失敗）が判明したため、0023の対象外だった属性について新たに決定する。

0023を直接改訂せず新規番号にしたのは、0023の決定（novalidateの採否）自体は覆っていないため。0023 §影響の「maxlengthは据え置く」という記述は、本決定により表示名欄については上書きされる（ユニークID・パスワード欄は対象外、今回の論点は表示名の文字数制限のみ）。

## 決定（2026-07-21 ユーザー判断）

**表示名の入力欄（`register.html`・`profile_edit.html`）から `maxlength="15"` を外す。** ユニークID・パスワード欄には手を触れない。クライアント側での文字数の即時抑止は無くなり、サーバ側検証（`domain/validation.rs`、コードポイント単位、decision 0003）が唯一の砦になる。これは0023がnovalidateで採った「クライアント側検証を減らし、サーバ側エラーを画面テキストで見せる」方針の延長である。

## 影響

- `app/templates/register.html`・`app/templates/profile_edit.html`: 表示名 `<input>` から `maxlength="15"` を削除。register.html に付けていたWhy-not コメント（maxlengthのUTF-16コードユニット単位とC-03のコードポイント単位の乖離をBMP限定運用として許容する趣旨）は、maxlength自体が無くなるため撤去し、「なぜmaxlengthを付けないか」のWhyコメントに置き換えた。
- `app/tests/register_test.rs`・`app/tests/profile_edit_test.rs`: サーバ側検証（16コードポイントの表示名が拒否されDBが変化しない）を確認する結合テストは既存（`post_register_display_name_too_long_is_rejected`、`post_profile_edit_16_chars_name_is_rejected_and_keeps_old_value`）。前者にDB未作成の確認（`select count(*) from users`）を追加した。maxlengthはテンプレート属性でありRustのテストからは直接見えないが、除去後もサーバ側の振る舞いが変わっていないことをこれらのテストが担保する。
- decision 0023 §影響の「maxlengthは据え置く」という記述は、表示名欄について本決定で上書きされる。0023自体は改訂しない（決定内容は変わっていない）。
