# 0033 検索語qのトリムと最大長

- 関連論点: D07（decision 0011の続き。レビューで新たに判明）
- 関連原典: AC11-2。原典は検索語の前処理・長さ制限を規定していない（なし）。
- 影響範囲: `app/src/web/params.rs::ListParams::parse`、`app/templates/thread_list.html`

## 背景（原典が何を言い、何を言っていないか）

- **[事実]** decision 0004 により、スレッド本文・コメント本文は前後空白をトリムしてから保存される。
- **[事実]** decision 0011 は検索方式を「素朴な部分文字列一致」「正規化なし」「空クエリは全件表示」と定めたが、**検索語`q`自体の前処理（トリム）や長さの上限には触れていない**。
- **[空白]** 末尾・先頭に空白を含む検索語を入力した場合の挙動。トリムしないと、本文側は保存時にトリム済みのため、末尾空白付きの検索語は原理的に一致しなくなる（例: `"Rust "` で検索しても、保存済みの本文中の `"Rust"` には一致しない）。
- **[空白]** 検索語の長さの上限。現状は`LIKE`の全走査に加え、C-13のページ送りリンク（`/?q=...&sort=...&page=...`）2本（前後）に`q`をパーセントエンコードして埋め込むため、非ASCII文字を含む検索語は最大3倍程度の長さに展開されてレスポンスHTMLへ埋め込まれる。上限が無いと、極端に長い入力でレスポンスサイズが不必要に膨張する。

## 選択肢

### トリムについて

1. トリムせずそのまま検索語として使う。
2. **decision 0004と同じ`trim`（前後空白除去）を検索語にも適用してから検索・再表示する。**

### 最大長について

1. 上限を設けない。
2. 上限を設け、超過分は黙って切り詰める（画面上は何が起きたか分からない）。
3. **上限を設け、超過時はトリム後の先頭N文字に切り詰めて検索を実行し、切り詰めが起きたことを画面上に明示する。**

## 提案（および理由）

**トリムは選択肢2、最大長は選択肢3を採る。**

- **トリム**: 本文・コメント本文が保存時にトリムされている（decision 0004）以上、検索語だけトリムしないのは非対称で、末尾空白付きの検索語が原理的にヒットしないという分かりにくい不具合を生む。`domain::validation::trim`（decision 0004で確立済みの実装、全角スペース・NBSPを含め正しく空白を判定する）をそのまま再利用する。
- **最大長**: 実用上、掲示板の検索語が数百文字を超える必要は無い一方、上限が無いと`LIKE '%...%'`の全走査コストとページ送りリンクへの重複展開（最大3本×パーセントエンコードで最大3倍長）が入力長に比例して膨張する。**200文字**を上限とする。C-03（表示名15文字）・本文長そのものには原典上の上限が無いため参考にできないが、実際の検索キーワード（シナリオ04-1は`Rust`のような数文字〜十数文字）を大きく超える余裕を持たせつつ、悪意・誤操作による極端な長さの入力を弾くのに十分な値として選んだ（決め打ちの根拠は「シナリオが使う長さの一桁以上上」という以上のものではなく、実測に基づく値ではない）。
- **黙って切り詰めない**: `dev-docs/ui-ux-guidelines.md` §1（状態の網羅）とH-12（`agent-browser`による自然言語での観測可能性）により、入力と実際の検索語が食い違う場合はその事実を画面上で観測できる必要がある。切り詰め後の値を検索窓・検索結果見出しに再表示するだけでなく、切り詰めが起きた旨のメッセージを明示する。

## 決定（2026-07-20 ユーザー判断）

**提案どおり採用する。**

1. **検索語`q`は`domain::validation::trim`でトリムしてから検索・再表示に使う。**
2. **検索語の最大長は200コードポイント（decision 0003と同じ数え方）とする。** トリム後の長さが200を超える場合、先頭200文字に切り詰めて検索を実行する。
3. **切り詰めが起きたことを画面上に明示する。** `ListParams`が`q_truncated: bool`を持ち、`web/thread_list.rs`がこれをテンプレートへ渡し、`thread_list.html`の共通メッセージエリアに警告文言（「検索キーワードが200文字を超えたため、先頭200文字で検索しました。」）を表示する。検索窓・「『{{ q }}』の検索結果」の見出し・ページ送りリンクは、いずれも切り詰め後の値を使う（実際に検索された値と表示が一致する）。

## 影響

- `app/src/web/params.rs::ListParams`: `q`フィールドはトリム・切り詰め後の値を持つように変更し、新たに`q_truncated: bool`を追加する。`MAX_QUERY_LEN: usize = 200`を定数として定義する。
- `app/src/web/thread_list.rs`: `ThreadListTemplate`に`q_truncated`を追加し、`ListParams`からそのまま引き渡す。
- `app/templates/thread_list.html`: 共通メッセージエリアに切り詰め通知を追加する。新しい`.message-warning`CSSクラス（`app/static/app.css`）を使う（既存の`.message-success`/`.message-error`のどちらの意味にも当てはまらないため）。
- decision 0011（検索の照合方式・空クエリは全件表示）・decision 0004（トリム方針）は変更しない。本決定はそれらが規定していなかった`q`自体の前処理という別論点への追加であり、既存の決定を覆すものではない。
- 200文字は実測に基づく値ではなく「シナリオが使う長さより十分大きい」という参考値である。将来、正当な理由で200文字を超える検索語が必要になった場合は、この決定を変更履歴付きで更新する。
