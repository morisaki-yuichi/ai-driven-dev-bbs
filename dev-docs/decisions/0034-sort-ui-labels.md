# 0034 F12ソートUIの表示ラベル文言

- 関連論点: なし（`dev-docs/requirements-analysis.md` §3に該当論点なし。issues/12が挙げる3ソート条件の表示文言そのものが未規定）
- 関連原典: `docs/product/issues/12_sort_function.md`（ソート条件3種）、`docs/product/designs/ui_design.md`（P03のソート切替セレクトボックス「作成日時順」「コメント数順」「最終更新日時順」という例示）、シナリオ04-2
- 影響範囲: `app/templates/thread_list.html`の`<select name="sort">`内の`<option>`表示文言、および同フォームの送信ボタンのラベル

## 背景（原典が何を言い、何を言っていないか）

**[事実]** issues/12は3ソート条件を「作成日時順(降順/昇順)」「コメント数順(多い順)」
「最終更新日時順(降順)」と述べる。`ui_design.md`は同じくセレクトボックスの例として
「作成日時順」「コメント数順」「最終更新日時順」を挙げるが、これは3値の例示であり、
`SortKey`が持つ4値(`CreatedAsc`・`CreatedDesc`・`CommentCountDesc`・`LastUpdatedDesc`)
それぞれに対応する**具体的な表示文言(セレクトボックスの`<option>`テキスト)までは
指定していない**。

**[空白]** 4値それぞれの日本語ラベル文言そのもの。decision 0012(D12)は
シナリオ04-2-2が「作成日時(昇順)」を名指しするため昇順が選択肢として必須である
ことを既に確定しているが、その表示文言までは決めていない。

C-01(固定文言4種、`dev-docs/ui-ux-guidelines.md` §8)には該当しない ―― C-01は
削除済みコメント本文・ヘッダー等の4文言のみを対象とし、ソートラベルは含まれない。

## 選択肢

1. issues/12の見出しをほぼそのまま使う: 「作成日時順(降順)」「作成日時順(昇順)」
   「コメント数順」「最終更新日時順」。
2. 「〜順」を省き、括弧で向きを示す簡潔な形にする: 「作成日時（新しい順）」
   「作成日時（古い順）」「コメント数（多い順）」「最終更新日時（新しい順）」。
3. 矢印記号(↓↑)等の記号を使う。

## 提案（および理由）

**選択肢2を採用する。**

- 一覧のスレッドカードは既に「コメント {{ n }} 件・最終更新 {{ t }}」という
  「項目名＋値」の簡潔な表記を使っており(`thread_list.html`)、ソートラベルも
  同じトーンで揃える方が一貫する。
- 「降順」「昇順」は数学・DB用語寄りで、一般利用者向けの画面としては
  「新しい順」「古い順」「多い順」の方が`ui-ux-guidelines.md`が想定する平易さに合う。
- 選択肢3(記号)は`agent-browser`がテキストとして解釈する際に曖昧さを生みうる
  (ui-ux-guidelines §6: 状態変化・情報を色や記号だけに依存させない、という
  原則の類推)ため避ける。

## 追加論点: 送信ボタンのラベル（F12レビューで追加）

**[空白]** ソート用`<select>`と検索窓が共有する送信ボタンの文言。原典はこのボタン自体に言及しない。

**問題**: プロジェクトはJSを使わない方針（decision 0008: SSR/MPA、`onchange`での自動送信をしない）なので、`<select>`の変更は**送信ボタンを押すまで一覧に反映されない**。ボタンのラベルが「検索」のままだと、「並び替えるにはこのボタンを押す」ことが画面上のどこにも書かれていない状態になる。`agent-browser`（H-12）が「コメント数順に並び替える」と指示されたとき、optionを選んだだけで一覧が変わらず、**検索語を変えていないのに「検索」を押す**という発想に至らずに手詰まりになりうる（H-02の操作性・H-12の観測性）。

**選択肢**: (a) ボタンを「検索」のままにし、`<select>`の近くに説明文を添える。(b) ボタンのラベル自体に並び替えを含める。(c) 検索用と並び替え用にボタンを2つ置く。

**採る案: (b)**。(a)は説明文が増えるだけで、押すべき対象がボタンであることの結び付きが弱い。(c)は`<select>`と検索窓が同一`<form>`にある（`q`を失わないための構成、本ファイル上部および`thread_list.html`のコメント参照）以上どちらのボタンを押しても挙動が同じで、2つ置く意味が無いうえ画面を無駄に複雑にする。ラベル自体が操作の帰結を語る(b)が、`agent-browser`にとって最も観測可能。

## 決定（2026-07-20 ユーザー判断）

`<select name="sort">`の4つの`<option>`文言を次のとおりとする。

| `SortKey` | `value` | 表示文言 |
| :--- | :--- | :--- |
| `CreatedDesc`(既定、decision 0009) | `created_desc` | 作成日時（新しい順） |
| `CreatedAsc` | `created_asc` | 作成日時（古い順） |
| `CommentCountDesc` | `comment_count_desc` | コメント数（多い順） |
| `LastUpdatedDesc` | `last_updated_desc` | 最終更新日時（新しい順） |

あわせて、このフォームの送信ボタンのラベルを **`検索・並び替え`** とする（従来は`検索`）。ユーザーが承認した。

## 影響

- `app/templates/thread_list.html`の`<select id="sort_select" name="sort">`が
  この表のとおりに実装されている(F12実装セッション)。
- 同フォームの`<button type="submit">`のラベルが`検索・並び替え`になっている
  (F12レビュー対応セッション)。
- C-01(固定文言表)には追加しない ―― C-01は原典が厳密一致を要求する4文言のみを
  対象とし、本決定のラベル(`<option>`・ボタンとも)は対象外。`ui-ux-guidelines.md` §8の
  表は書き換えない。
- コード側(`app/templates/thread_list.html`の`<select>`直上および送信ボタン直上の
  コメント)に決定番号を残している。
