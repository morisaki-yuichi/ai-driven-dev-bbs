# 0028 エラーページへの認証状態の反映方式（マーカー＋ミドルウェア後処理）

- 関連論点: 新規（F10 スレッド詳細表示のレビュー指摘により判明）
- 関連原典: C-09（未ログインは一律ログイン画面へ）、C-10（存在しない／削除済みは一律404）、decision 0021（CSRF二重送信トークン）
- 影響範囲: `app/src/web/error.rs`、`app/src/web/middleware.rs`、`app/src/web/mod.rs`、`app/templates/error.html`（`layout.html` のヘッダー分岐）

## 背景（原典が何を言い、何を言っていないか）

**[事実]** 原典は「存在しないスレッドは404」（C-10 / AC10-5）と「未ログインは一律ログイン画面へ」（C-09）を定めている。
`layout.html` のヘッダーは、ログイン中なら表示名とログアウトボタン、未ログインならログイン・新規登録リンクを出す。

**[空白]** **エラーページ（404・403）のヘッダーを、そのリクエストの認証状態に合わせるかどうか**を原典は一言も述べていない。
どの画面にもヘッダーは出るので実装上は必ず何かを選ぶことになるが、原典に手掛かりが無い。

さらに実装上、これは単なる見た目の一貫性の問題ではなく**構造の問題**として現れた。404を返す経路は
`?` による `DomainError` → `AppError` の変換であり、`From<DomainError>` にリクエストコンテキストは渡せない。
つまり「どこかで誰かが認証情報を積み直す」必要があり、その「誰か」の置き場所を決める必要がある。

## 選択肢

1. **ハンドラが `.map_err(|e| e.with_current_user(..))` で後付けする**（当初の実装）。
2. **`AppError` に認証情報を持たせ、専用エクストラクタで自動注入する。**
3. **常に認証情報を解決してリクエスト拡張に載せるミドルウェアを置く。**
4. **レスポンスにマーカーを載せ、ミドルウェアの後処理で本文を描き直す。**

## 提案（および理由）

**選択肢4を採る。** `AppError` が404を描くとき、レスポンス拡張に `AuthAwareErrorPage` マーカーを載せる。
ルータ全体に掛けたミドルウェア `reflect_auth_on_error_page` がマーカーを見つけたときだけセッションを解決し、
認証済みヘッダーで本文を描き直す。ステータス・ヘッダー（`Cache-Control: no-store`、`Set-Cookie`）は元の応答のまま保つ。

- **付け忘れが起きない。** ハンドラ側は `?` で NotFound を返すだけでよい。F06・F07・F08 が
  NotFound や Forbidden を返すようになっても、追加の配線なしで正しいヘッダーが出る。
- **`fallback`（未知URL）も同じ1箇所で面倒を見られる。** `fallback` は未ログインでも404を返す必要があるため
  `require_auth` の外にあり、認証を必須にできない。「セッションがあれば解決し、無ければ未ログイン扱い」という
  緩い解決にすることで、認証必須画面と `fallback` の両方を同じ仕組みで扱える。
- **既定が安全側に倒れる。** `error.rs` は常に未ログイン用ヘッダーで描画し、セッションが解決できたときだけ
  上書きする。後処理が働かない経路でも表示が壊れず、認証情報を漏らす方向には倒れない。

**却下した理由**:

- **選択肢1**: opt-in なので付け忘れをコンパイラが検出できない。実際、`fallback` は `require_auth` の外にあるため
  付けようがなく、**ログイン中に存在しないURLを踏むと未ログイン用ヘッダーが出る**不具合が残っていた。
  加えて成功経路（引数渡し）と失敗経路（`map_err`）で `current_user` を二重に受け渡すことになり、
  片方だけ直す事故の余地があった。
- **選択肢2**: `From<DomainError>` が使えなくなり（`?` が壊れる）、全ハンドラの戻り値型に波及する。得られるものに対して改変が大きい。
- **選択肢3**: 全リクエスト（静的ファイル含む）にセッション参照のDBクエリが1回増える。マーカー方式ならエラーページを返すときだけ引けばよい。

### CSRFエラーページは対象外にする

**CSRF検証失敗（403）にはマーカーを付けない。** よってCSRFエラーページは常に未ログイン用ヘッダーで描画される。

理由: ログイン中ヘッダーはログアウトフォームを含み、そのフォームはCSRFトークンを要する（decision 0021）。
CSRF検証に失敗した文脈で有効なトークンを埋めたフォームを描くのは筋が通らず、空トークンで描けば
押した瞬間にまた403になるボタンを見せることになる。

この不変条件は次の2点で構造的に保たれる。

1. `csrf_error()` がマーカーを付けず、`AppError` には認証情報を後から積む手段が無い（`with_current_user` は削除した）。
2. ルータでCSRFの2ミドルウェアを `reflect_auth_on_error_page` より**外側**に置いてあり、
   `same_origin_guard` が撥ねた応答はそもそも後処理へ来ない（二重の防御）。

## 決定（2026-07-20 ユーザー判断）

**提案どおり選択肢4を採用する。** CSRFエラーページは対象外とし、常に未ログイン用ヘッダーで描画する。ユーザーが承認した。

## 影響

- `app/src/web/error.rs`: `AppError` から `current_user` フィールドと `with_current_user` を削除。
  `AuthAwareErrorPage` マーカーと `render_not_found_body` を追加。
- `app/src/web/middleware.rs`: `reflect_auth_on_error_page` を追加。
- `app/src/web/mod.rs`: 上記ミドルウェアをルータ全体に適用（CSRFの2ミドルウェアより内側）。
- `app/src/web/thread_detail.rs`: `.map_err(...)` を削除し、`current_user` の二重の受け渡しを解消。
- 回帰テスト: `tests/thread_detail_test.rs` に未知URLの404でのヘッダー検証（ログイン中／未ログイン）、
  `tests/thread_create_test.rs` にCSRFエラー画面がログアウトフォームを持たないことの検証を追加。
- **F06・F07・F08 への含意**: これらが NotFound / Forbidden を返すようになっても、
  ヘッダーのために追加で書くコードは無い。
