# 0030 コメント削除の確認方式（確認ダイアログなしで即削除）

- 関連論点: D18（`dev-docs/requirements-analysis.md` §3、削除確認ダイアログの要否）。
- 関連原典: issues/08（コメントの削除機能）AC「コメント作成者が、自分のコメントに対して削除操作を実行できること」。削除確認UIの要否そのものは原典に明記が無い。
- 影響範囲: `app/src/web/thread_detail.rs`（`delete_comment`ハンドラ）、`app/templates/thread_detail.html`（削除フォーム）。

## 背景（原典が何を言い、何を言っていないか）

**[事実]** issues/08 は「削除操作を実行できること」「復元できないこと（不可逆性）」を要求するが、削除実行前に確認ダイアログを挟むかどうかは規定していない。`dev-docs/requirements-analysis.md` の D18 がこの論点を「任意」として明示的に洗い出している。

**[事実]** `dev-docs/ui-ux-guidelines.md` §6 は「ネイティブダイアログ（`window.confirm`等）は避ける。削除確認ダイアログを実装する場合（D18、任意）、`window.confirm`は`agent-browser`からの検知・操作性に影響しうるため、HTML内のモーダル要素など通常のDOM操作で完結する方式を使う」と定めている。加えて decision 0008（SSR/MPA限定、JavaScriptを使わない）がある以上、`window.confirm`のようなJS依存の確認手段はそもそも採れない。HTML+CSSのみのモーダル（`<dialog>`要素やチェックボックスハックなど）を確認手段として実装することは技術的には可能だが、追加のマークアップ・状態管理（開閉状態をURLかフォーム送信の往復で表現する必要がある）を要する。

**[事実]** 削除は論理削除（C-07）であり、`comments`テーブルの行も本文も物理的には消えない。`deleted_at`が立った後の表示は固定文言（C-01）に置き換わるだけで、DB上のデータそのものは消滅しない。

**[空白]** 削除前の確認ステップを設けるかどうかは、原典・既存decisionのいずれも「必須」とは述べていない（D18が「任意」）。

## 選択肢

### (a) 確認なしで即削除する

POSTを受けたら確認ステップを挟まず即座に論理削除を実行し、結果を`?comment_deleted=1`等のフラッシュ通知で伝える（decision 0024と同じ方式）。

- 利点: 実装が単純（追加のマークアップ・状態遷移が不要）。`agent-browser`から見て「削除ボタンを押す→即座に結果が観測できる」という最短の操作系列になり、H-02（操作性）と相性が良い。JSを使わない前提（decision 0008）と自然に整合する。
- 欠点: 誤操作への保護が無い。ただし論理削除であり物理的にデータが消えるわけではない（DB上は残り続ける）。

### (b) HTML+CSSのみの確認モーダルを挟む

削除ボタン押下で確認用のHTML要素（`<dialog>`等）を表示し、そこでの再確認後に実際のPOSTを送る。

- 利点: 誤操作への保護になる。
- 欠点: `<dialog>`のオープン状態はクライアント側のDOM操作（`showModal()`等、多くはJS）を要することが多く、JSなしで実装しようとすると追加のGET/POST往復や複雑なCSSハックが要る。実装量に対して得られる保護は「論理削除で復元不能ではない」という前提のもとでは相対的に小さい。

## 提案（および理由）

**(a) を採る。** 理由は3点。

1. 固定文言が残る論理削除であり、削除は「完全な不可逆」ではない（C-07。物理削除であるスレッド削除（decision 0014）とは性質が異なる）。誤操作の実害は「その箇所の表示が固定文言に変わる」程度に限定される。
2. `window.confirm`はH-02（`agent-browser`操作性）に悪影響がある（ui-ux-guidelines §6で明記）。避けるべき手段が確定している一方、代替のHTML+CSSモーダルは実装コストに見合う保護を提供しない。
3. decision 0008（JSなし・SSR/MPA限定）の一貫性を保つ。確認モーダルを足すと、この機能のためだけに例外的な状態管理パターンを持ち込むことになる。

## 決定（2026-07-20 ユーザー判断）

コメント削除は確認ダイアログを挟まず、POSTを受けたら即座に論理削除を実行する。削除結果（成功・権限なし・削除済み）はdecision 0024と同じクエリパラメータ経由のフラッシュ通知方式で、`/threads/{thread_id}`への再リダイレクト後にui-ux-guidelines §2の共通メッセージエリアへ表示する。ユーザーが承認した。

## 影響

- `app/src/web/thread_detail.rs`: `delete_comment`ハンドラがPOST受理後、確認ステップなしで`db::comments::delete`を呼ぶ。
- `app/templates/thread_detail.html`: 削除ボタンは通常の`<form method="post">`+`<button type="submit">`のみで構成し、`onclick`・`confirm(`等のJS確認処理を含まない。
- `dev-docs/ui-ux-guidelines.md`§6の「ネイティブダイアログを避ける」要件を満たす形で実装する。
