# 0003 文字数の数え方と文字種の定義

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: D15, D16
- 関連原典: C-02（パスワード12文字以上・許可記号のみ）、C-03（表示名15文字以内）、C-04（ユニークID重複不可）
- 影響範囲: `Bbs.Validation`、DB のカラム定義、フロントの文字数カウンタ

## 背景

- **[事実]** 「12文字以上」「15文字以内」とだけ書かれ、**単位の定義がない**（D15）。
- **[事実]** 「英字・数字・記号をすべて含む／許可された記号のみ」と書かれるが、**「英字」の定義がない**。
- **[事実]** ユニークIDについて原典が言うのは**重複不可のみ**。文字種・長さ・大文字小文字の同一視は一切規定なし（D16）。
- `テストユーザー01` は 9 コードポイント / 23 バイト。単位次第で判定が変わる。

## 提案

1. **文字数は Unicode コードポイント単位で数える**（バイト数は誤り）。書記素クラスタとの差（結合文字・絵文字）は無視する。
   Lean の `String.length` がこの意味なので、モデルはそのまま使える。
2. **「英字」は ASCII の `a-z` / `A-Z` に限る**（`Validation.isAlpha`）。全角英字・アクセント付き文字は英字に数えない。
3. **パスワードは「英数字＋許可記号」以外の文字を一切許さない**。「許可された記号のみ」を最も厳しく読み、
   空白・日本語・その他 Unicode 文字を `disallowedChar` として弾く。
4. **ユニークIDは「空でない」以外の形式制限を置かない**。原典が要求していない制限を勝手に足さないため。
   大文字小文字は**区別する**（`TestUser` と `testuser` は別ID）。

## 決定（2026-07-19 ユーザー判断）

**提案どおり 1〜4 をすべて採用する。** 判断が要った 2 点は以下のとおり確定した。

### パスワードの文字種: 英数字＋許可記号以外を全拒否

「許可された記号のみ」を最も厳しく読む。日本語・空白・全角英数字・絵文字を含むパスワードは
`disallowedChar` で弾く。判定が一意でテストしやすく、実装のぶれが出ないため。

| 入力 | 判定 |
| :--- | :--- |
| `TestPassword123!` | OK（シナリオ01 で使用） |
| `password` | NG: `tooShort`, `noDigit`, `noSymbol` |
| `パスワーAbc123!!!` | NG: `disallowedChar` |
| `Test Password123!` | NG: `disallowedChar`（空白） |
| `TestＰassword123!` | NG: `disallowedChar`（全角Ｐ） |

### ユニークIDの大文字小文字: 区別する

`testuser_01` と `TestUser_01` は別ID。原典が要求していない正規化を足さないため。

## 影響

- **DB の照合順序に注意が要る。** PostgreSQL の `text` + `UNIQUE` はそのままで区別するが、
  **MySQL のデフォルト照合順序（`utf8mb4_general_ci` 等）は区別しない**ため、
  `utf8mb4_bin` / `utf8mb4_0900_as_cs` を明示指定しないとこの決定に反する。
  → フェーズ1 の DB 選定・スキーマ定義でチェックすること。decision 0011（検索の大文字小文字）と同じ落とし穴。
- パスワード文字種の判定はアプリ層で行う（DB は関与しない）。モデルの
  `Validation.passwordWeaknesses` がそのまま実装仕様になる。
- ユニークIDの**長さ上限は設けない**が、DB のカラム長は有限なので実装時に上限値を決める必要がある
  （原典由来の制約ではなく実装都合。`VARCHAR(255)` 程度で足りる）。
