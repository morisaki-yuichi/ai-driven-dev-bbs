# 0002 操作が失敗したときの状態（部分書き込みの禁止）

- ステータス: 決定済
- 起票日: 2026-07-19
- 関連論点: 新規（形式化により判明）
- 関連原典: なし（**原典は一言も述べていない**）
- 影響範囲: 全操作、トランザクション境界、DB スキーマ設計

## 背景

形式モデルの作用型を `Action α = StateM Db (Except Error α)` と置いた時点で、
**「失敗したとき、どの状態を返すか」が型では決まらない**ことが判明した。
`StateT Db (Except Error)` なら失敗時に状態は捨てられるが、`StateM Db (Except Error _)` は
「状態は必ず返る／値は失敗しうる」形なので、書き込み済みの中間状態をそのまま返すことも可能である。

- **[事実]** 原典には「操作が途中で失敗したとき部分的な変更が残ってよいか」の記述が一切ない。
- **[解釈]** シナリオ05-1-4→5→6（正常保存 → 16文字で失敗 → 再度正常保存）は、
  失敗した保存が**直前の保存値を壊さない**ことを暗黙に要求している。これが唯一の間接的な手がかり。

## 選択肢

1. **失敗時は操作開始前の状態へ完全に巻き戻す**（原子性）。
2. 失敗時も途中までの書き込みを残す。
3. 操作ごとに個別判断する。

## 提案

**選択肢1**。`Action.bind` を、失敗時に束縛前の状態を返すよう定義した（`formal/Bbs/Db.lean`）。

理由:
- 選択肢2 は、たとえば「ユーザー行を挿入した後に表示名バリデーションで失敗」といった状態を許すことになり、
  Wf 不変条件（`formal/Bbs/Invariant.lean`）を壊す。
- 実装上は「1リクエスト＝1トランザクション」に対応する。RDBMS を使うなら自然に満たせる。
- 巻き戻しを**明示的な定義**にしてあるので、`NoWriteOnError` として証明対象にできる（`Invariant.register_atomic` ほか）。

## 決定（2026-07-19 ユーザー判断）

**選択肢1（完全に巻き戻す）を採用する。**

## 影響

- **実装では、リクエスト単位でトランザクションを張ることが必須**になる。フレームワーク選定時に、
  リクエスト境界でのトランザクション管理が素直にできるかを評価軸に入れること（フェーズ1）。
- 形式モデル側では `Invariant.NoWriteOnError` として 4 操作分の言明を置いた（証明は未着手）。
  これは**満たすべき性質として確定**したので、証明が失敗したらモデルか実装のバグである。
- 永続化方式として RDBMS 以外（ファイル・KVS 等）を選ぶ場合、原子性を自前で用意する必要がある。
  H-03 は方式を自由としているが、この決定は **RDBMS を選ぶ動機**になる。

## 補足: 「1リクエスト＝1トランザクション」の字義どおりの読みに対する例外（2026-07-20）

**実質の要求は `NoWriteOnError`（失敗した操作の部分書き込みが残らないこと）であり、
「リクエストの最初の1バイトから最後までを1つのトランザクションで包むこと」ではない。**
トランザクション区間は、**その操作のすべての書き込みを含む最小の範囲**であればよい。

この区別が必要になった実例が `app/src/web/login.rs` である。当初はハンドラの入口から
トランザクションを開いていたが、認証情報の照合（読み取り）と **argon2 の検証**が
その内側に入っていた。argon2 は意図的に数十〜数百ミリ秒の CPU を使うため、
その間トランザクションが開いたままになり、sqlx プールの接続と PostgreSQL 側の
バックエンドを `idle in transaction` の状態で占有する。`POST /login` は**未認証で叩ける**
ので、同時ログイン試行が sqlx プール（既定 `max_connections` 10）を飽和させ、
同じプールを共有する**アプリ全体**を接続待ちで停止させうる。

そこで `login.rs` のトランザクション範囲を「リクエスト全体」から
**「検証通過後の書き込み（セッション作成）のみ」**へ狭めた。この操作の書き込みは
セッション作成の 1 回だけなので、`NoWriteOnError`（`formal/Bbs/Invariant.lean` の
`login_atomic`）は変わらず成立する。

適用範囲の判断基準:

- 高コストな純粋計算（ハッシュ化・検証など）や外部 I/O は、**トランザクションの外に置く**。
- 書き込みが複数ある操作では、それら**すべて**を1つのトランザクションに含める
  （範囲を狭める口実にしない）。
- 読み取った値を書き込みの前提にする場合（read-modify-write）は、読み取りも
  トランザクションに含める。`login.rs` はセッション作成が既存行に依存しない
  （decision 0007: 多重セッション許可）ため、この条件に当たらない。
