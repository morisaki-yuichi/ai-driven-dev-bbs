# 開発ワークフロー

- 位置づけ: ブランチ運用・コミットの粒度・コミットメッセージの書き方を定める。`CLAUDE.md` の「Definition of Done」「意図の置き場所」から参照される。

---

## 1. ブランチ運用

- **`main` は保護する。** 直接コミットしない。
- 機能・作業単位ごとに**短命ブランチ**を切り、完了したら `main` にマージしてすぐ消す。長期間残るブランチを作らない。
- ブランチ名は内容が分かる英小文字ケバブケース（例: `thread-creation`, `foundation-compose`）。プレフィックスは必須としない。

## 2. コミットの粒度

**1論理変更 = 1コミット。** 複数の意図が混ざった変更を1コミットにまとめない。「ついでに」見つけた無関係な修正は別コミットに分ける。

逆に、1つの論理変更を意味なく複数コミットへ分割しない（例: 「実装」「テスト」を別コミットにする、など）。テストとその対象実装がTDDのRed→Greenで一体なら同じコミットに含めてよい。

## 3. コミットのタイミング

**コミットはユーザーの承認を得てから行う。** 変更内容を提示し、ユーザーから明示的な指示（「コミットして」等）を受けるまで `git commit` を実行しない。承認なしに複数の変更を積み上げた場合も、まとめてではなく論理単位ごとに承認を取る。

## 4. コミットメッセージ

**Conventional Commits** に従う（`feat:` `fix:` `refactor:` `test:` `docs:` `chore:` 等）。

```
<type>: <ヘッダ = 簡潔なWhat>

<本文 = Why（なぜこの変更が必要か）>

Related: dev-docs/decisions/<番号>-<slug>.md（該当する場合）
```

- **ヘッダ**: 何をしたか（What）を簡潔に。命令形・現在形。
- **本文**: なぜその変更が必要だったか（Why）。**大きなWhy（仕様解釈・トレードオフの判断）は decision の内容をコミット本文に書き写さず、`dev-docs/decisions/<番号>-<slug>.md` への参照のみを書く。** 重複させない。本文に書くのは、その decision を踏まえてこの変更が必要になった、という繋がりの部分。
- 些末な変更（typo修正など）で本文に書くことが無ければ、ヘッダのみでよい。

### 例

```
feat: パスワードをargon2でハッシュ化して保存する

平文保存を避けるため。ハッシュ方式の選定はdecision 0016で確定済み。

Related: dev-docs/decisions/0016-tech-stack.md
```

## 5. 意図の置き場所（`CLAUDE.md` からの再掲・実務ルール）

コミット単位で見たとき、以下の使い分けを徹底する。詳細と理由は `CLAUDE.md` の「意図の置き場所」を参照。

| 置き場所 | 内容 | やらないこと |
| :--- | :--- | :--- |
| コミット本文 | Why（なぜこの変更をしたか） | 大きなWhyをdecisionから複製しない。参照のみ |
| コメント | Why・Why-not（非自明な理由、却下した代替案） | Howの再説明をしない |
| テスト | What（振る舞いの仕様） | 実装の内部手段をテストで縛らない |

これらはコミットレビュー時（＝ユーザーが変更内容を承認する前）にも確認する観点である。コミット本文にHowが書かれていたり、コメントが変数名を日本語で言い換えているだけだったりする場合は、コミット前に見直す。
