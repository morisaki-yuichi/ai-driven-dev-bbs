# UI/UX ガイドライン

- 位置づけ: 画面・フォームなどUIに関わる実装に着手する前に必ず読む（`CLAUDE.md` 参照）。評価が自然言語シナリオを `agent-browser` で解釈・実行する形で行われる（H-12、`dev-docs/harness-invariants.md`）ため、UIは**AIが自然言語で状態を観測でき、かつ操作できる**ことが常に前提になる。
- 個別画面の要素定義は `docs/product/designs/ui_design.md` と `dev-docs/requirements-analysis.md` §1.3（P01〜P07）が正典。本ファイルはそれらを実装するときに横断的に守るべき品質基準を定める。

---

## 1. 状態の網羅

ユーザーが操作する要素（フォーム、削除ボタン、ページ送りなど）は、起こりうる状態をすべて画面に表現する。**「その状態が来ないはず」という想定で分岐を省略しない**——分岐漏れは `agent-browser` 越しにしか気づけない、最もコストの高い欠陥になる（`CLAUDE.md` の実装方針、[decision 0016](decisions/0016-tech-stack.md) §4 のE2/E4）。

最低限、フォーム・操作ボタンごとに次の状態を意識する。

| 状態 | 要件 |
| :--- | :--- |
| 初期表示 | 未入力／初期値が正しく表示されている（例: P06の表示名は現在の値が初期値） |
| 送信中 | §2 参照 |
| 成功 | §2 参照。成功後の遷移先・表示が正しい |
| 失敗（バリデーションエラー） | §2・§3 参照 |
| 失敗（サーバエラー・権限エラー） | 共通メッセージエリアに異常系通知を出す。500相当を握りつぶして空白表示にしない |
| 操作不可（権限なし・条件未達） | ボタン自体を非表示にするか無効化する（例: 自分のスレッドでない場合は削除ボタンを出さない＝AC10-4。コメントが1件以上——**削除済みコメントも件数に数える**（C-06）——あるスレッドは削除ボタンを出さない） |
| 空データ | 一覧が0件のときに何も表示されない空白画面にしない（該当0件が分かる表示にする） |

## 2. 送信中／成功／失敗フィードバック

フォーム送信・削除操作などの非同期な操作は、**送信中・成功・失敗の3状態を必ず区別して表示する**。

- **送信中**: 二重送信を防ぐため、送信ボタンは処理完了まで再送信できない状態にする。SSR/MPA構成（[decision 0008](decisions/0008-browser-back-is-out-of-model.md)）なのでページ遷移そのものが送信中の合図になる場合が多いが、遷移前にボタンを無効化するなど、連打で二重投稿が起きないようにする。
- **成功**: 共通メッセージエリアに正常系通知を出す（例:「保存しました」「削除しました」、緑系推奨——`docs/product/designs/common_layout.md` に準拠、C-01/F14）。成功後の遷移先は画面ごとの定義（P01〜P07の遷移図）に従う。
- **失敗**: 共通メッセージエリアに異常系通知を出す（例:「IDまたはパスワードが正しくありません」、赤系推奨）。**失敗時に入力済みの値を消さない**（シナリオ05-1-5→6のような「失敗後に再度正しい値で保存」という操作をやり直しやすくするため）。

## 3. クライアント + サーバの二重バリデーション

すべてのユーザー入力（パスワード強度 C-02、表示名15文字 C-03、ユニークID重複 C-04、タイトル・本文の空 C-05）は**クライアント側とサーバ側の両方で検証する**。

- **サーバ側検証を省略しない。** クライアント側はUX向上のための即時フィードバックであり、`agent-browser` がJSをバイパスして直接POSTした場合や、JS無効時にも、サーバ側で最終的に弾けることが必須。
- **クライアント側検証はサーバ側と矛盾しない。** 判定基準（文字数の数え方はコードポイント単位——[decision 0003](decisions/0003-character-counting-and-charset.md)、空の定義とトリム方針——[decision 0004](decisions/0004-empty-input-definition.md)）を両側で揃える。
  - **サーバ側（Rust）の落とし穴**: `str::len()` は**バイト数**を返す。コードポイント数は `chars().count()` を使う。`テストユーザー01` は9コードポイント・23バイトであり（`requirements-analysis.md` D15）、`len() <= 15` と書くとこの正規の表示名を誤って弾く。
  - **クライアント側（JS）の落とし穴**: `String.length` は**UTF-16コードユニット数**であり、コードポイント数と常に一致するとは限らない（サロゲートペアを要する文字で乖離する）。表示名・タイトル・本文のような日本語入力を扱う以上、`Array.from(str).length` 等でコードポイント単位に揃える。
- バリデーションエラーは**該当フィールド付近に赤系テキストで表示**する（`docs/product/designs/common_layout.md` 準拠）。複数違反がある場合の粒度は [decision 0006](decisions/0006-validation-order-and-multiple-errors.md) に従う。

## 4. アクセシビリティ（a11y）

- フォーム要素には `<label>` を関連付ける（`for`/`id` またはネスト）。プレースホルダのみをラベル代わりにしない。
- 見出し・ランドマークにはセマンティックなHTML要素（`<header>` `<nav>` `<main>` `<footer>` `<form>` 等）を使う。
- ボタン・リンクはキーボードで到達・操作できること（`<div onclick>` のような非フォーカス要素にクリックハンドラだけを付けない）。
- 共通メッセージエリア（成功・失敗通知）は、動的に内容が変わることをスクリーンリーダーが検知できるよう `aria-live` 等の付与を検討する。
- 色だけに依存して情報を伝えない（緑=成功／赤=失敗はテキスト内容と併用する。色覚特性に関わらずメッセージの文言自体で判別できること）。
- 十分なコントラスト比を確保する。CSSは最小限のプレーンCSSでよいが（[foundation-plan §1.3](foundation-plan.md#13-テンプレート--描画askamassr--最小限のプレーン-css)）、可読性は犠牲にしない。

## 5. レスポンシブ

固定幅レイアウトで極端に小さい・大きいビューポートで操作不能にならないようにする。相対単位（`%`, `rem`, `flex`/`grid`）を基本とし、テーブル状の一覧やページネーションが画面幅からはみ出す場合はスクロール可能にするなど、要素が失われない形にする。凝ったブレークポイント設計は要求しない（H-02はCSSなしのDOMだけでも可としている）が、**操作対象が画面外に隠れて到達不能になる状態は避ける**。

## 6. `agent-browser` での操作可能性（H-02）

評価はAIが `agent-browser` でUIを直接操作して行う。**見えない要素にアクションを強いたり、極端に操作性の悪いデザインを避ける**（H-02）。

- **隠し要素にアクションを強いない。** ホバーでのみ現れるメニュー、スクロールしないと存在に気づけない操作、CSSで視覚的には隠しているが操作自体は必要、という設計をしない。操作可能なボタン・リンクは常に見える形で存在させる。
- **ネイティブダイアログ（`window.confirm` 等）は避ける。** 削除確認ダイアログを実装する場合（D18、任意）、`window.confirm` は `agent-browser` からの検知・操作性に影響しうるため、HTML内のモーダル要素など通常のDOM操作で完結する方式を使う。
- **状態変化は画面内のDOMに反映する。** アラート的なポップアップやトースト消失が速すぎるなど、AIが自然言語で読み取る前に消える表現は避ける。
- 検索結果からのスクロール連携（AC11-3、D19）は、URLフラグメント等で遷移直後に対象要素がビューポート内にあることが確認できる形にする。
- ページネーションの「前に戻る」「次に進む」は、境界（1ページ目・最終ページ）で**要素ごと非表示にする**（C-12、AC09-5）。無効化してラベルを残すだけの実装は不可——C-12は「表示しない」ことを要求しており、`agent-browser` が「ボタンが存在する」と観測してしまう余地を残さない。

## 7. 認証ガード・エラー画面との整合

- 未ログインで認証必須URLにアクセスした場合は一律ログイン画面へリダイレクトする（C-09）。個別画面でガードを書き漏らさないよう、`web/middleware.rs` に集約する（[foundation-plan §5](foundation-plan.md#5-実装前に作る基盤の一覧) #7）。
- 存在しない・削除済みリソースへのアクセスは一律エラー画面（404相当、C-10）。エラー画面には「トップページへ戻る」導線を必ず置く（P07）。
- ログアウト後のブラウザバックで認証必須画面のキャッシュが表示されないよう、該当画面に `Cache-Control: no-store` を付与する（C-11 / AC03-2）。
- 一覧・ページネーションの文言や実装で、**総件数や「特定スレッドが何ページ目にあるか」を固定的に仮定しない**。評価シナリオは01→05を先行データが残った状態で逐次実行するため（H-08〜H-11の合成的帰結、`dev-docs/harness-invariants.md`）、件数は実行結果に依存する。「全10件」のような決め打ち表示・ページ番号のハードコードをしない。

## 8. 固定文言（C-01）

`requirements-analysis.md` C-01 が**厳密一致**を要求する4文言。実装時にコピー&ペーストして使う（記憶で打ち直さない。特に1つめは全角山括弧であり、半角 `<>` や `《》` との取り違えが起きやすい）。

| 用途 | 文言 | 出現箇所 |
| :--- | :--- | :--- |
| 削除済みコメントの本文置換 | `＜このコメントは削除されました＞` | AC08-2 / AC10-3 / F08（`thread_detail.html` 等、コメント表示テンプレート） |
| ヘッダーのアプリタイトル | `AI掲示板（仮）` | F14（`layout.html`） |
| フッター | `© 2026 AI駆動開発教材プロジェクト` | F14（`layout.html`） |
| ヘッダーのユーザー表示 | `[表示名] さん` | F14（`layout.html`。`[表示名]` 部分のみ動的） |

これらはAskamaのコンパイル時型検査では**守られない**（フィールドの有無は検査されるが、文字列リテラルの正しさは検査対象外）。[decision 0016](decisions/0016-tech-stack.md) §4 がE4（テンプレート不整合）の代表例として名指ししている箇所であり、人間／AIの注意にのみ依存する。実装・レビュー時にこの表と字句を直接突き合わせること。
