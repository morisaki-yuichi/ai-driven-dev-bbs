## 指摘事項

### 重大

### 中

#### 中-1: 0016 §6.2 の取り返し勘定に、自らの A 採点が「差は現金化されない」とした E3 を再計上している

- **箇所**: `0016-tech-stack.md` §6.2 項目3。
- **内容**: §5.1 は「E3 は3案とも ◎。強度は R > G ≈ T だが、本件の E3 は『列名 typo』『JOIN の取り違え』の水準であり、3案とも同じように捕まえる。**差は現金化されない**」と採点した。さらに案T については「Drizzle は `schema.ts` がマイグレーションの生成元でもあるためドリフトが**構造的に起きない**」と自ら論証している。ところが §6.2 項目3 は、B 超過を取り返す項目として「`sqlx::query_as!` は実 DB に対して SQL を検証する（案T・案G はスキーマ宣言に対する検証にとどまる）。**マイグレーションを書き換えた直後にクエリ側を直し忘れる、という E3 の典型がビルド時に落ちる**」を挙げる。この具体シナリオは、§5.1 の自らの論証によれば案T では構造的に起きず、案G でも sqlc が `schema.sql`（マイグレーションの正）に対して検証するため同様に捕まる。つまり **A 採点で「差がつかない」と判定した項目を、B の回収勘定では差があるかのように再計上している**。
- **影響**: §6.2 の取り返し勘定は実質、項目1（E2 ○→◎）・項目2（E5 ○→◎）・項目4（儀式頻度の相対比較）の3本で立っており、項目3 を除いても §6.2 末尾の「純粋な損失を承知のうえで E を理由に選んだ」という結論の形は変わらない。ただしこの決定記録は「AI の判断過程の正確な記録」自体が教材価値である以上、勘定の水増しは残すべきでない。
- **提案**: 項目3 を勘定から外すか、「宣言に対する検証と実 DB に対する検証の差」が**本件で**現金化される具体シナリオ（§5.1 の論証と両立するもの）を示して残すか、いずれかに改訂する。

### 軽

#### 軽-1: G-3 の Tier 0 配置が、自ら宣言した「空ゲートを置かない」規則と体裁上不整合

- **箇所**: `0016-tech-stack.md` §3 の表。
- **内容**: §2 は「実際に候補を落とせる基準のみを Tier 0 に置く。どの候補も落とさない基準は空のゲートなので Tier 0 に置かない」と宣言しているが、G-3 は「落ちる候補はない」と自認しつつ Tier 0 の表内に置かれている。「Tier 1 相当の減点として扱った」と明記されているため実害はない（自覚的である）が、減点先の軸（A か B か C か）が明示されておらず、foundation-plan §1.5 では「G-3」がそのまま採用理由の軸ラベルとして使われている。0011 の照合により、SQLite/MySQL への減点自体の事実関係（既定が決定済み仕様を静かに破る）は正確であることを確認した。
- **提案**: G-3 を表から「Tier 1 減点に転換した基準」として節を分けるか、減点先の軸を一言で明示する。

#### 軽-2: 軸ラベル「G」（Tier 2: 再現性）と「G-1〜G-3」(Tier 0 ゲート) の名前衝突

- **箇所**: `0016-tech-stack.md` §2・§3、`foundation-plan.md` §1（冒頭の凡例、§1.2、§1.5、§1.9）。
- **内容**: Tier 2 の「G. 再現性のあるビルド」と Tier 0 ゲートの「G-1〜G-3」が同じ文字を共有しており、G-1〜G-3 が G の細目に見える。実際には別系統である（§1.2 の Rocket 却下「＝G」は再現性、§1.5 の「G-3」はゲート由来の減点）。foundation-plan §1 冒頭は「A〜G / G-1〜G-3」と併記して両者の存在は示しているが、読み手が誤読しうる。
- **提案**: ゲート側を「T0-1〜T0-3」等に改名するか、凡例に一文加える。

#### 軽-3: B 緩和策の mold / lld リンカ指定が評価者の Docker ビルド経路へ漏れた場合のリスクが未記載

- **箇所**: `foundation-plan.md` §5 #14・§6。
- **内容**: mold / lld のリンカ指定を `.cargo/config.toml` 等でリポジトリにコミットすると、その設定は評価者経路の Docker ビルド（コンテナ内 `cargo build`）にも効く。Dockerfile 側に mold / lld が入っていなければ **clone 直後のビルドが失敗し H-06 / H-13 を破る**。計画には「開発側（ホスト cargo）の緩和策」とだけあり、評価者経路に漏らさない（または Dockerfile に同梱する）旨の一文がない。実装時に自然に気づける可能性は高いが、H-06 は計画自身が「最も後戻りコストが高い」と位置づけた不変条件なので、計画段階で一言あるべき。
- **提案**: §6 の表に「リンカ指定は評価者の Docker ビルドを壊さない形にする（ホスト側限定の設定にするか、Dockerfile に同梱して検証する）」を追記する。

## 観点ごとの判定

### 観点1: 選定軸の一貫適用 — 問題なし（軽-1・軽-2 の体裁上の注記を除く）

- 0016 §3〜§6 の比較は宣言どおり Tier 0（G-1・G-2 で実際に候補を落とす）→ Tier 1（A を §4 のエラー面 E1〜E5 で、B を §5.2 の摩擦のみで採点）→ Tier 2（E がタイブレークとして決着、§6.1 に明記）の階層で進んでいる。
- foundation-plan §1 の各レイヤの理由付けも軸ラベル（A / B / C / D / E / F / G / G-1〜G-3）で 0016 §2 の定義に遡れる形になっている。

### 観点2: A / B の定義ガードレール — 概ね遵守（中-1 のみ逸脱）

- **A の relevant 採点**: §4 はエラー面 E1〜E5 を正典の制約（C-10、C-01、C-12、C-16、0015 の JOIN 等）から具体的に導出し、「E3・E4 が支配的」「評価は agent-browser 越しなのでランタイム露出が最も高コスト（H-12）」という重み付けの根拠を示している。型システムの絶対的強度による採点にはなっていない。
- **静的保証の論証つき判定**: §4 末尾は「全状態は PostgreSQL 上（0002・0007 の帰結）→ アプリプロセス内に共有可変状態が存在しない → 所有権・借用検査・Send/Sync が守る対象はほぼ空」というこのアプリ固有の論証を経て、採用案自身の看板の利点を A から明示的に除外している（論証なしの断定ではない）。逆方向も一貫しており、GC 候補への「データ競合リスク」による D 減点も行っていない。候補ごとの論証も揃っている（Go: ゼロ値が E2 に静かな穴／Python: Jinja の未定義変数と非検査 SQL／T: `never` チェックの規律依存）。
- **B の純粋摩擦限定**: 「コンパイル段がある／型があること自体は罰しない」を宣言し、借用検査の反復を「§4 の論証により捕まえたバグを伴わない純粋な摩擦」として B に正しく分類している。cycles-to-green は摩擦（減点）側に置かれており、shift-left の便益として偽装されていない。早期検出の便益は A に1回だけ計上され、B で二重に減点されていない。
- **E の密輸なし**: §6.1 が「Tier 1 のみなら案T が推奨だった。採否を分けたのは Tier 2 の E（Lean 対応を重視するユーザー判断）」と事実どおり記録し、A には Lean 対応の「検出可能性の部分のみ」を算入したと明示している。§5.1 の該当箇所（網羅マッチでモデルとの乖離がコンパイル時に出る）を確認したが、これは E5 の検出可能性であり A の定義内である。
- **実現可能性のゲート化回避**: 「AI が完成させられる」等は空ゲートとして Tier 0 から外し、程度差を B・C に回したことが §3 に明記されている。
- **高 B の取り返し論証**: §6.2 が存在し、しかも「取り返せていない部分」（テンプレート1行修正の数十秒・借用検査反復）を純粋な損失として明記したうえで E によるユーザー判断に帰属させている。論証の存在と誠実性は要求どおり。ただし勘定の項目3 に内部矛盾がある（→ 中-1）。

### 観点3: 比較の公平性 — 問題なし

- 代替案はストローマンではない。案T は Hono + JSX + Drizzle、案G は templ + sqlc という、各言語で E3 / E4 を shift-left できる最良級の構成（strong-man）で組まれている。
- 採用案に不利な事実の開示が徹底している: 「B は3候補中最下位」（0016 §5.2・foundation-plan §1.1）、「Tier 1 のみの採点では案T が推奨だった」（§6.1）、「Askama が C の最大リスク箇所」（§7.1・foundation-plan §1.3）、SQLite の「B・F・D では優位だった」（foundation-plan §1.5）、maud が「B と C ではこちらが優位」で第一代替（§1.3）。
- 却下理由も検証可能な形で書かれており（Tera のランタイム評価、Go のゼロ値、Jinja の既定挙動）、0011 の照合で確認した範囲では事実関係に誤りはない。

### 観点4: Tier 0 制約適合 — 問題なし

採用スタック（Rust + Axum + Askama + sqlx + PostgreSQL 17、SSR / MPA、compose の app + db）は:

- G-1（SSR / MPA）: Axum + Askama のサーバサイドレンダリング構成で充足。
- G-2（RDBMS + リクエスト単位トランザクション + セッション DB 永続化）: PostgreSQL + §3 の「1リクエスト＝1トランザクション」規律 + `sessions` テーブルで充足。
- G-3（`LIKE` の大文字小文字区別）: 0011 本文の表と照合し、PostgreSQL は既定のまま充足することを確認。
