## 1. 総評

フェーズ2 成果物の骨格は良好である。特に次の点は正典を正確に踏まえている。

- CLAUDE.md の「確定スタック」表は 0016 §6 の採用レイヤ表と**行単位で一致**しており、改変・追加は無い。「このスタックの採否をこのプロジェクトの中で再判断しない」という凍結宣言も置かれている。

## 2. 重大

### 重大-1 `ui-ux-guidelines.md` §6 のページネーション記述が C-12 に反し、かつ同一文内で自己矛盾している

**該当**: `dev-docs/ui-ux-guidelines.md` §6 最終項

> ページネーションの「前に戻る」「次に進む」は、境界（1ページ目・最終ページ）で**要素ごと非表示にする**（C-12）。無効化のみでラベルを残す実装は許容されるが、`agent-browser` が「表示されていない」ことを判定できる形（非表示または明確なdisabled状態）にする。

**問題**: 正典 `requirements-analysis.md` の C-12 および F13 は「1ページ目では『前に戻る』を、最終ページでは『次に進む』を**表示しない**」と定めている（AC09-5 が検証）。「無効化のみでラベルを残す実装は許容される」は正典に無い緩和であり、**制約の改変**にあたる。加えて前段で「要素ごと非表示にする」と断定した直後に許容を書いており、実装者はどちらに従えばよいか決められない。

**影響**: シナリオ04-3 で `agent-browser` が「『前に戻る』が表示されていない」を確認する。disabled ボタンが残っていると、AI は「前に戻るボタンは存在する」と観測しうる。H-12（自然言語での観測）の下では、この差はそのまま判定の揺れになる。C-12 は「表示しない」だけを要求しているので、緩和する理由も無い。

**推奨**: 「無効化のみでラベルを残す実装は許容される」を削除し、「要素ごと非表示にする」に一本化する。

### 重大-2 C-01 の固定文言が、フェーズ2 成果物のどこにも転記されていない

**該当**: `CLAUDE.md` 全体、`dev-docs/ui-ux-guidelines.md` 全体

**問題**: `requirements-analysis.md` C-01 は4つの固定文言の**厳密一致**を要求する。

| 文言 | 出現箇所 |
| :--- | :--- |
| `＜このコメントは削除されました＞`（全角山括弧） | AC08-2 / AC10-3 / F08 |
| `AI掲示板（仮）` | ヘッダー / F14 |
| `© 2026 AI駆動開発教材プロジェクト` | フッター / F14 |

このうち**一つもフェーズ2 成果物に字句として現れない**。CLAUDE.md は「UI/UX 必須要件」節で `ui-ux-guidelines.md` を読めと指示し、Definition of Done も同ファイルの充足をチェック項目にしているが、その `ui-ux-guidelines.md` は §2 で「C-01/F14」と番号参照するのみで、字句を一つも列挙していない。**UI 作業の直前に読むよう設計された唯一の文書が、字句一致要求の存在を実質的に伝えていない。**

**影響**: 特に `＜このコメントは削除されました＞` は、全角山括弧 `＜＞` を半角 `<>` や `《》` に取り違えると AC08-2 が落ちる。0016 §4 は E4（テンプレート不整合）の具体例としてまさに「固定文言 `＜このコメントは削除されました＞` の埋め込みミス」を挙げ、これを**支配的エラー面**と位置づけている。にもかかわらず、Askama が検査できるのは「フィールドの有無」であって「文字列リテラルの正しさ」ではない。この一点は型検査で守られず、**人間／AI の注意にしか依存していない**のに、注意を促す文書が存在しない。

**推奨**: `ui-ux-guidelines.md` に「固定文言（C-01）」節を設け、4文言をコピー可能な形で列挙する。あわせて Definition of Done に「C-01 の固定文言を変更していない／新規に埋め込んだ場合は字句一致を確認した」を追加する。

### 重大-3 Stop フックの transcript 検査が、実質的にリマインダを恒久沈黙させる

**該当**: `.claude/hooks/remind-decision-log.sh` 22〜28行目

```bash
if grep -qE '/log-decision|記録不要' "$TRANSCRIPT" 2>/dev/null; then
  exit 0
fi
```

**問題**: 検査対象がセッションの transcript 全体であり、**「AI が `/log-decision` を実行した」以外の経路でこの文字列が transcript に載る**。具体的には少なくとも次の3経路がある。

1. **このフック自身の `systemMessage`**（30行目）が `/log-decision` という文字列を含む。リマインダが一度でも出れば、その systemMessage が transcript に記録され、以後同一セッションで二度と発火しない。
2. **`CLAUDE.md` の本文**が `/log-decision` を3箇所（決定記録ルール節、Definition of Done、`.claude/skills/` への言及）に含む。プロジェクト指示としてコンテキストに載る内容が transcript に記録される実装であれば、**セッション開始直後から常に一致する**。
3. ユーザーやAIが「今回は decision の記録不要ですね」と会話で触れただけで一致する。「記録不要」は極めてありふれた語であり、否定文（「記録不要とは言えない」）でも一致する。

依頼の観点は「恒常的な誤検知を起こさないか」だが、このフックの失敗様式は**誤検知ではなく恒常的な取りこぼし（false negative）**である。リマインダとしての実効性がほぼ失われている。

**補足（現時点では発火しないこと自体は仕様どおり）**: 18行目の `git status --porcelain -- app/src` は、`app/src` が未作成の現在は常に空を返すため、フェーズ2 時点で発火しないのは正常。したがって上記の欠陥は**実装フェーズに入って初めて表面化する**。今のうちに直すべき理由がここにある。

**推奨**: transcript 全体の grep をやめ、次のいずれかに置き換える。

- `.claude/` 配下や `/tmp` にセッションIDをキーとした「抑止フラグファイル」を置き、`/log-decision` の実行後および「記録不要」宣言時に AI がそれを作る運用にする。
- transcript が JSONL であることを利用し、`jq` で**ユーザー発話メッセージのみ**を抽出してから照合する（`systemMessage` と CLAUDE.md 由来の一致を排除できる）。
- 抑止条件を「`dev-docs/decisions/` に未コミットの新規ファイルがあるか」という**成果物ベース**に変える（文字列照合を完全に廃止でき、最も頑健）。

## 3. 中

### 中-1 C-06 の「削除済みコメントも件数に数える」が `ui-ux-guidelines.md` から脱落している

`ui-ux-guidelines.md` §1 の「操作不可」行は「コメント1件以上あるスレッドの削除ボタンも同様」とのみ書く。正典 C-06 は「作成者本人 **かつ** コメント0件（**削除済みコメントも件数に数える**）」であり、括弧内が C-06 の核心である（AC06-2 が名指しで要求、シナリオ03-2 が「削除済み1件＋通常1件」の状態で直接検証する）。0016 §4 も E5 の具体例として「C-06 の条件取り違え」を挙げている。foundation-plan §3 には正しく書かれているが、UI 実装時に開く文書側で条件が緩んで見えるのは危険。

### 中-2 Cookie 属性の転記が不完全で、「セキュリティ必須要件」節と衝突しうる

CLAUDE.md の確定スタック表は「HttpOnly / SameSite=Lax Cookie」までしか書かない。foundation-plan §1.6 はこれに加えて **`Path=/`**、**「`Secure` は HTTP 評価環境のため付けない」**、**「有効期限は設けない（0007）」** を定めている。特に `Secure` を付けない判断が落ちていることが問題で、CLAUDE.md には別途「セキュリティ必須要件」節があるため、実装時に「セキュアにする」方向へ寄せて `Secure` を付けると、**HTTP で動く評価環境ではブラウザが Cookie を送出せず、ログイン状態が一切維持できない**（AC02-4 全滅）。落とした理由が明示されていない省略は、後から「付け忘れ」に見えてしまう。

### 中-3 Playwright MCP が有効化されているが、H-12 との関係がどこにも整理されていない

`.mcp.json` は `playwright` MCP を定義し、`.claude/settings.json` の `enabledMcpjsonServers` がそれを有効化している。一方 CLAUDE.md は冒頭で「Playwright等の自動テストランナーは使われない（H-12）」と明記する。**両者は直接には矛盾しない**（H-12 は評価者側の手段の話で、開発中の AI がブラウザ確認に使うことを禁じてはいない）が、その位置づけが成果物のどこにも書かれていない。

実害は2つ。(a) 開発中の検証を Playwright だけで済ませると、`ui-ux-guidelines.md` §6 が定める `agent-browser` 固有の操作性（隠し要素・ネイティブダイアログ・トーストの消失速度）を検証しないまま通してしまう。(b) 将来のセッションが「Playwright があるのだから E2E テストを書こう」と解釈し、H-12 が想定しない資産を積む。CLAUDE.md に一文（「Playwright MCP は開発中の目視確認用であり、評価は `agent-browser` による自然言語判定で行われる。Playwright ベースの自動テストは作らない」）を置くべき。

### 中-4 Rust 固有の文字数カウントの落とし穴が明示されていない

`ui-ux-guidelines.md` §3 は「文字数の数え方はコードポイント単位（decision 0003）」と正しく参照している。しかし採用スタック固有の事実——**Rust の `str::len()` はバイト数を返し、コードポイント数は `chars().count()`** ——が、CLAUDE.md にも guidelines にも無い。requirements-analysis D15 は「`テストユーザー01` は9コードポイント・**23バイト**」と具体例まで挙げており、`len() <= 15` と書けばこの正規の表示名が弾かれる。0003 が「コードポイント」と決めていても、Rust では最も自然に書ける式が誤りになるため、スタック固有の注記が要る。バリデーションはクライアント側（JS の `String.length` は **UTF-16 コードユニット**）とも数え方が異なり、guidelines §3 の「両側で揃える」を満たすには両方の注記が必要。

### 中-5 Askama のバージョン固定と「記憶で書かない」規律が CLAUDE.md に転記されていない

0016 §7.1 は「**Askama のバージョンは実装初回に実測で確定・固定する**」「rinja のフォークと統合を経て API に変遷があり、**C（生成信頼性）で本スタック最大のリスク箇所**」「バージョンを `Cargo.toml` でピン留めし、公式ドキュメントを参照して書く（記憶で書かない）」と明記している。foundation-plan §1.3・§5 #1 も同旨。**0016 が自ら「本スタック最大のリスク」と名指しした唯一の項目**であるにもかかわらず、恒久ルールである CLAUDE.md には一切現れない。

しかも `.mcp.json` の `context7` はまさにこの用途（最新ライブラリドキュメントの参照）に合致するのに、CLAUDE.md はその接続を書いていない。「Askama を書くときは context7 で API を確認する」という一文があれば、MCP 設定が単なる環境設定から運用ルールに昇格する。

### 中-6 「1リクエスト＝1トランザクション」（0002・critical）が CLAUDE.md に無い

0002 は `decisions/README.md` の critical 索引に載る4件のひとつで、foundation-plan §3 は「ハンドラの入口でトランザクションを開始し、`Err` を返す経路では必ずロールバックする。**この規律をミドルウェアまたはヘルパで1箇所に固める**」と実装制約まで定めている。CLAUDE.md の「実装方針: Action / Calculation / Data の分離」節は副作用の隔離を語りながらトランザクション境界に触れず、Definition of Done にもチェックが無い。critical 決定のうち 0007（セッション）・0008（SSR/MPA）・0016（スタック）は CLAUDE.md に反映されているのに、0002 だけが漏れている。

### 中-7 mold / lld のリンカ設定が評価者経路を壊す制約が、CLAUDE.md にも DoD にも無い

foundation-plan §6.1 は、この点を「**実装時の気づきに頼らず制約として明記する**」と宣言し、`.cargo/config.toml` にリンカ指定をコミットすると Docker ビルドが失敗し **H-06 と H-13 を破る**と警告している。にもかかわらず、恒久ルールである CLAUDE.md にはリンカの「リ」も無い。CLAUDE.md には「評価者経路の検証」というコマンド節があり `docker compose up` を「定期的に確認する」と書いてはいるが、Definition of Done にはこの検証項目が入っていない。**「定期的に」は誰も実行しない頻度である。**

**推奨**: DoD に「ビルド設定（`.cargo/config.toml` / `Cargo.toml` / `Dockerfile`）を変更した場合、クリーンな clone から `docker compose up` が通ることを確認した」を追加する。

### 中-8 `log-decision` スキルが「コード側に同じ番号のコメントを残す」を指示していない

`decisions/README.md` の decided_by 表と CLAUDE.md「決定記録ルール」の両方が、`decided_by: ai`（＝未承認の暫定決定）について **「コード側にも同じ番号のコメントを残すこと」** を要求している。これは「レビューされていない前提の上に実装が積まれている」状態を後から洗い出すための唯一の仕掛けである。しかし `log-decision/SKILL.md` は status を `提案` にし、報告時に「未承認の暫定決定であること」を伝えるところまでで止まり、コードコメントの規律に触れていない。スキルは「番号採番から索引更新まで抜け漏れなく機械的に行うために存在する」と自称しているのだから、この一項も手順に含めるべき。

### 中-9 `.claude/settings.json` の `human-guide/` 保護が `Read` のみ

CLAUDE.md は「`human-guide/` を**読み込み・編集しない**」と定めるが、settings.json の deny は `Read(./human-guide/**)` 一つだけである。`Edit` / `Write` は塞がれておらず、`Grep` / `Glob` / `Bash(cat ...)` 経由の読み取りも通る。ルール文が禁じている範囲より機械的な保護が狭い。少なくとも `Edit(./human-guide/**)` と `Write(./human-guide/**)` を追加すべき（Grep/Glob 経路まで塞ぐのは現実的でないため、CLAUDE.md 側の規律に委ねる整理で構わないが、その旨を書いておくと後任が誤解しない）。

### 中-10 CSRF / セッション鍵と H-07 の関係が未明示

CLAUDE.md は「CSRF対策：方式の詳細はフェーズ2以降の decision で確定する（D05）」と正しく留保している。しかし **H-07（秘匿キーがある場合のみユーザーに伝える。それ以外はすべて AI が自動実行できること＝対話的な手作業を要求しない）** との接続が書かれていない。foundation-plan §2 は `.env.example` に「秘匿キーは無い想定 / H-07」と注記しており、**「アプリが起動時に人間の入力を要する秘密鍵を要求してはならない」が既に上流の前提になっている**。

CSRF トークンやセッション Cookie 署名に固定鍵を導入し、それを必須環境変数にすると、評価者は clone 直後（H-13）に値を用意できず `docker compose up` が起動しない。D05 を決める時点でこの制約が視界に入っていないと、素直な実装（`SECRET_KEY` 必須）を選んで H-06／H-07 を破る。D05 起票時の前提として、CLAUDE.md か foundation-plan §7 に一行の注記が要る。

## 4. 軽

### 軽-1 P-08 の転記で「事後承認を得る」が「報告する」に弱まっている

decision 0001 と requirements-analysis P-08 はいずれも「実施後に人間へ報告し、**承認を得る**（事後承認）」と書く。CLAUDE.md は「実施後に人間へ報告する」で止まっている。運用上の差は小さいが、承認プロセスの有無は他の箇所（コミット承認）と揃えるべき性質の記述。

### 軽-2 `log-session` スキルの手順4が「5項目」と言いながら4項目しか列挙していない

`session-log-format.md` の見出し構成は **フェーズ／今回やったこと／決めたこと／次にやること／未解決事項** の5項目。`log-session/SKILL.md` 手順4の箇条書きは「フェーズ」を欠いた4項目で、その直後に「5項目はすべて必須」と書いている。スキルは「形式を複製しない」方針なので、箇条書き自体を削って `session-log-format.md` の参照に一本化するのが方針とも整合する。

### 軽-3 `Bash(sqlx migrate*)` の許可が方針と噛み合わない

foundation-plan §4.1 は「`sqlx::migrate!` でバイナリに埋め込み、起動時に自動適用」「評価者が別途マイグレーションコマンドを打つ手順は発生しない」と定める。開発中も同じ経路を通るため、`sqlx migrate` を手で叩く場面は原則存在しない。許可があること自体は無害だが、**「手動マイグレーション経路もある」という誤ったシグナル**になり、compose 経路とホスト経路が同じマイグレーションを使うという §4.1 の構造的保証を崩す入口になりうる。

### 軽-4 再現性（G）の固定資産が CLAUDE.md に無い

foundation-plan §1.9・§5 #1 は `Cargo.lock` と `rust-toolchain.toml` のコミットを再現性の要件としている。CLAUDE.md には記述がなく、`.sqlx/` だけが DoD に入っている。同じ「コミット必須の生成物」という括りなので、並べて書くほうが落ちにくい。

### 軽-5 `cargo fmt` が Definition of Done に無い

主要コマンドには載っているが DoD のチェックリストには `cargo test` と `cargo clippy` しかない。差分ノイズの抑制はレビュー可能性に直結するため、DoD に入れるか、主要コマンドから落とすかのどちらかに揃えるべき。

### 軽-6 `log-decision` の索引更新手順が、README の説明文の陳腐化に触れていない

手順4-2 は「『該当なし（0件）』の文言だった場合は削除して表形式に変える」と指示する。しかし `decisions/README.md` の当該節にはその後に説明文が続き、「2026-07-19 時点で、0001〜0016 の**すべてが `ai+user`**」と書かれている。表形式に変えてもこの文が残れば、索引が自己矛盾する。手順に「同節の説明文も更新する」を加えるとよい。

### 軽-7 H-08〜H-11 の合成的帰結が `ui-ux-guidelines.md` に反映されていない

harness-invariants の「合成的な帰結」は **「一覧の総件数や『特定スレッドが何ページ目にあるか』を固定的に仮定した実装・文言にしてはならない」** と定める。これは画面の文言に直接効く制約（例:「全 N 件中」の固定表示、ページ番号の決め打ち）だが、`ui-ux-guidelines.md` §1 の「空データ」行は0件時の表示にしか触れていない。DoD 経由で harness-invariants を参照する経路はあるので欠落は軽微だが、UI 側の文書に一行あると確実。

### 軽-8 `settings.local.json` が `Bash(git commit *)` を許可している

CLAUDE.md（「ユーザーから明示的な指示がない限り commit しない」）と workflow §3（「ユーザーの承認を得てから」）に対し、権限設定は commit を無確認で通す。permission allow は「確認プロンプトを省く」だけでルールを上書きしないので**規約違反ではない**が、承認ゲートが CLAUDE.md の文面だけに依存する構成になる。ローカル設定なのでチーム共有の問題は無い。指摘に留める。

### 軽-9 フックの `jq` 依存

`remind-decision-log.sh` は `jq` を使う。H-05 の前提ツール4点に `jq` は含まれないが、**これは開発者側のフックであり評価者経路には現れない**ため H-05 違反ではない。`jq` 不在時は `|| echo false` で握り潰され、transcript を取得できずリマインダが出る側に倒れる（安全側）。動作は壊れない。念のための記録。

## 6. 上流エスカレーション

**1件のみ記録する。その場で修正はしていない。**

### `foundation-plan.md` §5 の基盤一覧に、固定文言 `＜このコメントは削除されました＞` の集約先が割り当てられていない

foundation-plan §5 は「実装前に作る基盤の一覧」14項目を挙げ、#9 で `templates/layout.html` を **「C-01 の固定文言（`AI掲示板（仮）` / `© 2026 AI駆動開発教材プロジェクト` / `[表示名] さん`）を1箇所に固定」** と定義している。ここで列挙されている C-01 の文言は**3つ**であり、4つめの `＜このコメントは削除されました＞` が含まれていない。

これはヘッダー／フッターに属さない文言なので `layout.html` に置けないのは正しい。問題は、**§5 の他のどの項目にもこの文言の集約先が無い**ことである。§5 の設計思想は「横断的制約（404・認証・固定文言・クエリパラメータ）を1箇所に固める層を先に作らないと、各機能に散らばって漏れが生じる」（同節「着手順の意図」）というものであり、C-01 の4文言のうち**最も字句一致がクリティカルな1つだけ**（AC08-2 がシナリオ03 で直接検証し、0016 §4 が E4 の代表例として名指しした文言）が、その保護から外れている。

結果として、この文言は `thread_detail.html` のテンプレート内リテラルとして書かれ、Askama の型検査でも守られない位置に置かれることになる。§5 に「削除済みコメント表示の固定文言を定数として1箇所に置く」項目を足すか、#9 の説明を C-01 全体をカバーする形に改めるのが望ましい。

**これはレビュー時点の観察であり、上流の判断を蒸し返す意図はない。フェーズ3 の着手時に上流側で扱いを決めてほしい。**
