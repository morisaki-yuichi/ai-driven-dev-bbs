## H-01 ブラウザで動作すること
アプリケーションはブラウザ上で動作する必要がある。
出典: `docs/product/technical_spec.md` §2 / `docs/misc/人が最初に書いたメモ.md` / `README.md` §2

## H-02 AIが `agent-browser` で操作できるUIにすること
AIが `agent-browser` を用いて動作確認を行うため、見えない要素にアクションを強いたり、極端に操作性が悪いデザインは避ける。CSSの有無自体は自由（一切CSSのないHTMLのDOMだけでも可）。
出典: `docs/product/technical_spec.md` §2 / `docs/misc/人が最初に書いたメモ.md`

## H-03 データ永続化は必須
保存方法（DB／ファイル／インメモリ等）の選定はAIに任されるが、**永続化が行われるように実装**すること。登録したユーザー情報が永続化され、その情報でログイン可能な状態になっていることが受け入れ基準に含まれる。
出典: `docs/product/technical_spec.md` §2 / `docs/product/issues/01_user_registration.md` AC

## H-04 ローカルPCにミドルウェアがある前提にせず、Dockerで起動すること
ローカルPCに直接ミドルウェア（PostgreSQL, Redis等）がインストールされていることは前提としない。必要な場合はDockerコンテナとして起動するように設定する。
出典: `docs/product/technical_spec.md` §2 / `docs/misc/人が最初に書いたメモ.md`

## H-05 前提としてよいツールは4点のみ
1. **git** がインストールされていること
2. **Docker / Docker Compose** がインストールされていること
3. **OS** は Ubuntu（WSL等）または Windows（**macOS固有のライブラリ依存等、Macでしか動作しないシステムは許可されない**）
4. 選定した技術スタックに応じた**パッケージマネージャ**（例: Node.js なら npm/yarn/pnpm、Python なら pip/uv）

**上記以外を前提条件として設定しないこと。**
出典: `docs/product/technical_spec.md` §1 / `docs/misc/人が最初に書いたメモ.md`

## H-06 README.md を起点にセットアップと起動ができること
インフラ構築手順（Dockerコマンド等）を含め、AIが自律的に発見・実行できるように `README.md` **または README.md から参照されるドキュメント／スクリプト**にまとめる。評価者のAIは以下のプロンプトを与えられ、人間の手助けなく①初回セットアップ ②起動 を完了できなければならない。**手順はREADMEを起点に辿って発見可能な場所に置くこと**（プロンプトは「README以外を読まない」とまでは述べていないが、READMEから到達できない手順は発見されない前提で設計する）。

> このプロジェクトのREADME.mdを読み込み、アプリケーションの初期セットアップ方法を把握してください。README.mdから参照されているドキュメントやスクリプトがあれば、それらも参照してください。その後、手順に従って初回のセットアップを実行してください。（後略）

> このプロジェクトのREADME.mdを読み込み、アプリケーションの起動方法を把握してください。README.mdから参照されているドキュメントやスクリプトがあれば、それらも参照してください。その後、手順に従ってアプリケーションを起動してください。（後略）

出典: `README.md` §3 / `docs/product/technical_spec.md` §3 / `docs/misc/人が最初に書いたメモ.md`

> **H-06 の充足方法（決定済 2026-07-19・論点 D00 クローズ）**
>
> 現在の `README.md` にはセットアップ／起動手順が存在しないため、H-06 を満たすには手順ドキュメントの追加と、README.md からそこへの参照が必要になる。原典はこれを禁じていない（`docs/guide/development_rules.md` が編集を禁じているのは **人間** の操作のみ。むしろ `technical_spec.md` §3 は手順のREADME集約を要求している）。
>
> ユーザー判断により、以下の運用が確定した:
>
> - `README.md` は**原則編集しない**が、完全な不可侵ではない。
> - **手順への参照追加に限った最小限の追記**を認める。
> - 手順の**本文**は README.md に書かない。`docs/` 配下でもない別ドキュメントに置き、README.md からはリンクのみを追加する。
> - **追加のみ・再構成なし**（既存記述の削除・書き換え・並べ替えを行わない）。
> - この範囲に収まる限り**事前承認は不要**（実施後に報告・事後承認）。範囲を超える変更は事前承認を要する。
> - `docs/` 配下は引き続き読み取り専用（AIも編集しない）。
>
> 詳細は `requirements-analysis.md` の D00 および P-08。

## H-07 秘匿キーの扱い
ユーザーが書き込むべき秘匿キー等がある場合は、その旨をユーザーに伝える運用となる。**それ以外はすべてAIが自動で実行できる**こと（＝対話的な手作業や人間の判断を要求する手順を含めない）。
出典: `README.md` §3

## H-08 評価は空のDBから開始する
すべてのテストを開始する直前は、データベースの内容が**完全に空（クリーン）**な状態である必要がある。
出典: `docs/evaluation/scenarios/README.md` §1

## H-09 シナリオ01→05を逐次実行する
テストシナリオは、ファイル名の先頭にある番号（`01_`, `02_`, ...）の**順に**実行される。
出典: `docs/evaluation/scenarios/README.md` §2

## H-10 テスト間でDBを消さない
シナリオ間ではデータベースのクリアを行わず、**前のテストで作成されたユーザーやデータをそのまま利用**する。
出典: `docs/evaluation/scenarios/README.md` §2

## H-11 シードデータ投入は不要（＝前提にしない）
テストに必要なユーザー・スレッド・コメントは、シナリオの中で手順として作成される。**事前のシードデータ投入は不要**であり、シードが存在することを前提とした実装をしてはならない。
出典: `docs/evaluation/scenarios/README.md` §3

## H-12 テストは自然言語シナリオをAIが解釈して実行する
Playwright等の自動テストランナーでシナリオを実行するのではなく、**自然言語で記述されたシナリオをもとにAIが `agent-browser` を利用して**テストする。したがって、画面上の表示・ラベル・状態変化がAIにとって自然言語で観測可能である必要がある。
出典: `docs/misc/人が最初に書いたメモ.md`「実装後の動作確認について」

## H-13 clone直後の状態から検証が始まる
gitからcloneした直後のリポジトリに対してセットアップが実行される。リポジトリに含まれない生成物・手動配置ファイルに依存してはならない。
出典: `docs/misc/人が最初に書いたメモ.md`

## 合成的な帰結（H-08〜H-11 の組み合わせから導かれる実装上の含意）

- シナリオ01で作成された `testuser_01` はシナリオ05まで生存し続ける（`testuser_02` も同様にシナリオ02以降生存）。
- スレッドの残存状況は次のとおり——シナリオ01はスレッドを作成せず、シナリオ02で作成したスレッドは同シナリオ内で削除される。したがって**シナリオ04のページネーション検証時点で先行シナリオから残っているのは、シナリオ03の「コメントテスト用」1件のみ**であり、そこにシナリオ04で作成するスレッドA・スレッドBおよび追加分が積み上がる。件数は先行シナリオの実行結果に依存するため、**一覧の総件数や「特定スレッドが何ページ目にあるか」を固定的に仮定した実装・文言にしてはならない**。
- 評価を再実行するには**空DB状態へ戻す手段**が必要になる（論点 D14）。初回セットアップが空DBを構築すれば H-08 は満たせるが、リトライのための明示的なリセット手段を README 系ドキュメントから発見可能にしておくことが望ましい。
- シナリオ03で「削除済みコメント1件＋通常コメント1件」の状態が作られ、その状態でスレッド削除が拒否されることが検証される（削除済みコメントもコメント数に数える）。
